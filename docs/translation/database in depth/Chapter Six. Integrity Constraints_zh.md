# 第六章. 完整性约束

**我在前面章节中多次提及了`完整性约束`（`integrity constraints`）的问题**，但现在是时候具体展开讲了。首先，这里给出一个粗略的定义（重复自第一章）：一个`完整性约束`（简称`约束`）本质上就是一个必须求值为`TRUE`的`布尔表达式`。之所以称为`约束`，是因为它们限制了在特定上下文中合法出现的值。我们感兴趣的约束主要分为两大类：`类型约束`和`数据库约束`；本质上，`类型约束`定义了构成某个给定`类型`的值，而`数据库约束`定义了可以在给定`数据库`中出现的值。

顺便提一下，值得注意的是，广义上的`约束`可以被视为某些人所谓的`业务规则`的形式化版本。我将在下一章再次讨论这一点。

## 类型约束

正如我们在第二章所看到的，当我们定义一个`类型`时，必须做的一件事是指定构成该`类型`的值。以下是一个示例：

```
1  TYPE QTY
2       POSSREP QPR
3            { Q INTEGER
4                    CONSTRAINT Q ≥ 0 AND Q ≤ 5000 } ;
```

_`解释：`_

- 第1行仅仅说明我们正在定义一个名为`QTY`（“数量”）的`类型`。
- 第2行说明数量具有一个名为`QPR`的`可能表示法`（`possible representation`）。正如我们从第二章所知，`物理表示法`始终对用户隐藏。然而，**`Tutorial D`**要求每个`TYPE`语句至少包含一个`POSSREP`规范[^1]，这表明所讨论`类型`的值可以“以某种特定方式被表示”；而且与`物理表示法`不同，`可能表示法`（通常简称为`possreps`）对用户绝对是可见的（在示例中，用户明确知道数量有一个名为`QPR`的`possrep`）。但请非常仔细地注意，这里绝对没有暗示指定的`可能表示法`与`物理表示法`（无论它实际是什么）是相同的；它可能是，也可能不是，但无论如何这对用户都没有影响。
- 第3行指出`possrep` `QPR`具有一个名为`Q`的`组件`，其`类型`为`INTEGER`；换句话说，`QTY`类型的值可以“以整数形式被表示”（并且用户知晓这一事实）。
- 最后，第4行指定这些整数必须位于0到5000的范围内（含）——正是这一事实构成了`QTY`类型的`类型约束`；换句话说，有效的数量正是那些可以由指定范围内的整数表示的值。

以下是一个稍复杂的示例：

```
  TYPE POINT
       POSSREP CARTESIAN { X NUMERIC, Y NUMERIC
               CONSTRAINT SQRT ( X ** 2 + Y ** 2 ) ≤ 100.0 } ;
```

`POINT`类型表示二维空间中的点；它有一个名为`CARTESIAN`的`possrep`，包含两个数值`组件`，分别称为`X`和`Y`（大概对应笛卡尔坐标），并且存在一个`类型约束`，该约束实际上指出：我们感兴趣的点仅限于那些位于以原点为中心、半径为100的圆内或圆上的点。

### 选择器与 THE\_ 运算符

在继续讨论`类型约束`本身之前，我想稍微岔开话题片刻；`QTY`和`POINT`的示例引出了若干需要我在某处解决的问题，而这里正是一个合适的地方。

回顾第二章的内容，像`QTY`和`POINT`这样的用户定义`类型`都关联有`选择器`（`selector`）和`THE_运算符`。这些运算符与`possrep`概念密切相关；事实上，`选择器运算符`与`possrep`一一对应，而`THE_运算符`与`possrep组件`一一对应。以下是一些示例：

**`QPR ( 250 )`**
: 该表达式是`QTY`类型的`选择器调用`。该`选择器`的名称与该`类型`的唯一`possrep`名称`QPR`相同；它接受一个参数，该参数与该`possrep`的唯一`组件`相对应且类型相同，并返回一个数量值（即`QTY`类型的值）。_`注：`_ 在实际应用中，`possreps`的名称通常与关联的`类型`相同——我在`QTY`示例中使用了不同的名称，是为了清楚表明`possrep`与`类型`之间存在逻辑差异，但通常的做法是不加区分的。事实上，有一条语法规则规定：如果我们愿意，可以完全从`TYPE`语句中省略`possrep`名称，此时它将默认为关联的`类型`名称。因此，让我们相应地修改`QTY`类型的定义：

```
TYPE QTY POSSREP { Q INTEGER CONSTRAINT Q ≥ 0 AND Q ≤ 5000 } ;
```

现在，`possrep`和对应的`选择器`都称为`QTY`，前面所示的`选择器调用`就变成了简单的`QTY(250)`——这也是我在本书此前部分一直使用`选择器`的风格。在本章剩余部分，我将采用`QTY`类型的这个修订版定义。

**`QTY ( A + B )`**
: `QTY选择器调用`的参数可以指定为任意复杂的表达式（只要它的类型是`INTEGER`即可）。如果该表达式是一个字面量，如前一个示例所示，那么该`选择器调用`本身也是一个字面量；换句话说，字面量是`选择器调用`的一种特殊情况，正如我们在第三章已经知道的那样。

**`THE_Q ( QZ )`**
: 该表达式是`QTY`类型的`THE_运算符调用`。该运算符之所以命名为`THE_Q`，是因为`Q`是`QTY`类型唯一`possrep`的唯一`组件`的名称；它接受一个参数（指定为任意复杂的表达式），类型为`QTY`；并返回该参数`possrep`中`Q`组件所对应的整数值。

现在，让我们重新定义`POINT`类型，使其`possrep`与类型同名（为简化起见，我们暂时也去掉`类型约束`）：

```
  TYPE POINT POSSREP { X NUMERIC, Y NUMERIC } ;
```

继续示例：

**`POINT ( 5.7, -3.9 )`**
: 这是一个`POINT选择器调用`（实际上是一个`POINT`字面量）。

**`THE_X ( P )`**
: 该表达式返回一个`NUMERIC`值，即变量`P`中存储的点的笛卡尔`可能表示法`的`X`坐标。（当然，该变量必须是`POINT`类型。）

顺便提一下，`POINT`是一个很好的示例，展示了我们可能希望为其定义多个`possrep`的`类型`——例如：

```
  TYPE POINT POSSREP CARTESIAN { X NUMERIC, Y NUMERIC }
           POSSREP POLAR { R NUMERIC, THETA NUMERIC } ;
```

这里的两个`possrep`反映了二维空间中的点确实可以“可能地”用笛卡尔坐标或极坐标来表示。每个`possrep`都有两个`组件`，类型均为`NUMERIC`，且均对用户可见。继续示例：

**`POLAR ( H, K )`**
: 这是一个`POLAR选择器调用`；它返回一个`POINT`类型的值。

**`THE_THETA ( P )`**
: 该表达式返回一个`NUMERIC`值，即变量`P`中存储的点的极坐标`可能表示法`的`THETA`坐标（同样，`P`必须是`POINT`类型）。

### 类型约束（续）

现在让我们回到`类型约束`本身。假设我按如下方式定义`QTY`类型，且不带显式的`约束`：

```
  TYPE QTY POSSREP { Q INTEGER } ;
```

此定义被规定为以下定义的简写形式：

```
  TYPE QTY POSSREP { Q INTEGER CONSTRAINT TRUE } ;
```

使用这两种定义中的任何一种，任何可能由整数表示的值都是合法的`QTY`值（因此`QTY`类型仍然受某种`约束`的制约，尽管该约束相当宽松）。换句话说，指定的`possrep`施加了一种*`先验的`*（_`a priori`_）`类型约束`，而如果存在`CONSTRAINT`规范，则会在该先验约束之上施加额外的`约束`。然而，非正式地，我们通常将“`类型约束`”一词专指`CONSTRAINT`规范中声明的内容。

到目前为止，我回避的一个重要问题是`类型约束`何时被检查。事实上，它们是在*`每次调用选择器时`*进行检查的。再次假设`QTY`类型的值受限于必须能表示为0到5000范围内整数的`约束`。那么表达式`QTY(250)`是`QTY选择器`的一次调用，该调用成功。相比之下，表达式`QTY(6000)`也是此类调用，但它会失败。一般而言，我们*`绝不能`*容忍一个本应表示某种`类型`*`T`*的值但实际上却不是该类型的表达式；的确，一个“不是`类型`*`T`*的值的`类型`*`T`*的值”在术语上是自相矛盾的。因此，没有任何变量——尤其是没有任何`关系变量`（`relvar`）——可以被赋予一个不属于正确`类型`的值。

_`关于 SQL 的说明：`_ 您会注意到，本节中的所有示例都是使用**`Tutorial D`**表达的，而不是`SQL`。这是因为，信不信由你，`SQL`根本不支持`类型约束`——当然，除了那些相当琐碎的先验约束外[^2]。换句话说，尽管`SQL`允许您定义`QTY`类型（例如）并指定数量必须可表示为整数，但它*`不会`*让您指定这些整数必须位于某个范围内。出于此类及其他原因，我不想在本书中详细讨论`SQL`的用户定义`类型`。不过，我至少会展示一下我们已见过的**`Tutorial D`** `QTY`和`POINT`定义的`SQL`对应形式：

```
  CREATE TYPE QTY AS INTEGER FINAL ;

  CREATE TYPE POINT AS ( X NUMERIC, Y NUMERIC ) NOT FINAL ;
```

本节最后一点：将某物定义为特定`类型`当然会对该物施加一个`约束`。特别是，将`关系变量` `SP`的`属性` `QTY`（例如）定义为`QTY`类型，就施加了这样一个`约束`：`SP`关系变量中的任何`元组`在`QTY`位置上都永远不会包含不满足`QTY类型约束`的值。（这有时被称为`属性约束`的一个示例。）

## 数据库约束

回顾一下，`数据库约束`限制了可以在给定`数据库`中出现的值。在**`Tutorial D`**中，此类`约束`通过`CONSTRAINT`语句（或某些在效果上等效的简写形式）指定；在`SQL`中，它们通过`CREATE ASSERTION`语句（或再次，某些等效简写形式）指定。我不想深入探讨这些简写形式的细节——至少目前不想——因为它们本质上只是语法问题；现在，让我们继续使用“`长格式`”形式。我将以一系列示例开始（和往常一样，左侧为**`Tutorial D`**，右侧为`SQL`）。

### 示例 1

状态值必须位于1到100的范围内（含）：

```
  CONSTRAINT C1 IS_EMPTY        | CREATE ASSERTION C1 CHECK
  ( S WHERE STATUS < 1       | ( NOT EXISTS
      OR    STATUS > 100 ) ; | ( SELECT S.* FROM S
                                |   WHERE  S.STATUS < 1
                                |   OR     S.STATUS > 100 ) ) ;
```

该`约束`仅涉及单个`关系变量`的单个`属性`；因此，对于给定的供应商`元组`，只需孤立地检查该`元组`即可验证它——无需查看`关系变量`中的任何其他`元组`或`数据库`中的任何其他`关系变量`。（因此，这类`约束`有时被非正式地称为`元组约束`。）

### 示例 2

位于伦敦的供应商必须具有状态20：

```
  CONSTRAINT C2 IS_EMPTY        | CREATE ASSERTION C2 CHECK
  ( S WHERE CITY = 'London'     | ( NOT EXISTS
      AND   STATUS ≠ 20 ) ;  | ( SELECT S.* FROM S
                                |   WHERE  S.CITY = 'London'
                                |   AND    S.STATUS <> 20 ) ) ;
```

该`约束`涉及两个不同的`属性`（属于同一个`关系变量`），但与约束`C1`的情况一样，对于给定的供应商`元组`，仍然只需孤立地检查该`元组`即可。

### 示例 3

`关系变量` `S`中没有任何两个`元组`具有相同的供应商编号（换句话说，{`SNO`}是该`关系变量`的`键`）：[^3]

```
  CONSTRAINT C3             |   CREATE ASSERTION C3 CHECK
     COUNT ( S ) =          |  ( UNIQUE ( SELECT S.SNO
     COUNT ( S { SNO } ) ;  |             FROM   S ) ) ;
```

与约束`C1`和`C2`类似，该约束仍然只涉及一个`关系变量`；然而，显然不能仅通过孤立地检查给定的供应商`元组`来验证它。当然，在实际应用中，约束`C3`极不可能以所示的`长格式`指定——显然更倾向于使用某种显式的`KEY`简写。我给出长格式仅仅是为了说明一点：这些简写在最终分析下确实只是简写而已。

顺便提一下，约束`C3`的`SQL`表述需要稍作解释。`UNIQUE`是一个`SQL`运算符，当且仅当其参数表中的每一行都互不相同时返回`TRUE`；因此约束中的`UNIQUE`调用当且仅当表`S`中没有两行具有相同的供应商编号时返回`TRUE`。但请注意，如果我遵循我偏好的纪律，始终指定`DISTINCT`，如下所示：

```
  UNIQUE ( SELECT DISTINCT S.SNO FROM S )
```

那么`UNIQUE`根本不可能返回`FALSE`！事实上，从`关系型`角度来看，`UNIQUE`是毫无意义的；它之所以在`SQL`中需要，仅仅是因为一般而言`SQL表`不是`关系`。

出于兴趣，以下是约束`C3`的一种`SQL`表述，它更接近**`Tutorial D`**的表述：

```
  CREATE ASSERTION C3 CHECK
   ( ( SELECT COUNT ( * ) FROM S ) =
     ( SELECT COUNT ( SNO ) FROM S ) ) ;
```

### 示例 4

状态小于20的供应商不能供应零件`P6`：

```
  CONSTRAINT C4 IS_EMPTY       | CREATE ASSERTION C4 CHECK
  ( ( S JOIN SP )              | ( NOT EXISTS
    WHERE STATUS < 20       | ( SELECT *
    AND   PNO = PNO('P6') ) ;  |   FROM   S, SP
                               |   WHERE  S.SNO = SP.SNO
                               |   AND    S.STATUS < 20
                               |   AND    SP.PNO = PNO('P6') ) ) ;
```

该`约束`涉及——实际上，相互关联——两个不同的`关系变量`，`S`和`SP`；当然，一般而言，一个`数据库约束`可能涉及或相互关联任意数量的不同`关系变量`。

> **注意**
>
> _`术语：`_ 仅涉及单个`关系变量`的`约束`被非正式地称为`关系变量约束`（有时为强调起见称为`单关系变量约束`）。涉及两个或更多不同`关系变量`的`约束`被非正式地称为`多关系变量约束`。

### 示例 5

`关系变量` `SP`中的每个供应商编号必须出现在`关系变量` `S`中：

```
  CONSTRAINT C5                    | CREATE ASSERTION C5 CHECK
     SP { SNO } ⊆ S { SNO } ; | ( NOT EXISTS
                                   |    ( SELECT SP.SNO
                                   |      FROM   SP
                                   |      EXCEPT
                                   |      SELECT S.SNO
                                   |      FROM   S ) ) ;
```

如您所见，该`约束`的**`Tutorial D`**表述涉及一个`关系比较`。然而，`SQL`不支持`关系比较`，因此在`SQL`表述中我们不得不进行一些迂回表达。当然，鉴于{`SNO`}是`关系变量` `S`的`键`——事实上是*`唯一`*的`键`——显然约束`C5`基本上就是从`SP`到`S`的`外键约束`。因此，通常的`FOREIGN KEY`语法可以被视为约束`C5`这类约束的简写。

那么`数据库约束`何时被检查？传统观点认为，`单关系变量约束`检查是*`立即的`*（意味着它在相关`关系变量`被更新时执行），而`多关系变量约束`检查会*`延迟`*到`事务结束`（“`提交时`”）。然而，我想论证的是，_`所有`_`数据库约束`检查都必须是`立即的`，而`延迟检查`（受`SQL标准`支持，且确实存在于某些`SQL`产品中）是一个逻辑错误。为了解释这一非正统观点，我需要暂时岔开话题讨论一下`事务`。

## 事务

`事务理论`本身就是一个庞大的主题。然而，正如第四章所述，它与`关系模型`本身关系不大（至少不是直接关系），因此我不想在此详细讨论。无论如何，您是一位`数据库`专业人士，我相信您熟悉基本的`事务`概念。标准参考文献——顺便强烈推荐阅读——是 Jim Gray 和 Andreas Reuter 所著的 _Transaction Processing: Concepts and Techniques_（Morgan Kaufmann, 1993）。我在这里只想简要回顾一下所谓的`事务`的`ACID特性`。

`ACID`是一个首字母缩写词；它代表*`原子性`*（_`atomicity`_）- _`一致性`_（_`consistency`_）- _`隔离性`_（_`isolation`_）- _`持久性`_（_`durability`_）。以下是简要解释：

**`原子性`**
: `事务`是“全有或全无”的。

**`一致性`**
: 任何给定的`事务`都会将`数据库`的`一致状态`转换为另一个`一致状态`，而不必在所有中间点都保持`一致性`。[^4]

**`隔离性`**
: 任何给定`事务`的更新对其他所有`事务`都是隐藏的，直到该给定`事务`提交。

**`持久性`**
: 一旦给定的`事务`提交，其更新就会在`数据库`中存活，即使随后发生系统崩溃。

现在，支持`事务`的一个论点始终是它们应该充当“`完整性的单位`”（这就是`一致性属性`的全部意义）。但我不相信这个论点；正如我已经或多或少说过的那样，我认为*`语句`*必须是那个单位，因此`数据库约束`必须在*`语句边界`*得到满足。紧接的下一节将给出我对此立场的论证。

## 为什么数据库约束检查必须是立即的

我至少有五个理由认为`数据库约束`必须在语句边界得到满足。第一个也是最大的理由如下。正如我们从第四章所知的，一个`数据库`可以被视为一组`命题`的集合，这些命题我们认为是真的。如果该集合被允许包含任何不一致，那么*`一切都将无法保证`*。正如我将在本章后面的“约束与谓词”一节中展示的那样，我们绝不能信任从不一致`数据库`中获得的答案！尽管由于`隔离性属性`，可能最多只有一个`事务`会看到任何特定的不一致，但事实仍然是，该特定`事务`确实看到了不一致，因此可能产生错误的答案。

现在，我认为第一个论点本身已经足够有力，但为了参考起见，我还是给出其他四个论点。第二，我无论如何也不同意任何给定的不一致只能被一个`事务`看到；也就是说，我不相信`隔离性属性`。这里的部分问题在于，*`隔离`*一词在`事务`世界中的含义与在日常英语中的含义不同——特别是，它并不意味着`事务`之间不能相互通信。因为如果`事务`*`T1`*在`数据库`中或其他地方产生了一些结果，随后被`事务`*`T2`*读取，那么*`T1`*和*`T2`*并未真正彼此隔离（无论*`T1`*和*`T2`*是并发运行还是其他方式，此评论均适用）。特别是，因此，如果 (a) *`T1`*看到`数据库`的不一致状态并因此产生错误结果，并且 (b) 该结果随后被*`T2`*看到，那么 (c) *`T1`*看到的不一致实际上已传播给*`T2`*。换句话说，无法保证如果允许不一致，它*`将`*仅被一个`事务`看到。

第三，我们肯定不希望每个程序（或其他“`代码单元`”）都必须应对在调用时`数据库`可能不一致的情况。如果一个假设一致的程序在约束检查延迟时无法安全使用，那么正交性将严重丧失。换句话说，我希望能够独立于代码单元是作为`事务`本身执行还是仅作为`事务`的一部分执行来指定它们。（事实上，我希望支持`嵌套事务`，但那是另一天的话题了。）

第四，_`可互换性原则`_（`基本关系变量`和`视图`之间——见第四章）意味着完全相同的`约束`在一种`数据库`设计中可能是`单关系变量约束`，而在另一种设计中则可能是`多关系变量约束`。例如，回想一下第四章中的这两个`视图`：

```
    VAR LS  VIRTUAL ( S WHERE CITY = 'London' ) ;

    VAR NLS VIRTUAL ( S WHERE CITY ≠ 'London' ) ;
```

这些`视图`满足“没有供应商编号同时出现在两者中”的`约束`。然而，无需显式声明该`约束`，因为它由{`SNO`}是`关系变量` `S`的`键`这一`单关系变量约束`所隐含（加上每个供应商恰好有一个城市这一事实，这隐含在`关系变量` `S`的设计中）。但假设我们将`LS`和`NLS`设为`基本关系变量`，并将它们的并集定义为一个名为`S`的`视图`。那么该`约束`就必须显式声明：

```
    CONSTRAINT C6 IS_EMPTY   |  CREATE ASSERTION C6 CHECK
     ( LS { SNO } JOIN       |  ( NOT EXISTS
      NLS { SNO } ) ;        |    ( SELECT *
                             |    FROM   LS, NLS
                             |  WHERE  LS.SNO = NLS.SNO ) ) ;
```

现在，之前的`单关系变量约束`变成了`多关系变量约束`。因此，如果我们同意`单关系变量约束`必须被`立即`检查，我们肯定也必须同意`多关系变量约束`必须被`立即`检查。

第五也是最后一点，存在一种称为`语义优化`的优化技术（它涉及`表达式转换`，但我故意没有在第五章相应名称的章节中讨论它）。例如，考虑表达式`(SP JOIN S){PNO}`。显然，这里的`连接`是一个`外键匹配主键的连接`；因此，每个`SP元组`都会连接到某个`S元组`，从而为整体结果贡献一个零件编号。所以根本不需要进行`连接`！——该表达式可以简化为`SP{PNO}`。然而，需要注意的一点是，这种转换有效*`仅仅是因为`*情境的语义；一般而言，`连接`的每个操作数都将包含一些在另一个操作数中没有对应项的`元组`，因此不会对整体结果做出贡献，因此像刚才所示的转换将无效。然而，在当前情况下，由于存在`完整性约束`——实际上是一个`外键约束`——规定每次发货必须有一个供应商，因此每个`SP元组`*`必须`*在`S`中有对应项，所以转换最终是有效的。仅因为某个`完整性约束`生效而有效的转换称为`语义转换`，由此产生的优化称为`语义优化`。

原则上，任何`约束`都可以用于`语义优化`（我们不仅限于`外键约束`）。例如，假设供应商-零件`数据库`受以下`约束`制约：“`所有红色零件必须存储在伦敦`”，并考虑以下查询：

> 获取仅供应红色零件且位于他们供应的至少一个零件所在城市的供应商。

这是一个相当复杂的查询。但由于`完整性约束`，我们发现它可以被转换——我的意思是，由优化器转换，而不是由用户转换——为这个简单得多的查询：

> 获取仅供应红色零件的伦敦供应商。

在这里，性能提升很容易达到几个数量级。因此，尽管据我所知，在撰写本文时很少有产品大量进行`语义优化`，但我绝对期望它们在未来做得更多，因为回报是如此巨大。

回到讨论的主线，我现在指出，如果某个给定的`约束`要可用于`语义优化`，那么该`约束`必须*`在所有时刻`*（更准确地说，在语句边界）得到满足，而不仅仅是在事务边界。正如我们所见，`语义优化`意味着使用`约束`来简化查询以提高性能。显然，如果某个`约束`在某个时刻被违反，那么基于该`约束`的任何简化在该时刻都将无效，并且基于该简化的查询结果在该时刻也将是错误的（一般而言）。

> **注意**
>
> 当然，我们可以采取较弱的立场，即“`延迟约束`”（指检查被延迟的`约束`）根本不能用于`语义优化`——但我认为这种立场实际上只是意味着我们搬起石头砸了自己的脚，仅此而已。

总之，`数据库约束`必须得到满足——也就是说，给定`数据库`中当前出现的值，它们必须求值为`TRUE`——_`在语句边界`_（或者非常非正式地说，“`在分号处`”）；换句话说，它们必须在任何可能导致它们被违反的语句末尾进行检查。如果任何此类检查失败，则 offending 语句对`数据库`的影响将被撤销，并引发异常。

## 但某些检查难道不需要延迟吗？

正如我所说的，传统观点是，至少`多关系变量约束`检查必须延迟到`提交时`（尽管有上一节的论点）。举例来说，假设供应商-零件`数据库`受以下`约束`制约：

```
  CONSTRAINT C7
    COUNT ( ( S WHERE SNO = SNO('S1') ) { CITY }
        UNION
        ( P WHERE PNO = PNO('P1') ) { CITY } ) < 2 ;
```

该`约束`指出供应商`S1`和零件`P1`绝不能位于不同的城市。详细说明如下：如果`关系变量` `S`和`P`分别包含供应商`S1`和零件`P1`的`元组`，那么这些`元组`必须包含相同的`CITY`值（如果不是，`COUNT`调用将返回值2）。然而，`关系变量` `S`可以不包含`S1`的`元组`，或者`关系变量` `P`可以不包含`P1`的`元组`，或者两者都不包含（在这种情况下，`COUNT`调用将返回1或0）。根据我们通常的示例值，以下每个`SQL UPDATE`在`立即检查`下都将失败：[^5]

```
  UPDATE S SET CITY = 'Paris' WHERE SNO = SNO('S1') ;

  UPDATE P SET CITY = 'Paris' WHERE PNO = PNO('P1') ;
```

现在，此问题的传统解决方案是延迟检查，并将这两个`UPDATE`捆绑到一个`事务`中，如下所示：

```
  BEGIN TRANSACTION ;
  UPDATE S SET CITY = 'Paris' WHERE SNO = SNO('S1') ;
  UPDATE P SET CITY = 'Paris' WHERE PNO = PNO('P1') ;
  COMMIT ;
```

在这种传统解决方案中，`约束`在`提交时`检查，并且在两个`UPDATE`之间`数据库`处于不一致状态。特别是，如果`事务`在两个`UPDATE`之间提出问题“`供应商S1和零件P1是否位于不同的城市？`”（并且如果我们假设存在`S1`和`P1`的`元组`），它将得到答案*`是`*。

### 多重赋值

解决上述问题的更好方法是支持*`多重`*赋值形式，它允许执行任意数量的单个赋值“`同时`”。例如（现在切换回**`Tutorial D`**）：

```
  UPDATE S WHERE SNO = SNO('S1') ( CITY := 'Paris' ) ,
  UPDATE P WHERE PNO = PNO('P1') ( CITY := 'Paris' ) ;
```

_`解释：`_ 首先，请注意逗号分隔符，这意味着这两个`UPDATE`属于同一个整体语句。其次，`UPDATE`本质上就是赋值，因此上述“`双重UPDATE`”只是以下一般形式的`双重赋值`的简写：

```
  S := ... , P := ... ;
```

此`双重赋值`将一个新值赋给`关系变量` `S`，将另一个新值赋给`关系变量` `P`，所有这些都作为同一个整体操作的一部分。一般而言，`多重赋值`的语义如下：

- 首先，计算右侧的所有表达式。
- 其次，然后按书写顺序依次执行所有构成赋值。[^6]

请注意，正是因为右侧的表达式在任何构成赋值执行之前都被求值，所以这些构成赋值中没有任何一个可以依赖于任何其他的结果。此外，由于`多重赋值`被视为单个操作，因此不会在任何此类赋值的“`中间`”执行完整性检查。（事实上，这一事实正是首先支持`多重赋值`的主要理由。）因此，在示例中，`双重赋值`成功了，而两个单独的单一赋值却失败了。

特别注意示例中，现在`事务`无法在两个`UPDATE`之间看到`数据库`的不一致状态，因为“`在两个UPDATE之间`”的概念现在已毫无意义。进一步注意，现在根本不再需要延迟检查。

顺便提一下，`SQL`确实对`多重赋值`有一些支持；事实上，它多年来一直有*`一些`*支持。首先，`CASCADE`等`参照动作`实际上意味着单个`DELETE`或`UPDATE`语句可以作为单个操作的一部分“`同时`”更新多个表。其次，更新（例如）某些`连接视图`的能力也意味着同样的事情。第三，`FETCH INTO`和`SELECT INTO`在某种程度上都是`多重赋值`操作。第四，`SQL:2003`引入了`SET`语句的`多重赋值`形式。等等（这不是一个详尽的列表）。然而，`SQL`目前不支持的`多重赋值`类型是对多个不同表的显式赋值（当然，这正是上述示例说明的情况）。

最后一点：请理解，支持`多重赋值`并不意味着我们可以放弃对`事务`的支持；`事务`对于恢复和并发目的仍然是必要的，至少如此。我只是在说，`事务`并不是它们通常被认为的“`完整性的单位`”。

## 约束与谓词

回想第四章，`关系变量`的`关系变量谓词`是该`关系变量`的*`预期解释`*——粗略地说，就是该`关系变量`的*`含义`*。例如，`关系变量` `S`的`谓词`如下所示：

> 供应商`SNO`已签订合同，名为`SNAME`，状态为`STATUS`，位于城市`CITY`。

因此，在理想世界中，此`谓词`将作为`关系变量` `S`上更新可接受性的标准——也就是说，它将决定对该`关系变量`的给定`INSERT`、`DELETE`或`UPDATE`操作是否可以被接受。但这一目标显然是无法实现的：

- 一方面，系统无法知道“`供应商`”“`已签订合同`”或“`位于`”某处意味着什么；重申一遍，这些是解释问题。例如，如果供应商编号`S1`和城市名称`London`恰好出现在同一个`元组`中，那么用户可以将这一事实解释为供应商`S1`位于`London`，[^7]但系统无法做任何类似的事情。
- 另一方面，即使系统能够知道供应商已签订合同或位于某处意味着什么，它仍然无法*`先验地`*知道用户告诉它的内容是否为真！如果用户通过某种更新向系统断言存在一个名为`Lopez`、状态为30、城市为`Madrid`的供应商`S6`，系统无法知道该断言是否为真。系统能做的只是检查用户的断言是否违反了任何`完整性约束`。假设没有违反，那么系统将接受用户的断言*`并从此将其视为真`*（直到用户通过执行另一次更新告诉系统它不再为真为止）。

因此，_`实用的`_“`更新可接受性标准`”（与理想标准相对）不是`谓词`，而是对应的`约束`集，因此可以将其视为系统对`谓词`的近似。[^8] 等价地：

> 系统无法强制保证真值，只能强制保证一致性。

换句话说：系统无法保证`数据库`仅包含真实的`命题`——它只能保证不包含违反任何`约束`的内容（意味着不包含不一致）。可悲的是，真值和一致性不是一回事。具体来说：

- 如果`数据库`仅包含真实的`命题`，那么它是一致的，但反之不一定成立。
- 如果`数据库`不一致，那么它至少包含一个虚假的`命题`，但反之不一定成立。

更简洁地说，*`正确`*蕴含*`一致`*（但反之不成立），而*`不一致`*蕴含*`不正确`*（但反之不成立）——其中，说`数据库`是*`正确`*的，意味着它真实地反映了现实世界的真实状况，不多也不少。

现在让我尝试更精确地界定这些概念。设*`R`*为一个`基本关系变量`——我马上会讨论`视图`——并设*`C1, C2, . . . , Cn`*为所有提及*`R`*的已定义`数据库约束`（单关系变量或多关系变量）。为简化起见，假设每个*`Ci`*只是一个`布尔表达式`（换句话说，为简化起见，让我们忽略`约束`名称）。那么`布尔表达式`：

```
  ( C1 ) AND ( C2 ) AND ... AND ( Cn ) AND TRUE
```

是`关系变量`*`R`*的*`总关系变量约束`*（但为了本书的目的，我将其简称为*`R`*的*`约束`*）。顺便注意最后的“`AND TRUE`”；其含义是，如果为给定的`关系变量`未定义任何`约束`，则默认为`TRUE`。

现在设*`RC`*为`关系变量`*`R`*的“`约束`”。显然，*`R`*绝不允许具有导致*`RC`*求值为`FALSE`的值。这种状况是我喜欢称为*`黄金规则`*的（第一版）动机：

> 任何更新操作都绝不得导致任何关系变量约束求值为 FALSE。

现在设*`V`*为一个`视图`。那么*`V`*也有一个“`总关系变量约束`”（为简化起见，我通常将其简称为*`V`*的*`约束`*），该约束显然派生自定义*`V`*所基于的`关系变量`所适用的`约束`。例如，设*`SC`*为`基本关系变量` `S`的总`约束`。那么`视图` `LS`（“`伦敦供应商`”）的总`约束`为：

```
  ( SC ) AND ( CITY = 'London' )
```

现在设*`DB`*为一个`数据库`，并设*`DB`*包含`关系变量`_`R1, R2, . . . , Rn`_（仅此而已）。设这些`关系变量`的`约束`分别为*`RC1, RC2, . . . , RCn`*。那么*`DB`*的*`总数据库约束`*（设为*`DBC`*——为了本书的目的，我将其简称为*`DB`*的*`约束`*）是所有这些`关系变量约束`的`AND`：

```
  ( RC1 ) AND ( RC2 ) AND ... AND ( RCn )
```

以下是相应扩展（事实上是最终版）的*`黄金规则`*：

> 任何更新操作都绝不得导致任何数据库约束求值为 FALSE。

特别请注意——根据我所有完整性检查必须是立即的立场——该规则谈论的是更新操作，而不是事务。

现在我可以处理一件未完成的事了。我说过我们绝不能信任从不一致`数据库`中获得的答案。以下是证明。正如我们所知，一个`数据库`可以被视为一组`命题`的集合。假设该集合不一致；也就是说，假设它暗示*`p`*和 NOT *`p`*均为真，其中*`p`*是某个`命题`。现在设*`q`*为任意`命题`。那么：

- 从*`p`*为真，我们可以推断*`p`* OR *`q`*为真。
- 从*`p`* OR *`q`*为真以及 NOT *`p`*为真，我们可以推断*`q`*为真。

但*`q`*是任意的！因此，在不一致系统中，任何`命题` whatsoever（即使是明显虚假的命题如 1 = 0）都可以被证明为真。

## 其他问题

在本节中，我想讨论一些与完整性相关但不太适合放在前面任何一节中的问题。

首先，由于它本质上是一个必须求值为`TRUE`的`布尔表达式`，因此从形式上看，一个`约束`就是一个`命题`。以下是前面“数据库约束”一节中的约束`C1`的示例：

```
  CONSTRAINT C1 IS_EMPTY ( S WHERE STATUS < 1 OR STATUS > 100 ) ;
```

给定`关系变量` `S`的特定值，`布尔表达式`：

```
  IS_EMPTY ( S WHERE STATUS < 1 OR STATUS > 100 )
```

（可以粗略地视为`约束本身`）确实无条件地要么为真要么为假，这正是成为`命题`的定义（见第四章）。

其次，假设`关系变量` `S`已经包含一个在刚刚显示的`CONSTRAINT`语句执行时违反约束`C1`的`元组`。那么该执行必然失败。更一般而言，每当尝试定义新的`数据库约束`时，系统必须首先检查该`约束`在当时是否被`数据库`满足。如果不满足，则必须拒绝该`约束`；否则，必须接受并从那时起强制执行。

第三，回想第一章，`关系模型`包含我所谓的“`泛型`”`完整性规则`：即`参照完整性规则`（我故意忽略了`实体完整性规则`）。但应该清楚的是，`参照完整性规则`在性质上与我们在本章中一直检查的`约束`不同。它实际上是一种*`元约束`*（_`metaconstraint`_）；例如，它指出在包含`关系变量` `S`、`P`和`SP`的特定`数据库`中，`SP`与`S`之间以及`SP`与`P`之间必须存在某些特定的`约束`（`外键约束`）——因为如果不存在，那么该`数据库`可能会违反`参照完整性元约束`。同样，在包含`关系变量` `EMP`和`DEPT`（见第一章）的特定`数据库`中，`EMP`和`DEPT`之间必须存在特定的`外键约束`，因为如果不存在，该`数据库`同样可能会违反`参照完整性元约束`。

第四，我从未明确提及这一点，但我相信这很明显：我们希望`约束`以*`声明式`*方式陈述。尽管`SQL标准`确实包含对`声明式约束`相当广泛的支持，但至少一些主要的`SQL`产品并不支持；相反，它们假设您将使用`触发过程`（也称为`触发器`）来强制执行完整性。（标准也包含对`触发器`的显式支持。）然而，正如我在第一章所指出的，如果可用，*`声明式解决方案`*始终优于*`过程式解决方案`*。此外，特别是`声明式约束`为进行`语义优化`打开了大门，而`触发器`则不能。

我之前没有提及的另一个问题是支持`状态转换约束`（`transition constraints`）的可能性。`状态转换约束`是对某种变量（特别是`关系变量`）从一个值到另一个值的合法转换施加的约束。例如，一个人的婚姻状况可以从“`未婚`”变为“`已婚`”，但不能反向改变。以下是一个示例（“`供应商的状态绝不得降低`”）：

```
  CONSTRAINT C8 IS_EMPTY
     ( ( ( S' { SNO, STATUS } RENAME ( STATUS AS STATUS' ) )
       JOIN
       ( S { SNO, STATUS } ) )
     WHERE STATUS' > STATUS ) ;
```

_`解释：`_ 我采用这样的约定：带撇号的关系变量名称（如`S'`）指的是在*`所考虑的更新之前`*的指示`关系变量`。因此，约束`C8`指出：“`如果我们将S的旧值与新值连接，并将结果限制为旧状态大于新状态的元组，则最终结果必须为空。`”（由于连接基于`SNO`，连接中任何旧状态大于新状态的`元组`将表示状态已降低的供应商。）

最后，我希望您同意本章涵盖的所有内容表明`约束`是*`至关重要的`*——然而，它们在商业产品中似乎支持得非常差；事实上，它们充其量似乎被低估了，如果不是完全被误解的话。商业世界的重点似乎总是*`性能、性能、性能`*；其他目标，如易用性、`数据独立性`，特别是完整性，似乎经常被牺牲给——或者至少退居——这一压倒性目标之后。[^9] 但是，如果我们不能确信从中获取的信息是正确的，系统运行得快又有什么意义呢？坦率地说，如果我觉得无法信任系统为我查询提供正确答案，我根本不在乎它运行得有多快。

## 总结

我讨论了两种基本的`约束`：`类型约束`和`数据库约束`。`类型约束`定义了构成给定`类型`的值集；在**`Tutorial D`**中，它被指定为相关`类型`定义的一部分，并根据该`类型`的`可能表示法`（`possrep`）来表达。`possrep`本身对`类型`施加了一种*`先验的`*（_`a priori`_）`约束`（且`SQL`除了此类先验约束外不支持任何`类型约束`）。`类型约束`作为`选择器运算符`执行的一部分进行检查。作为插叙，我详细阐述了`选择器`的概念以及相关的`THE_运算符`概念，这两者都与`possrep`概念密切相关。

`数据库约束`限制了可以在给定`数据库`中出现的值（如果它们仅适用于一个`关系变量`，则有时被非正式地称为`关系变量约束`）。它们在**`Tutorial D`**中通过`CONSTRAINT`语句指定，或在`SQL`中通过`CREATE ASSERTION`语句指定，并且应该在“`分号处`”进行检查（尽管在`SQL`中可能并非如此）。在分号处检查`约束`意味着在任何可能导致它们被违反的语句末尾进行检查——这基本上意味着`关系赋值`，因为`关系赋值`从根本上说是唯一可以更新`数据库`的操作。我给出了一系列论点来驳斥“至少`多关系变量约束`无需在提交前检查”的传统观点（并在此过程中简要讨论了`语义优化`的概念）。我还介绍了`多重赋值`的重要概念。

接下来，我展示了`关系变量`*`R`*的“`约束`”可以被视为系统对*`R`*的`关系变量谓词`的近似；特别是，它作为*`R`*上更新可接受性的标准。“`系统无法强制保证真值，只能强制保证一致性。`”*`黄金规则`*指出：

> 任何更新操作都绝不得导致任何数据库约束求值为 FALSE。

我最后断言，完整性`约束`绝对至关重要。事实上，对我来说，它们就是数据库系统的核心所在。再说一遍：如果我不能信任它给出的答案，我根本不在乎您的系统运行得有多快。
