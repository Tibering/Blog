# 第三章 元组与关系

**通过前两章的学习，你应该已经对 `元组` 和 `关系` 有了相当好的理解**，至少从非正式的角度来看是这样。现在我要更精确地定义这些概念，并探讨这些精确定义的一些推论。也许我应该提醒你，这些定义看起来可能有点令人生畏，但这在形式定义中并不罕见；一旦你努力克服了形式主义的障碍，`元组`[\*](#fn1) 和 `关系` 概念本身是相当直接的，而且你现在应该已经准备好去做了，因为至少术语对你来说应该已经相当熟悉了。

有了这个警告之后，我现在可以确切地说出什么是 `元组` 了：

> _`定义：`_ 设 _`T1, T2, . . . ,Tn`_ (_`n`_ ≥ 0) 是 `类型` 名，不一定都不同。与每个 _`Ti`_ 关联一个不同的 `属性` 名 _`Ai;`_ 由此产生的 _`n`_ 个 `属性名:类型名` 组合中的每一个都是一个 _`属性`_。与每个 `属性` 关联一个 `类型` _`Ti`_ 的 `值` _`vi`_；由此产生的 _`n`_ 个 `属性:值` 组合中的每一个都是一个 _`分量`_。这样定义的所有 _`n`_ 个 `分量` 的集合，记作 _`t`_，是 `属性` _`A1, A2,..., An`_ 上的一个 _`元组值`_（或简称 _`元组`_）。_`值`_ _`n`_ 是 _`t`_ 的 _`度`_；度为一的 `元组` 是 _`一元`_ 的，度为二的 `元组` 是 _`二元`_ 的，度为三的 `元组` 是 _`三元`_ 的，...，更一般地，度为 _`n`_ 的 `元组` 是 _`n`_ 元的。所有 _`n`_ 个 `属性` 的集合是 _`t`_ 的 _`表头`_。

例如，参考前面两张图片中的任何一张（我们的供应商 `S1` 的常用 `元组`），我们有：

**`度`**
: 四。`表头` 也被说成具有四度。

**`类型名`**
: `SNO`、`NAME`、`INTEGER` 和 `CHAR`。

**`对应的属性名`**
: `SNO`、`SNAME`、`STATUS` 和 `CITY`。

**`对应的属性值`**
: `SNO('S1')`、`NAME('Smith')`、20 和 `'London'`。

> **注意**
>
> 我在图片中以简化的形式显示了这些 `属性值` 中的大多数（而且在本书中，我将继续在此类图片中使用类似的简化）。然而，严格来说，说例如我们正在讨论的 `元组` 中的 `SNO 值` 仅仅是 `'S1'` 或（甚至更不严谨）仅仅是 `S1` 是不正确的。`类型` `SNO` 的 `值` 是 `类型` `SNO` 的 `值`，而不是 `类型` `CHAR` 的 `值`！——而且表达式 `SNO('S1')` 是 `类型` `SNO` 的一个 _`字面量`_。（更准确地说，它是 _`SNO 选择器的调用`_，正如我们在第 2 章中看到的。`字面量` 精确地说是一个 `选择器` 调用，其中的参数本身又都是 `字面量`。）

**`表头`**
: 这里最简单的做法是再展示一张图片：

![Figure Three-3](httpatomoreillycomsourceoreillyimages247875.png)

当然，这张图片表示一个集合，`属性` 的顺序是任意的。这是同一个 `表头` 的另一张图片：

**图 3-4**

![Figure Three-4](httpatomoreillycomsourceoreillyimages247883.png)

现在，`元组`[\*](#fn5) 是一个 `值`；因此像所有 `值` 一样，它有一个 `类型`（正如我们从第 2 章已经知道的），而且那个 `类型`，像所有 `类型` 一样，有一个名字[\*](#fn6)。在 **`Tutorial D`** 中，这样的名字采用 `TUPLE{`_`H`_`}` 的形式，其中 {_`H`_} 是 `表头`。在我们的例子中，名字是：

```
TUPLE { SNO SNO, SNAME NAME, STATUS INTEGER, CITY CHAR }
```

（但指定 `属性` 的顺序是任意的）。

重申一下，`元组` 是一个 `值`。因此像所有 `值` 一样，它必须由某个 `选择器` 调用返回（如果 `值` 是 `元组`，当然是 `元组选择器`[\*](#fn7) 调用）。这是我们例子的一个 `元组选择器` 调用（又是 **`Tutorial D`**）：

```
TUPLE { SNO SNO('S1'), SNAME NAME('Smith'),
                   STATUS 20, CITY 'London' }
```

（其中指定 `分量` 的顺序是任意的）。注意在 **`Tutorial D`** 中，每个 `分量` 仅指定为 `属性名:属性值` 对；`属性类型` 被省略了，因为它总是可以从表示 `属性值` 的 `表达式` 的 `类型` 中推断出来。

> **注意**
>
> _`关于语法的说明：`_ 正如你所看到的，关键字 `TUPLE` 在 **`Tutorial D`** 中身兼二职——它既用于 `元组选择器` 调用，也用于 `元组类型`[\*](#fn8) 名。类似的说明也适用于关键字 `RELATION`（参见即将讨论的"什么是关系？"一节）。

最后，关于 `SQL` 的一句话。当然，`SQL` 支持 `行`，而不是 `元组`；特别是，它支持 _`行类型`_、_`行类型构造器`_ 和 _`行值构造器`_，它们分别类似于 `元组类型`、`TUPLE 类型生成器` 和 `元组选择器`。但这些类比充其量只是松散的。例如，这个 `SQL 表达式`：

```
ROW ( 1, 2 )
```

这是 `行值构造器` 的一个例子，(a) 具有未命名的 `分量`，(b) 不代表与 `行值构造器` `ROW (2,1)` 相同的 `行`。（`SQL` 的 `行` 支持和我们的 `元组` 支持之间还有其他差异，但它们超出了当前讨论的范围。）

> **注意**
>
> `SQL 行值构造器` 中的关键字 `ROW` 是可选的。

## 一些重要的推论

现在我要强调上述定义的一些重要推论。第一个是：==_`任何元组都不包含任何空值`_。==原因是，根据定义，每个 `元组` 都包含其每个 `属性` 的一个 `值`（适当 `类型` 的），而且我们在第 1 章中看到 `空值` 不是 `值`。[\*](#fn9) 当然，如果没有任何 `元组` 包含任何 `空值`，那么 _`当然`_ 也没有任何 `关系` 包含，所以我们立刻就有了至少一个形式上的理由来拒绝 `空值` 的概念——但我在本章后面的"为什么禁止空值"一节中也会给出一些更实际的理由。

下一个推论是：==_`元组的每个子集都是元组，表头的每个子集都是表头`_。==例如，给定我们常用的供应商 `S1` 的 `元组`，我们可以在那个 `元组` 中称之为"{`SNO`,`CITY`} `值`"的东西本身是另一个 `元组`：

**图 3-5**

![Figure Three-5](httpatomoreillycomsourceoreillyimages247913.png)

它的 `表头` 如所示，它的 `类型` 是：

```
TUPLE { SNO SNO, CITY CHAR }
```

同样，下面也是一个 `元组`（它的 `类型` 是 `TUPLE {SNO SNO}`）：

**图 3-6**

![Figure Three-6](httpatomoreillycomsourceoreillyimages247836.png)

因此，如果我们想访问实际的 `属性值`——例子中的 `SNO('S1')`——我们必须以某种方式从包含它的 `元组` 中 _`提取`_ 它。**`Tutorial D`** 为此使用形式为 `SNO FROM `_`t`_ 的语法（其中 _`t`_ 是任何表示具有 `SNO 属性` 的 `元组` 的 `表达式`）。`SQL` 使用点限定：_`t`_`.SNO`。

> **注意**
>
> 我们在第 2 章中看到，`元组` _`t`_ 和仅包含那个 `元组` _`t`_ 的 `关系` _`r`_ 不是同一回事。类似地，`值` _`v`_ 和仅包含那个 `值` _`v`_ 的 `元组` _`t`_ 也不是同一回事；特别是，它们具有不同的 `类型`。

现在，我确信你知道空集是每个集合的子集。因此，具有空 `表头` 的 `元组`，因此具有空 `分量` 集，是一个有效的 `元组`！——尽管在纸上画出这样一个 `元组` 的图片有点困难，我甚至不打算尝试。具有空 `表头` 的 `元组` 具有 `类型` `TUPLE{}`（它没有 `分量`）；事实上，我们有时明确地称它为 _`0 元组`_，以强调它是零度的事实。我们有时也称它为 _`空元组`_。现在，你可能会认为这样的 `元组` 在实践中不太可能有多大用处；事实上，事实证明，也许相当令人惊讶的是，它具有至关重要的重要性。我将在本章后面的"TABLE_DUM 和 TABLE_DEE"一节中对此多说一些。

我要在这里讨论的最后一个"重要推论"与 _`元组相等`_[\*](#fn10) 的概念有关。（回想一下第 2 章，`"=" 比较运算符` 为每个 `类型` 定义，`元组类型` 也不例外。）基本上，当然，两个 `元组` 相等当且仅当它们是同一个 `元组`（就像，例如，两个 `整数` 相等当且仅当它们是同一个 `整数`）。但值得明确说明 `元组相等` 的语义，因为 `关系模型` 中的很多东西都依赖于它——例如，`候选键`、`外键` 和 `关系代数` 的大多数 `运算符` 都是根据它定义的。因此，这里是一个精确的定义：

> _`定义：`_ `元组` _`t1`_ 和 _`t2`_ _`相等`_ 当且仅当它们具有相同的 `属性` _`A1, A2, ..., An`_——换句话说，它们是相同的 `类型`——而且，对于所有 _`i`_ (_`i`_ = 1, 2, ..., _`n`_)，_`t1`_ 中 _`Ai`_ 的 `值` _`v1`_ 等于 _`t2`_ 中 _`Ai`_ 的 `值` _`v2`_。

另外（这看起来可能很明显，但需要说明），两个 `元组` 是 _`重复`_ 的当且仅当它们相等。

顺便说一下，根据这个定义可以立即得出，所有 `0 元组` 彼此都是重复的。出于这个原因，我们有权利谈论 _`那个`_ `0 元组` 而不是 _`一个`_ `0 元组`，事实上我们通常就是这样做的。

最后注意，`比较运算符` `"<"` 和 `">"` 不适用于 `元组`（参见本章末尾的练习 3-11）。

## 什么是关系？

我将使用我们常用的供应商 `关系`[\*](#fn11)[\*](#fn12)[\*](#fn13)[\*](#fn14)[\*](#fn15)[\*](#fn16) 作为本节示例的基础。这是一张图片：

**图 3-7**

![Figure Three-7](httpatomoreillycomsourceoreillyimages247841.png)

这里是一个定义：

> _`定义：`_ 设 {_`H`_} 是一个 `元组表头`，设 _`t1, t2`_, . . . _`, tm`_ (_`m`_ ≥ 0) 是具有 `表头` {_`H`_} 的不同 `元组`。{_`H`_} 和 `元组` 集 {_`t1, t2`_, . . . _`, tm`_} 的组合，记作 _`r`_，是 `属性` _`A1, A2`_, . . . _`, An`_ 上的一个 _`关系值`_[\*](#fn17)（或简称 _`关系`_），其中 _`A1, A2`_, . . . _`, An`_ 是 {_`H`_} 中的 `属性`。_`r`_ 的 _`表头`_ 是 {_`H`_}；_`r`_ 具有与该 `表头` 相同的 `属性`（因此相同的 `属性名` 和 `类型`）和相同的 `度`。_`r`_ 的 _`表体`_ 是 `元组` 集 {_`t1, t2`_, . . . _`, tm`_}。`值` _`m`_ 是 _`r`_ 的 _`基数`_。

我将把它作为一个练习，让你根据上述定义解释供应商 `关系`。然而，我至少会解释为什么我们称这些东西为 `关系`。基本上，`关系` 中的每个 `元组` 代表一组 _`n`_ 个 `值`（每个 `元组属性` 一个 `值`）之间的一个 _`n`_ 元 `关系`，在普通的自然语言意义上，而给定 `关系` 中的完整 `元组` 集代表在某个给定时间恰好存在的这种 `关系` 的完整集合——而且，从数学上讲，这就是一个 `关系`。因此，经常听到的"解释"，即 `关系模型` 之所以这样称呼是因为它让我们能够"`关联` 一个 `表` 到另一个 `表`"，虽然在某种次要意义上是准确的，但确实错过了基本点。`关系模型` 之所以这样称呼，是因为它处理某些抽象，我们可以将其视为"`表`"，但在数学上正式称为 `关系`。

现在，`关系`，像 `元组` 一样，本身是一个 `值` 并且有一个 `类型`，而且那个 `类型` 有一个名字[\*](#fn18)。在 **`Tutorial D`** 中，这样的名字采用 `RELATION{`_`H`_`}` 的形式，其中 {_`H`_} 是 `表头`。这里是一个例子：

```
RELATION { SNO SNO, SNAME NAME, STATUS INTEGER, CITY CHAR }
```

（指定 `属性` 的顺序是任意的。）此外，每个 `关系值` 都由某个 `关系选择器`[\*](#fn19) 调用返回——例如：

```
RELATION {
   TUPLE { SNO SNO('S1'), SNAME NAME('Smith'),
                          STATUS 20, CITY 'London' } ,
   TUPLE { SNO SNO('S2'), SNAME NAME('Jones'),
                          STATUS 10, CITY 'Paris'  } ,
   TUPLE { SNO SNO('S3'), SNAME NAME('Blake'),
                          STATUS 30, CITY 'Paris'  } ,
   TUPLE { SNO SNO('S4'), SNAME NAME('Clark'),
                          STATUS 20, CITY 'London' } ,
   TUPLE { SNO SNO('S5'), SNAME NAME('Adams'),
                          STATUS 30, CITY 'Athens' } }
```

指定 `元组` 的顺序是任意的。

`SQL` 对上述思想提供了什么支持？答案是：不多。它实际上根本没有类似于 `关系类型`[\*](#fn20) 概念的东西；相反，`SQL 表` 被认为由 _`行`_（_`多重集`_ 或 _`包`_ 的 `行`，准确地说）组成，这些行具有某个 _`行类型`_。（它确实有它称之为 _`命名表类型`_ 的东西，但这个构造与我们的 `关系类型` 非常不同，我在这里不打算讨论它。）因此，`SQL` 实际上也没有类似于 `RELATION 类型生成器` 的东西。然而，它确实有一个称为 _`表值构造器`_ 的东西，在某种程度上类似于 `关系选择器`。这里是一个例子：

```
VALUES ( 1, 2 ), ( 2, 1 ), ( 1, 1 ), ( 1, 2 )
```

这个 `表达式` 求值为一个具有四行——不是三行！——和两 `列`（没有名字）的 `表`。

## 更多重要的推论

我在第 1 章中谈论的 `关系` 的大多数属性都是上一节定义的直接推论，但有几个我之前没有明确指出，我想详细阐述其中的一些。

我之前没有提到的第一点是 _`表体的每个子集都是表体`_（粗略地说，`关系` 的每个子集都是 `关系`）。特别是，因此，由于空集是每个集合的子集，`关系` 可以有一个由空 `元组` 集组成的 `表体`（我们称这样的 `关系` 为 _`空关系`_[\*](#fn21)）。例如，假设现在没有装运。在这种情况下，`关系变量` `SP` 将具有空装运 `关系` 作为其当前 `值`，我们可以这样画它：[\*](#fn22)

**图 3-8**

![Figure Three-8](httpatomoreillycomsourceoreillyimages247881.png)

注意，给定任何特定的 `关系类型`，恰好有一个该 `类型` 的空 `关系`——但不同 `类型` 的空 `关系` 不是同一回事，确切地说是因为它们具有不同的 `类型`。例如，空供应商 `关系` 不等于空零件 `关系`；它们的 `表体` 相等但它们的 `表头` 不相等。

我故意之前没有提到的第二点是这个。回想一下第 1 章，虽然 `关系` 可以画成 `表` 的图片，但 `关系` _`不是`_ `表`。（再说一次，事物的图片与事物本身不一样。）当然，将 `关系` _`视为`_ `表` 可能是方便的；毕竟，`表` 是"用户友好的"；事实上，正是我们可以非正式地将 `关系` 视为 `表`——有时更明确地视为 _`扁平`_ 或 _`二维`_ `表`——这一事实使得 `关系系统` 在直观上易于理解和使用，并且使得直观地推理这类系统的行为方式变得容易。换句话说，`关系模型` 的一个非常好的属性是它的基本 `数据结构`，`关系`，具有如此直观吸引人的图形表示。

不幸的是，许多人似乎被那个吸引人的图形表示所蒙蔽，认为 _`关系本身`_ 是"`扁平的`"或"`二维的`"。但它们不是。相反，如果 `关系` _`r`_ 有 _`n`_ 个 `属性`，_`那么 r 中的每个元组代表某个 n 维空间中的一个点`_（而且 `关系` 整体代表这样一组点）。例如，出现在我们常用的供应商 `关系变量` `S` 的示例 `值` 中的五个 `元组` 中的每一个都代表某个 4 维空间中的某个点，因此 `关系` 整体可以说是 4 维的。因此，`关系` 是 _`n`_ 维的，不是二维的！[\*](#fn23) 正如我在其他地方写的（事实上在很多地方）：_`让我们都发誓再也不说"扁平关系"了`_。

现在我要转向我想详细阐述的问题。有三个：重复 `元组`、`空值` 和没有 `属性` 的 `关系`。我将在各自的一节中讨论每一个。

## 为什么禁止重复元组

有许多实际的论据支持应该禁止重复 `元组`（简称"`重复`"）的立场。这里我只容纳一个——但我认为这是一个强有力的论据。然而，它确实依赖于我在这本书中还没有讨论过的某些概念，所以我需要做几个初步的假设：

- 我假设你知道 `关系数据库管理系统` 包括一个称为 _`优化器`_[\*](#fn24) 的组件，它的工作是试图找出实现用户查询等的最佳方式（其中"最佳"基本上意味着性能最好）。
- 我还假设你知道优化器做的事情之一是有时称为 _`查询重写`_[\*](#fn25) 的东西。`查询重写` 是将某个 `关系表达式` _`X1`_——代表某个用户查询，比如说——转换为另一个这样的 `表达式` _`X2`_ 的过程，使得 _`X1`_ 和 _`X2`_ 在求值时保证产生相同的结果，但 _`X2`_ 比 _`X1`_ 具有更好的性能特征（至少，我们希望如此）。

现在我可以提出我的论据了。我想提出的基本点是，某些 `表达式转换`，因此某些优化，在 `关系` 上下文中是有效的，但在存在 `重复` 的情况下 _`不`_ 有效。作为例子，考虑图 3-1 中显示的（非 `关系`）`数据库`。

![A nonrelational database, with duplicates](httpatomoreillycomsourceoreillyimages247888.png)

在继续之前，也许我应该问这个问题：在 `表` `P` 中有三个 ``<P1,Screw>`` `行` 而不是两个、四个或十七个意味着什么？[\*](#fn26) 它必须意味着什么，因为如果它什么都不意味着，那么为什么一开始就有 `重复`？正如我曾经听到 Ted Codd 说的，"如果某事是真的，说两次并不会使它更真。"

所以我必须假设有某种意义附加在重复上，尽管那种意义，无论它是什么，几乎不太明确。（因此我顺便指出，`重复` 违反了 `关系模型` 的原始目标之一：_`明确性`_。数据的意义应该尽可能明显和明确，因为我们 supposed 在谈论一个共享 `数据库`。`重复` 的存在意味着部分意义是隐藏的。）换句话说，鉴于 `重复` 确实有某种意义，大概会有基于例如 `表` `P` 中有三个 ``<P1,Screw>`` `行` 而不是两个、四个或十七个这一事实做出的商业决策。如果不是，那么为什么一开始就有 `重复`？

现在考虑以下查询："获取要么是螺丝要么由供应商 `S1` 供应，或者两者都满足的零件号。"这里是这个查询的一些候选 `SQL` 表述，以及每种情况下产生的输出：

```
1  SELECT P.PNO
   FROM   P
   WHERE  P.PNAME = NAME('Screw')
   OR     P.PNO IN
        ( SELECT SP.PNO
          FROM   SP
          WHERE  SP.SNO = SNO('S1') )

   结果：P1 * 3, P2 * 1.

2   SELECT SP.PNO
   FROM   SP
   WHERE  SP.SNO = SNO('S1')
   OR     SP.PNO IN
        ( SELECT P.PNO
          FROM   P
          WHERE  P.PNAME = NAME('Screw') )

   结果：P1 * 2, P2 * 1.

3   SELECT P.PNO
   FROM   P, SP
   WHERE  ( SP.SNO = SNO('S1') AND
            SP.PNO = P.PNO )
   OR     P.PNAME = NAME('Screw')

   结果：P1 * 9, P2 * 3.

4   SELECT SP.PNO
   FROM   P, SP
   WHERE  ( SP.SNO = SNO('S1') AND
            SP.PNO = P.PNO )
   OR     P.PNAME = NAME('Screw')

   结果：P1 * 8, P2 * 4.

5   SELECT P.PNO
   FROM   P
   WHERE  P.PNAME = NAME('Screw')

   UNION  ALL
   SELECT SP.PNO
   FROM   SP
   WHERE  SP.SNO = SNO('S1')

   结果：P1 * 5, P2 * 2.

6   SELECT DISTINCT P.PNO
   FROM   P
   WHERE  P.PNAME = NAME('Screw')
   UNION  ALL
   SELECT SP.PNO
   FROM   SP
   WHERE  SP.SNO = SNO('S1')

   结果：P1 * 3, P2 * 2.

7   SELECT P.PNO
   FROM   P
   WHERE  P.PNAME = NAME('Screw')
   UNION  ALL
   SELECT DISTINCT SP.PNO
   FROM   SP
   WHERE  SP.SNO = SNO('S1')

   结果：P1 * 4, P2 * 2.

8   SELECT DISTINCT P.PNO
   FROM   P
   WHERE  P.PNAME = NAME('Screw')
   OR     P.PNO IN
        ( SELECT SP.PNO
          FROM   SP
          WHERE  SP.SNO = SNO('S1') )

   结果：P1 * 1, P2 * 1.

9   SELECT DISTINCT SP.PNO
   FROM   SP
   WHERE  SP.SNO = SNO('S1')
   OR     SP.PNO IN
        ( SELECT P.PNO
          FROM   P
          WHERE  P.PNAME = NAME('Screw') )

   结果：P1 * 1, P2 * 1.

10   SELECT P.PNO
   FROM   P
   GROUP  BY P.PNO, P.PNAME
   HAVING P.PNAME = NAME('Screw')
   OR     P.PNO IN
        ( SELECT SP.PNO
          FROM   SP
          WHERE  SP.SNO = SNO('S1') )

   结果：P1 * 1, P2 * 1.


11   SELECT P.PNO
   FROM   P, SP
   GROUP  BY P.PNO, P.PNAME, SP.SNO, SP.PNO
   HAVING ( SP.SNO = SNO('S1') AND
            SP.PNO = P.PNO )
   OR     P.PNAME = NAME('Screw')

   结果：P1 * 2, P2 * 2.

12   SELECT P.PNO
   FROM   P
   WHERE  P.PNAME = NAME('Screw')
   UNION
   SELECT SP.PNO
   FROM   SP
   WHERE  SP.SNO = SNO('S1')

   结果：P1 * 1, P2 * 1.
```

> **注意**
>
> 实际上，上述某些表述——哪些？——有点可疑，因为它们假设每个螺丝至少由一个供应商供应。但这个事实对后面的论据没有实质性影响。

明显的第一点是，十二种不同的表述产生了九种不同的结果——也就是说，就它们的 _`重复度`_ 而言是不同的。（顺便说一下，我不声称这十二种不同的表述和九种不同的结果是唯一可能的；事实上，一般来说它们不是。）因此，如果用户真的关心 `重复`，他或她需要 _`极其`_ 小心地表述查询，以获得恰好期望的结果。

此外，当然，类似的说明也适用于系统本身：因为不同的表述可以产生不同的结果，优化器在其 `表达式转换` 任务中也必须极其小心。例如，优化器不能自由地将，比如说，表述 1 转换为表述 3 或反之，即使它想这样做。换句话说，重复 `行` 充当重要的 _`优化抑制剂`_。以下是这个事实的一些含义：

- 优化器代码本身更难编写，更难维护，而且可能更多漏洞——所有这些都结合起来使得产品同时更昂贵和更不可靠，以及延迟交付到市场。
- 系统性能可能比它本来可能的要差。
- 用户将不得不参与性能问题。更具体地说，他们将不得不花费时间和精力来弄清楚如何表述给定的查询以获得最佳性能——这是 `关系模型` expressly 旨在避免的一种状态！

> **使用 SELECT DISTINCT**
>
> 在原始草稿的这一点上，我说如果你发现总是指定 `DISTINCT` 的纪律令人烦恼，不要向我抱怨——向 `SQL` 供应商抱怨 instead。但我的审稿人对我建议你应该总是指定 `DISTINCT` 的反应几乎是 unanimous horror。一个人写道："那些真正精通 `SQL` 的人会对默认编码 `SELECT DISTINCT` 的想法感到震惊。"嗯，我想礼貌地建议，(a) 那些"对这个想法感到震惊"的人可能很了解实现，而不是 `SQL`，而且 (b) 他们的震惊可能是由于他们认识到那些实现在优化掉不必要的 `DISTINCT` 方面做得如此糟糕。如果我写 `SELECT DISTINCT S.SNO FROM S ...`，那个 `DISTINCT` 可以安全地忽略。如果我写 `EXISTS (SELECT DISTINCT ...)` 或 `IN (SELECT DISTINCT ...)`，那个 `DISTINCT` 可以安全地忽略。如果我写 `SELECT DISTINCT SP.SNO FROM SP ... GROUP BY SP.SNO`，那个 `DISTINCT` 可以安全地忽略。如果我写 `SELECT DISTINCT ... UNION SELECT DISTINCT ...`，那些 `DISTINCT` 可以安全地忽略。等等。为什么我，作为一个用户，必须花费时间和精力来弄清楚某个 `DISTINCT` 是否会造成性能打击以及逻辑上是否可以安全地省略它，并且记住 `SQL` 关于何时自动消除 `重复` 以及何时不消除的不一致规则的所有细节？嗯，我可以继续说下去。然而，我决定——违背我自己的更好判断，但为了维持良好的关系（我是说与我的审稿人）——不在本书的其余部分遵循我自己的建议，而只是在逻辑上似乎有必要时才显式请求重复消除。决定什么时候是这样也不总是容易的。但至少现在我可以加入那些向供应商抱怨的人的行列，我想。

鉴于在大多数情况下，用户可能 _`不`_ 关心结果中出现多少 `重复` 这一事实，`重复` 充当 `优化抑制剂` 尤其令人沮丧。换句话说：

- 不同的表述产生不同的结果。
- 但这些差异从用户的角度来看可能是不相关的。
- _`但`_ 优化器不知道后一个事实，因此被不必要地阻止执行它可能想执行的转换。

基于像上述这样的例子，我倾向于得出结论，除其他外，你应该始终确保查询结果不包含 `重复`——例如，通过在 `SQL` 查询中总是指定 `DISTINCT`——从而简单地忘记整个问题。而且如果你遵循这个建议，当然，那么一开始就允许 `重复` 就没有什么好理由了....

关于这个问题还有很多可以说的，但我只有空间再说四点。首先，`SQL` 中 `SELECT DISTINCT` 的替代方案是 `SELECT ALL`——而且 `SELECT ALL` 不幸是默认的。然而，麻烦是，`SELECT DISTINCT` 可能比 `SELECT ALL` 执行时间更长，即使 `DISTINCT` 实际上是一个"无操作"。我不想再进一步讨论这个，除了注意 `SQL 系统` 通常无法在重复消除上进行适当优化的原因是它们缺乏关于 _`键继承`_ 的知识（即，它们无法找出任意 `关系表达式` 结果的 `键`）。

其次，你可能会反对，并非不合理地，`基表` 在实践中从不包括 `重复`，因此我的例子直观上失败了。确实如此；但麻烦是，`SQL` 可以在查询结果中 _`生成`_ `重复`。事实上，同一查询的不同表述可以产生具有不同重复度的结果，正如我们已经看到的，即使输入 `表` 本身根本没有 `重复`。例如，考虑以下对查询"获取至少供应一个零件的供应商号"在常用的供应商和零件 `数据库` 上的表述：

```
SELECT S.SNO           |    SELECT S.SNO
FROM   S               |    FROM   S, SP
WHERE  S.SNO IN        |    WHERE  S.SNO = SP.SNO
     ( SELECT SP.SNO
       FROM   SP )
```

（给定我们常用的示例数据值，这两个 `表达式` 产生什么结果？）所以如果你不想具体地将图 3-1 中的 `表` 视为 `基表`，那很好：只需将它们视为先前查询的输出，其余的分析保持不变。

第三，还有另一个至少是心理上的反对 `重复` 的论据，我认为相当有说服力（感谢 Jonathan Gennick 提供这个）：如果，按照上一节介绍的关于 `关系` 的 _`n`_ 维视角，你将 `表` 视为某个 _`n`_ 维空间中点的绘图，那么重复 `行` 显然不增加任何东西；它们 simply 相当于将同一个点绘图两次。

我的最后一点是这个。假设 `表` _`T`_ 确实允许 `重复`。那么我们无法区分 _`T`_ 中的"真正的" `重复` 和由 _`T`_ 上数据录入错误产生的 `重复`！例如，如果负责数据录入的人无意中——即，_`错误地`_——将完全相同的 `行` 输入到 _`T`_ 两次，会发生什么？（顺便说一下，很容易做到。）有没有办法只删除"第二" `行` 而不删除"第一"行？注意我们大概确实想删除那个"第二" `行`，因为它一开始就不应该被输入。

## 为什么禁止空值

上一节的开头段落同样适用于这里（只有一个微小的文本替换），所以我基本上只是重复它：有许多实际的论据支持应该禁止 `空值` 的立场。这里我只容纳一个——但我认为这是一个强有力的论据。但它确实依赖于我在这本书中还没有讨论过的某些概念，所以我需要做几个初步的假设：

- 我假设你知道任何 `比较`，其中任一或两个比较对象是 `空值`，求值为 `UNKNOWN 真值` 而不是 `TRUE` 或 `FALSE`。这种状态的正当理由是将 `空值` 解释为 _`值未知：`_ 如果 _`A`_ 的 `值` 是未知的，那么显然不知道，例如，_`A`_ > _`B`_，无论 _`B`_ 的 `值` 是什么（即使——也许特别是——如果 _`B`_ 的 `值` 也是未知的）。顺便说一下，正是这个事实是术语 _`三值逻辑`_ (`3VL`) 的来源：`空值` 的概念不可避免地导致我们进入一个 `逻辑`，其中有三个 `真值` 而不是通常的两个。（相比之下，`关系模型` 基于传统的 _`二`_ 值 `逻辑`，`2VL`。）
- 我还假设你知道熟悉的 `运算符` `NOT`、`AND` 和 `OR` 的 `3VL 真值表`[\*](#fn27)（`T` = `TRUE`，`F` = `FALSE`，`U` = `UNKNOWN`）：

  **图 3-10**

  ![Figure Three-10](httpatomoreillycomsourceoreillyimages247911.png)

  特别观察，如果其输入是 `UNKNOWN`，`NOT` 返回 `UNKNOWN`；如果一个输入是 `UNKNOWN` 而另一个是 `UNKNOWN` 或 `TRUE`，`AND` 返回 `UNKNOWN`；如果一个输入是 `UNKNOWN` 而另一个是 `UNKNOWN` 或 `FALSE`，`OR` 返回 `UNKNOWN`。

现在我可以提出我的论据了。我想提出的基本点是，某些 `布尔表达式`——因此某些查询——产生根据 `三值逻辑`[\*](#fn28) 正确但 _`不`_ 在现实世界中正确的结果。作为例子，考虑图 3-2 中显示的（非 `关系`）`数据库`，其中零件 `P1` 的"`CITY` 是 `空值`"。仔细注意该图中的空白空间，在零件 `P1` 的 `CITY 值` 应该在的地方，代表 _`什么都没有`_；从概念上讲，在那个位置没有任何东西——甚至不是一串空格或空字符串（这意味着零件 `P1` 的"`元组`"实际上不是一个 `元组`，我将在本节末尾附近回到这一点）。

**图 3-2. 一个非关系数据库，带有一个空值**

![A nonrelational database, with a null](httpatomoreillycomsourceoreillyimages247909.png)

现在考虑以下（ admittedly rather contrived）对图 3-2 的 `数据库` 的查询："获取 `SNO`-`PNO` 对，其中供应商和零件城市不同或者零件城市不是巴黎（或两者）。"这里是这个查询的明显 `SQL` 表述：[\*](#fn29)

```
SELECT S.SNO, P.PNO
FROM   S, P
WHERE  S.CITY <> P.CITY
OR     P.CITY <> 'Paris'
```

让我们专注于 `WHERE` 子句中的 `布尔表达式`（为了清晰添加括号）：

```
( S.CITY <> P.CITY ) OR ( P.CITY <> 'Paris' )
```

对于我们 `数据库` 中仅有的数据，这个 `表达式` 求值为 `UNKNOWN OR UNKNOWN`，简化为仅仅是 `UNKNOWN`。现在，`SQL` 查询检索 `WHERE 子句` 中的 `表达式` 求值为 `TRUE` 的数据，不是 `FALSE` 也不是 `UNKNOWN`；因此，在例子中，根本没有检索到任何东西。

但当然零件 `P1` 在现实世界中确实有 _`某个`_ 对应的城市；换句话说，零件 `P1` 的"`空值 CITY`"确实代表某个真实的 `值`，比如 _`xyz`_。显然，要么 _`xyz`_ 是巴黎，要么不是。如果是，那么 `表达式`：

```
( S.CITY <> P.CITY ) OR ( P.CITY <> 'Paris' )
```

变成（对于我们仅有的数据）：

```
( 'London' <> 'Paris' ) OR ( 'Paris' <> 'Paris' )
```

这个 `表达式` 求值为 `TRUE`，因为第一项求值为 `TRUE`。或者，如果 _`xyz`_ 不是巴黎，`表达式` 变成（再次，对于我们仅有的数据）：

```
( 'London' <> xyz ) OR ( xyz <> 'Paris' )
```

这个 `表达式` 也求值为 `TRUE`，因为第二项求值为 `TRUE`。因此，`布尔表达式` 在现实世界中总是 `TRUE`，查询应该返回对 `S1`-`P1`，_`无论空值代表什么真实值`_。换句话说，根据 `逻辑`（即，`3VL`）正确的结果和在现实世界中正确的结果是不同的！

作为另一个例子，考虑以下查询（我没有用这个例子开头，因为它甚至比前一个更牵强，但在某些方面它仍然更有力地说明了这一点）：

```
SELECT P.PNO
FROM   P
WHERE  P.CITY = P.CITY
```

这里的现实世界答案显然是当前出现在 `P` 中的零件号集合（换句话说，给定图 3-2 中显示的示例数据，仅包含零件号 `P1` 的集合）。`SQL`，然而，将不返回任何零件号。

总结一下：如果你的 `数据库` 中有任何 `空值`，你的一些查询会得到错误的答案。而且，你一般来说无法知道，你究竟哪些查询得到了错误的答案；_`所有`_ 结果都变得可疑。_`你永远不能信任带有空值的数据库给出的答案`_。在我看来，这种状态完全是一个 showstopper。

与上一节中 `重复` 的事情一样，关于这个问题还有很多可以说的，但我只想以回顾反对 `空值` 的 _`形式`_ 论据来结束。回想一下，根据定义，`空值` 不是 `值`。因此：[\*](#fn30)

- 包含 `空值` 的"`类型`"不是 `类型`（因为 `类型` 包含 `值`）。
- 包含 `空值` 的"`元组`"不是 `元组`（因为 `元组` 包含 `值`）。
- 包含 `空值` 的"`关系`"不是 `关系`（因为 `关系` 包含 `元组`，而且 `元组` 不包含 `空值`）。

结果是，如果 `空值` 存在，那么我们肯定不是在谈论 `关系模型`（我不知道我们在 _`是`_ 谈论什么，但它不是 `关系模型`）；整个大厦崩溃，所有赌注都 off。

## TABLE_DUM 和 TABLE_DEE

前两节也许有点令人沮丧；相比之下，这一节要乐观得多。回想一下"一些重要的推论"一节，空集是每个集合的子集，因此存在空 `元组`（也称为 `0 元组`）这样的东西，而且当然它有一个空 `表头`[\*](#fn31)。我之前没有指出的是——足够明显——`关系` 也可能有一个空 `表头`（`表头` 是 `属性` 的集合，而且没有理由为什么那个集合不能是空的）。这样的 `关系` 是 `类型` `RELATION{}`，而且它的 `度` 是零。它有时也被说成是 0 元（或 _`零元`_），就像，例如，度为二的 `关系` 被说成是二元一样。（顺便说一下，这里的 _`零元`_ 与 `空值` 没有任何关系；`空值` 仍然是坏消息，但 _`零元关系`_ 确实是非常好的消息。然而，我更喜欢避免术语 _`零元`_，确切地说是因为它听起来好像与 `空值` 有关系。）

设 _`r`_ 是一个度为零的 `关系`。那么，有多少这样的 `关系`？答案是：只有两个。首先，当然，_`r`_ 可能是空的（意味着它不包含 `元组`）——记住任何给定 `类型` 恰好有一个空 `关系`。其次，如果 _`r`_ 不是空的，那么它包含的 `元组` 必须都是 `0 元组`。但只有一个 `0 元组`！——等价地，所有 `0 元组` 彼此都是重复的——因此 _`r`_ 不可能包含多于一个它们。因此确实只有两个没有 `属性` 的 `关系`：一个只有一个 `元组`，一个根本没有 `元组`。出于相当明显的原因，我不打算尝试画这些 `关系` 的图片；事实上，这是将 `关系` 视为 `表` 的想法确实 breakdown 的一个地方。

现在，你很可能在想："那又怎样？我究竟为什么会想要一个根本没有 `属性` 的 `关系`？"即使它们在数学上是体面的（它们确实是），它们肯定没有 _`实际`_ 意义？事实上，然而，事实证明它们具有非常重大的实际意义：如此之大，以至于我们有宠物的名字给它们——我们称它们为 `TABLE_DUM` 和 `TABLE_DEE`，或简称 `DUM` 和 `DEE`（`DUM` 是空的那个，`DEE` 是有一个 `元组` 的那个）。[\*](#fn32) 它们如此重要的原因是它们的 _`含义`_，对于 `DUM` 是 `FALSE`（或 `NO`），对于 `DEE` 是 `TRUE`（或 `YES`）。它们具有最基本的含义。

> **注意**
>
> 我将在下一章讨论 `关系` 一般来说意味着什么的整个概念。

顺便说一下，记住哪个是哪个的一个好方法是这个：`YES` 和 `DEE` 都有一个"E"；`NO` 和 `DUM` 没有。

我在这本书中还没有涵盖足够的内容来展示 `DUM` 和 `DEE` 的具体示例，可以说，但我们将在后面的页面中看到大量它们使用的示例。这里我只提到一点，在这个早期阶段至少应该直观上有意义：特别是 `DEE` 在 `关系代数` 中扮演的角色类似于零在传统算术中扮演的角色。而且我们都知道零有多重要；事实上，很难想象没有零的算术（古罗马人尝试过，但没有走得很远）。嗯，想象一个没有 `TABLE_DEE` 的 `关系代数` 应该同样困难。

## 总结

在本章中，我提出了基本概念 _`元组`_ 和 _`关系`_ 的精确定义。正如我在引言部分所说的，这些定义一开始可能有点令人生畏，但我希望你在阅读了前两章后能够理解它们。我还讨论了 `元组` 和 `关系类型`，并展示了一些 `元组` 和 `关系选择器调用` 的示例。我还讨论了定义的一些重要推论：

- 没有 `元组`，因此没有 `关系`，曾经包含任何 `空值`。
- `元组` 的每个子集都是 `元组`，`表头` 的每个子集都是 `表头`，`表体` 的每个子集都是 `表体`（而且那些子集总是可以为空）。
- 两个 `元组` 相等当且仅当它们是同一个 `元组`。

前面章节已经涵盖的其他推论包括：

- 每个 `元组` 为其每个 `属性` 包含恰好一个 `值`，因此 `关系` 总是在 `第一范式`。
- `元组` 的 `属性`，因此 `关系` 的 `属性`，从左到右是无序的。
- `关系` 的 `元组` 也是从上到下无序的。
- `关系` 从不包含任何重复 `元组`。

我还给出了一些重要的实际论据，支持禁止 `重复` 和 `空值`。最后，我简要介绍了非常重要的零度 `关系` `TABLE_DUM` 和 `TABLE_DEE`；`DUM` 意味着 `NO` 或 `FALSE`，`DEE` 意味着 `YES` 或 `TRUE`。
