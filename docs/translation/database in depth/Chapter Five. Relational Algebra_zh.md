# 第五章 关系代数

我将在本章开头简要回顾第一章的几个重要要点。首先，我要提醒的是，每个代数`运算符`至少接受一个`关系`作为输入，并产生另一个`关系`作为输出。其次，我要提醒的是，输出与输入属于同一类事物——它们都是`关系`——这一事实正是该代数的*`闭包`*性质，也正是这一性质使得我们能够编写嵌套的`关系表达式`。第三，我在第一章中概述了我所称的“原始八个运算符”（`restrict`、`project`、`product`、`intersect`、`union`、`difference`、`join`和`divide`）；现在，我想更严谨地定义这些运算符，以及其他一些运算符。不过，在我详细说明之前，需要先明确几点一般性原则：

- 首先，这些运算符是*`通用的`*：它们实际上适用于*`所有可能的关系`*。例如，我们不需要一个特定的`连接运算符`来连接员工和部门，也不需要另一个不同的`连接运算符`来连接供应商和发货记录。（顺便问一下，你认为面向对象系统是否也有类似的观察结论？）
- 其次，这些运算符是*`只读`*的：它们“读取”操作数并返回结果，但不会更新任何内容。换句话说，它们作用于*`关系`*，而不是*`关系变量`*。
- 当然，前一点并不意味着`关系表达式`不能引用`关系变量`。例如，_`R`_ `UNION` _`S`_（其中*`R`*和*`S`*是`关系变量`名）在**`Tutorial D`**中绝对是一个有效的`关系表达式`。然而，在此类表达式的上下文中，`关系变量`名并不表示该`关系变量`本身；而是表示该`关系变量`在当时恰好具有的当前值所对应的`关系`。换句话说，我们当然可以使用`关系变量`名来表示一个`关系`操作数，而这种*`关系变量引用`*因此构成了一个有效的`关系表达式`——但从原理上讲，我们同样可以通过适当的`关系字面量`来表示完全相同的操作数。（一个类比可能有助于澄清这一点。假设 N 是一个类型为`INTEGER`的变量，在时间*`t`*它的值为 3。那么 N+2 当然是一个有效的表达式，但在时间*`t`*，它的含义与 3+2 完全相同，不多也不少。）
- 最后，鉴于代数中的运算符都是只读的，因此可以推断，`INSERT`、`DELETE`和`UPDATE`（以及`关系赋值`）虽然*`确实`*是`关系运算符`，但它们本身并不属于该代数的一部分。

我还需要谈谈**`Tutorial D`**的设计，因为它对代数的支持在某些重要方面与`SQL`不同（也许更准确的说法是，它对代数的支持——当然是有意为之的——比`SQL`直接得多）。核心要点是，在需要建立操作数`属性`之间对应关系的操作（如`UNION`或`JOIN`）中，**`Tutorial D`**要求这些`属性`_`必须具有相同的名称`_（并且必然具有相同的类型）。例如，以下是零件和供应商按城市进行`连接`的**`Tutorial D`**表达式：

```
P JOIN S
```

根据定义，此`连接`是基于零件和供应商的城市执行的，因为 P 和 S 恰好只有`CITY`这个`属性`是共有的。相比之下，以下是`SQL`中的相同操作（请特别注意最后一行）：

```
SELECT P.PNO, P.PNAME, P.COLOR, P.WEIGHT, P.CITY,
S.SNO, S.SNAME, S.STATUS
FROM   P,S
WHERE  P.CITY = S.CITY
```

_`注`_：实际上，这个示例在`SQL`中可以有多种不同的表述方式，以下是另外三种。正如你所见，后两种更接近**`Tutorial D`**的精神（请特别注意，在这两种表述中，`连接`的结果只有一列名为 CITY，而*`没有`*名为 S.CITY 或 P.CITY 的列）：

```
SELECT P.PNO, P.PNAME, P.COLOR, P.WEIGHT, P.CITY,
S.SNO, S.SNAME, S.STATUS
FROM   P JOIN S
ON   P.CITY = S.CITY
SELECT P.PNO, P.PNAME, P.COLOR, P.WEIGHT, CITY,
S.SNO, S.SNAME, S.STATUS
FROM   P JOIN S
USING  ( CITY )
SELECT P.PNO, P.PNAME, P.COLOR, P.WEIGHT, CITY,
S.SNO, S.SNAME, S.STATUS
FROM   P NATURAL JOIN S
```

我之所以选择这种特定的表述方式，部分原因是它是`SQL`最初定义时唯一支持的形式，而更重要的原因是，它允许我就`SQL`与代数（至少如在**`Tutorial D`**中实现的那样）之间的差异提出一些额外的观点：

- `SQL`允许（有时要求）使用点限定的名称。**`Tutorial D`**不允许。
- **`Tutorial D`**有时需要重命名`属性`，以避免否则会出现的命名冲突或不匹配。`SQL`通常不需要（尽管它支持类似于**`Tutorial D`**用于此目的的`RENAME`运算符的对应物，我们将在下一节看到）。
- 部分作为前一点的结果，**`Tutorial D`**不需要`SQL`的“相关名”（实际上，它用`属性`有时需要重命名以“消除歧义”否则语法无效的表达式这一概念，取代了`SQL`的相关名概念）。
- 除了显式或隐式地支持`关系代数`的某些特性外，`SQL`还显式支持`关系演算`的某些特性（`EXISTS`就是一个例子——见附录 A）。**`Tutorial D`**不支持。这种差异的一个后果是，`SQL`往往是一种相当冗余的语言，因为它经常提供多种不同的方式来表述同一个查询（这一事实可能对优化器产生严重的负面影响）。
- `SQL`要求大多数查询符合其`SELECT - FROM - WHERE`模板。**`Tutorial D`**没有类似的要求。我将在本章后面的“扩展与汇总”一节中进一步讨论这个具体问题。

在下文中，我将同时展示**`Tutorial D`**和`SQL`的大多数示例。

## 关于闭包的进一步探讨

当我说每个代数运算的输出是另一个`关系`时，我希望明确的是，我是*`从概念层面`*来谈论的。我并不意味着系统实际上必须完整地物化（materialize）该输出。例如，考虑以下表达式（一个`连接`的`限制`——左侧为**`Tutorial D`**，右侧为`SQL`）：

```
( P JOIN S )             |  SELECT P.*,
WHERE PNAME > SNAME   |     S.SNO, S.SNAME, S.STATUS
                      |  FROM   P, S
                      |  WHERE  P.CITY = S.CITY
                      |  AND  P.PNAME > S.PNAME
```

显然，一旦形成`连接`的某个给定`元组`，系统就可以立即针对`限制`条件`PNAME > SNAME`（`SQL`版本中的`P.PNAME > S.SNAME`）测试该`元组`，以查看它是否属于最终输出，如果不是则丢弃它。因此，作为`连接`输出的中间结果可能根本不会作为完全物化的`关系`存在。（当然，出于明显的性能原因，系统实际上会竭尽全力*`不`*完整物化中间结果。）

然而，前面的例子引出了另一个重要观点。考虑**`Tutorial D`**版本中的`布尔表达式` `PNAME > SNAME`。该表达式在概念上应用于`P JOIN S`的结果，因此该表达式中的`属性`名`PNAME`和`SNAME`引用的是该结果的`属性`——_`而不是`_`关系变量` P 和 S 中同名`属性`。但我们如何知道结果包含任何此类`属性`？该结果的*`关系头`*究竟是*`什么`*？更一般地说，我们如何知道*`任何`*代数运算结果的关系头是什么？显然，我们需要一组*`推断规则`*——更具体地说，是*`关系类型推断规则`*——这样，如果我们知道任何给定运算的输入`关系`的类型，我们就能推断出该运算输出`关系`的类型。例如，在`连接`的情况下，这些规则指出`P JOIN S`的输出属于以下类型：

```
RELATION { PNO PNO, PNAME NAME, COLOR COLOR, WEIGHT WEIGHT,
CITY CHAR, SNO SNO, SNAME NAME, STATUS INTEGER }
```

事实上，对于`连接`，输出的`关系头`是输入`关系头`的*`并集`*——这里我所说的*`并集`*是指常规的集合论并集，而不是我将在本章后面讨论的特殊`关系并`。换句话说，输出包含输入的所有`属性`，但共有的`属性`——在示例中仅为 CITY——只出现一次，而不是两次。当然，这些`属性`没有任何从左到右的顺序，因此我同样可以说（例如）`P JOIN S`结果的类型是：

```
RELATION { SNO SNO, PNO PNO, SNAME NAME, PNAME NAME, CITY CHAR,
STATUS INTEGER, WEIGHT WEIGHT, COLOR COLOR }
```

请注意，某种类型的推断规则对于充分支持`闭包`性质是绝对必要的。`闭包`指出每个结果都是一个`关系`，而`关系`不仅有`关系体`，还有`关系头`；因此，每个结果必须具有正确的`关系头`以及正确的`关系体`。

本章开头提到的`RENAME`运算符在很大程度上就是为了支持`关系模型`的类型推断规则。`RENAME`接受一个`关系`作为输入并返回另一个`关系`作为输出；输出`关系`与输入`关系`完全相同，只是其中一个`属性`的名称不同。例如（左侧为**`Tutorial D`**，右侧为`SQL`）：

```
S RENAME ( CITY AS SCITY )  | SELECT S.SNO, S.SNAME, S.STATUS,
                            |        S.CITY AS SCITY
                            | FROM   S
```

那么，`SQL`是如何处理指定结果`属性`名（或者说列名）的呢？答案是：处理得并不好。首先，我们在第 3 章看到，`SQL`实际上根本没有“`关系类型`”的概念（它有的是`行类型`）。其次，它可以产生实际上根本没有名称的列的结果（例如，考虑 `SELECT DISTINCT P.WEIGHT * 2 FROM P`）。第三，它还可以产生具有重复列名的结果（例如，考虑 `SELECT DISTINCT P.CITY, S.CITY FROM P, S`）。最后，让我们再看一遍本节开头的示例：

```
( P JOIN S )             |  SELECT P.*,
WHERE PNAME > SNAME   |     S.SNO, S.SNAME, S.STATUS
                      |  FROM   P, S
                      |  WHERE  P.CITY = S.CITY
                      |  AND  P.PNAME > S.PNAME
```

正如你所见，**`Tutorial D`**的`PNAME > SNAME`在`SQL`版本中的对应部分是`P.PNAME > S.SNAME`——这很奇怪，因为该表达式本应应用于`FROM`子句的结果，而`关系变量` P 和 S 显然不是该结果的一部分！事实上，很难解释像`WHERE`和`SELECT`子句（以及可能在整体表达式其他地方）中的`P.PNAME`如何能与`FROM`子句的结果有任何意义上的关联。`SQL标准`确实对此进行了解释，但它为做到这一点所必须进行的复杂推导比**`Tutorial D`**的规则*`复杂得多`*——以至于我甚至不会在这里尝试解释它们，而是仅仅依赖这样一个事实：如果有必要，它们是可以被解释的。我之所以省略它，是基于本书应该是关于`关系模型`而不是`SQL`的事实。

现在，我想继续描述其他一些代数运算符。请注意，我下面的内容并不试图穷尽；我不会涵盖“所有已知运算符”，甚至不会完全通用地定义我涵盖的所有运算符。事实上，在大多数情况下，我只会给出一个严谨但略显非正式的定义，并展示一些简单示例。

## 原始运算符

本节给出了Codd定义的原始`关系运算符`集的定义：`restrict`（限制）、`project`（投影）、`join`（连接）、`intersect`（交）、`union`（并）、`difference`（差）、`product`（积）和`divide`（除）。

### 限制 (Restrict)

设*`bx`*为一个涉及零个或多个`属性`名的`布尔表达式`，使得所有提及的`属性`都是同一`关系`*`r`*的`属性`。则*`r`*根据*`bx`*的*`限制`*：

```
r WHERE bx
```

是一个与*`r`*具有相同`关系头`的`关系`，其`关系体`由*`r`*中所有使*`bx`*求值为 TRUE 的`元组`组成。例如：

```
S WHERE CITY = 'Paris'  | SELECT S.*
                        | FROM   S
                        | WHERE  S.CITY = 'Paris'
```

顺便提一下，我指出`restrict`有时也被称为*`select`*；但我更倾向于不使用这个术语，以免与`SQL`的`SELECT`混淆。

### 投影 (Project)

设`关系`*`r`*具有`属性`_`X, Y, . . . , Z`_（可能还有其他）。则*`r`*在*`X, Y, . . . , Z`*上的*`投影`*：

```
r { X, Y, ..., Z }
```

是一个`关系`，其 (a) `关系头`源自*`r`*的`关系头`，但移除了集合 {_`X,Y, . . . ,Z`_} 中未提及的所有`属性`，以及 (b) `关系体`由所有`元组`{_`X x,Y y, . . . ,Z z`_}组成，使得*`r`*中存在一个`元组`，其*`X`*值为*`x`*，*`Y`*值为*`y`*，...，*`Z`*值为*`z`*。例如：

```
S { SNAME, CITY, STATUS }  | SELECT DISTINCT S.SNAME,
                           |        S.CITY, S.STATUS
                           | FROM   S
```

重申一下，结果是一个`关系`；因此，用常用的话说，“重复项会被消除”，所以`SQL`表述中的`DISTINCT`确实是必需的。

顺便说一下，**`Tutorial D`**还允许通过指定要丢弃的`属性`来表达`投影`，而不是指定要保留的。因此，例如，**`Tutorial D`**表达式：

```
S { SNAME, CITY, STATUS }
```

和：

```
S { ALL BUT SNO }
```

是等价的。此功能可以节省大量书写（想想对一个度为 100 的`关系`在其 99 个`属性`上进行投影）。类似的评论也适用于`SUMMARIZE`（仅限 BY 形式）和`GROUP`（分别参见后面的“扩展与汇总”和“分组与取消分组”章节）。

在具体语法中，给`投影运算符`赋予较高的优先级会很方便。例如，在**`Tutorial D`**中，我们取这个表达式：

```
S JOIN P { PNO, CITY }
```

意思是：

```
S JOIN ( P { PNO, CITY } )
```

而不是：

```
( S JOIN P ) { PNO, CITY }
```

作为一个练习，请结合我们常用的示例数据，展示这两种解释之间的差异。

### 连接 (Join)

设`关系`*`r`*和*`s`*分别具有`属性`_`X1`_, _`X2`_, . . . , _`Xm`_, _`Y1`_, _`Y2`_, ..., *`Yn`*和*`Y1`*, _`Y2`_, . . . , _`Yn`_, _`Z1`_, _`Z2`_, . . . , _`Zp`_；换句话说，_`Y`_（共*`n`*个）是共有的`属性`，_`X`_（共*`m`*个）是*`r`*的其他`属性`，而*`Z`*（共*`p`*个）是*`s`*的其他`属性`。我们可以不失一般性地假设，由于`RENAME`的可用性，*`X`*中的任何名称都不会与*`Z`*中的任何名称相同。现在，将*`X`*整体简记为*`X`*，*`Y`*和*`Z`*同理。则*`r`*和*`s`*的*`自然连接`*（简称*`连接`*）：

```
r JOIN s
```

是一个`关系`，其 (a) `关系头`是*`r`*和*`s`*的`关系头`的（集合论）`并集`，以及 (b) `关系体`由所有`元组`*`t`*组成，使得*`t`*是出现在*`r`*中的一个`元组`与出现在*`s`*中的一个`元组`的（集合论）`并集`。换句话说，`关系头`是{_`X,Y,Z`_}，`关系体`由所有`元组`{_`X x,Y y,Z z`_}组成，使得*`r`*中存在一个`元组`，其*`X`*值为*`x`*且*`Y`*值为*`y`*，并且*`s`*中存在一个`元组`，其*`Y`*值为*`y`*且*`Z`*值为*`z`*。以下是介绍部分的示例再次出现：

```
P JOIN S  | SELECT P.PNO, P.PNAME, P.COLOR, P.WEIGHT, P.CITY,
          |        S.SNO, S.SNAME, S.STATUS
          | FROM   P, S
          | WHERE  P.CITY = S.CITY
```

现在，我很确定你已经熟悉我到目前为止关于`join`的论述，但你可能对以下几点不太熟悉。首先，我提醒你，`SQL标准`允许用一种更接近**`Tutorial D`**的替代风格来表达零件和供应商按城市的连接（这次我故意用简单的“\*”替换了`SELECT`子句中的长列引用列表）：

```
SELECT *
FROM   P NATURAL JOIN S
```

然而，并非所有`SQL`产品都实际支持此语法。

其次，更重要的是，*`交`*是`join`的一个特例。具体而言，如果*`m`* = _`p`_ = 0（意味着没有*`X`*也没有*`Z`*，且*`r`*和*`s`*因此类型相同），_`r`_ `JOIN` *`s`*退化为*`r`* `INTERSECT` _`s`_（见下一小节“交”）。

第三（同样重要），*`笛卡尔积`*也是`join`的一个特例：如果*`n`* = 0，意味着没有*`Y`*，且*`r`*和*`s`*因此完全没有共有的`属性`，_`r`_ `JOIN` *`s`*退化为*`r`* `TIMES` _`s`_（见后面的小节“笛卡尔积”）。为什么？因为 (a) 在这种情况下，*`r`*和*`s`*共有的`属性`集合是*`空`*`属性`集；(b) 每个可能的`元组`对于空`属性`集都具有相同的值（即 0-`元组`）；因此，(c) *`r`*中的每个`元组`都与*`s`*中的每个`元组`连接，从而得到所述的`笛卡尔积`。

第四，实践表明，许多需要`join`的查询实际上需要该运算符的一种扩展形式，称为*`半连接`*（_`semijoin`_）。（你可能以前没听说过`semijoin`，但它实际上非常重要。）定义如下：*`r`*与*`s`*的*`半连接`*是*`r`*和*`s`*的`连接`，然后投影回*`r`*的`属性`上。例如，考虑查询“_`获取至少供应一个零件的供应商`_”：

```
S SEMIJOIN SP  | SELECT DISTINCT S.*
               | FROM   S, SP
               | WHERE  S.SNO = SP.SNO
```

正如你所见，此查询可以理解为仅请求那些至少匹配一个发货记录的供应商。**`Tutorial D`**因此为`SEMIJOIN`提供了一种更友好的拼写方式，允许将查询表述为：

```
S MATCHING SP
```

现在观察如果*`p`* = 0（意味着*`s`*的`关系头`是*`r`*的`关系头`的子集——等价地，*`s`*的每个`属性`都是*`r`*的`属性`），_`r`_ `JOIN` *`s`*会发生什么。正如我希望你能看到的，在这种情况下，_`r`_ `JOIN` *`s`*退化为*`r`* `MATCHING` _`s`_。同样，如果*`m`* = 0，_`r`_ `JOIN` *`s`*退化为*`s`* `MATCHING` _`r`_（请注意，_`r`_ `MATCHING` *`s`*和*`s`* `MATCHING` *`r`*在一般情况下是不同的）。

第五，`join`本质上是一个*`二元`*运算符，意味着它接受两个操作数。然而，支持*`n`*元或前缀版本的运算符是可能且有用的（**`Tutorial D`**也确实支持），据此我们可以编写如下形式的表达式：

```
JOIN { r, s, ..., w }
```

以连接任意数量的`关系`_`r, s, . . . ,w`_。例如，零件和供应商的连接也可以表述为：

```
JOIN { P, S }
```

更重要的是，我们可以使用此语法来请求仅一个`关系`的“`连接`”，甚至零个`关系`的“`连接`”！单个`关系`的`连接`，`JOIN{`_`r`_`}`，就是*`r`*本身；这种情况实际重要性可能不大。但可能令人惊讶的是，零个`关系`的`连接`，`JOIN{}`，确实非常重要！——其结果是`TABLE_DEE`。（回想一下，`TABLE_DEE`是唯一没有`属性`且只有一个`元组`的`关系`。）为什么结果是`TABLE_DEE`？好吧，考虑以下几点：

- 在普通算术中，0 是“+”的*`单位元`*；也就是说，对于所有数字*`x`*，表达式*`x`* + 0 和 0 + *`x`*都恒等于*`x`*。因此，_`零个数字的和是 0`_。（要理解这一主张的合理性，请考虑一段代码，它通过将和初始化为 0 然后迭代那*`n`*个数字来计算*`n`*个数字的和。如果*`n`* = 0 会发生什么？）
- 同样，1 是“\*”的单位元；也就是说，对于所有数字*`x`*，表达式*`x`\* 1 和 1 \* *`x`_都恒等于_`x`\_。因此，零个数字的积是 1。
- 在`关系代数`中，_`TABLE_DEE是JOIN的单位元`_；也就是说，任何`关系`*`r`*与`TABLE_DEE`的`连接`就是*`r`*本身。因此，零个`关系`的`连接`是`TABLE_DEE`。

如果你对这个概念感到困惑，暂时不必过于担心。但如果你以后回来重读本节，我建议你试着让自己相信*`r`* `JOIN TABLE_DEE`和`TABLE_DEE JOIN `*`r`*确实都恒等于*`r`*。如果我指出所讨论的`连接`实际上是`笛卡尔积`，可能会对你有所帮助（对吧？）。

### 交 (Intersect)

设`关系`*`r`*和*`s`*类型相同。则这些`关系`的*`交`*，_`r`_ `INTERSECT` _`s`_，是一个同类型的`关系`，其`关系体`由所有`元组`*`t`*组成，使得*`t`*同时出现在*`r`*和*`s`*中。例如：

```
S { CITY }    | SELECT DISTINCT S.CITY
INTERSECT   | FROM   S
P { CITY }  | INTERSECT
            | SELECT DISTINCT P.CITY
            | FROM   P
```

（实际上，在`SQL`版本中我们不需要那些`DISTINCT`，但我更喜欢包含它们以保持明确性。见即将讨论的`union`。）

正如我们已经看到的，`intersect`实际上只是`join`的一个特例。然而，**`Tutorial D`**和`SQL`都支持它，即使仅仅是出于心理原因。**`Tutorial D`**还支持*`n`*元或前缀形式，但我将在此跳过细节。

### 并 (Union)

同样，设`关系`*`r`*和*`s`*类型相同。则这些`关系`的*`并`*，_`r`_ `UNION` _`s`_，是一个同类型的`关系`，其`关系体`由所有`元组`*`t`*组成，使得*`t`*出现在*`r`*或*`s`*或两者中。例如：

```
S { CITY } UNION P { CITY }  | SELECT DISTINCT S.CITY
                             | FROM   S
                             | UNION  DISTINCT
                             | SELECT DISTINCT P.CITY
                             | FROM   P
```

与`projection`一样，值得注意的是在`union`中“重复项会被消除”。然而，在`SQL`版本中，关键字`UNION`后的那个`DISTINCT`*`并非`*严格必需；尽管`UNION`提供与`SELECT`相同的选项（`DISTINCT`对比`ALL`），但`UNION`的默认值是`DISTINCT`，而不是`ALL`。（正如你在第 3 章回忆的，`SELECT`的情况正好相反。）因此，两个`SELECT`后的`DISTINCT`也不是必需的，尽管它们当然没有错。类似的评论也适用于`intersection`（也适用于我们即将讨论的`difference`）。

**`Tutorial D`**还支持*`不相交并`*（`D_UNION`），这是一种`union`版本，要求操作数*`不相交`*，这意味着它们没有共有的`元组`。例如：

```
S { CITY } D_UNION P { CITY }
```

给定我们常用的示例数据，此表达式将产生运行时错误，因为供应商城市和零件城市不是`不相交`的。`SQL`的对应物可能如下所示：

```
SELECT *
FROM ( SELECT S.CITY FROM S
UNION
SELECT P.CITY FROM P ) AS POINTLESS
WHERE  NOT EXISTS
( SELECT S.CITY FROM S
INTERSECT
SELECT P.CITY FROM P )
```

不过有一个区别：如果供应商城市和零件城市不是`不相交`的，该`SQL表达式`不会在运行时失败，而只会返回空结果。我还应该提到，并非所有`SQL`产品都允许子查询（如它们所称呼的）以所示方式嵌套在`FROM`子句中，尽管`SQL标准`允许。

> **注**
>
> 上述表达式中的相关名*`POINTLESS`*确实毫无意义——注意它从未被引用！——但标准的语法规则要求它。

**`Tutorial D`**还支持`UNION`和`D_UNION`的*`n`*元或前缀形式。我将在此跳过细节。

### 差 (Difference)

同样，设`关系`*`r`*和*`s`*类型相同。则这些`关系`之间的*`差`*，_`r`_ `MINUS` _`s`_（按此顺序），是一个同类型的`关系`，其`关系体`由所有`元组`*`t`*组成，使得*`t`*出现在*`r`*中但不出现在*`s`*中。例如：

```
S { CITY } MINUS P { CITY }  | SELECT S.CITY
                             | FROM   S
                             | EXCEPT
                             | SELECT P.CITY
                             | FROM   P
```

现在回想一下，有一个与`join`相关的运算符叫`semijoin`。同样，也存在一个*`半差`*（_`semidifference`_）运算符，但在这种情况下，运算符不仅仅是“相关”——而是，常规的`difference`是*`semidifference`*的*`一个特例`*（“所有的差都是半差”，你或许可以这么说）。并且如果`semijoin`在某些方面比`join`更重要，那么类似的评论也适用于这里，但力度更大——在实践中，大多数需要`difference`的查询实际上需要的是`semidifference`。定义如下：*`r`*和*`s`*之间的*`半差`*是*`r`*和*`r`* `MATCHING` *`s`*之间的`差`；也就是说，_`r`_ `SEMIMINUS` *`s`*等价于*`r`* `MINUS` (_`r`_ `MATCHING` _`s`_)。例如，考虑查询“_`获取根本不供应任何零件的供应商`_”：

```
S SEMIMINUS SP  | SELECT S.*
                | FROM   S
                | EXCEPT
                | SELECT S.*
                | FROM   S, SP
                | WHERE  S.SNO = SP.SNO
```

**`Tutorial D`**还提供了一种更友好的拼写方式，允许将查询表述为：

```
S NOT MATCHING SP
```

要理解常规的`difference`是`semidifference`的特例，请考虑如果*`r`*和*`s`*类型相同会发生什么（我将细节留作练习）。

### 笛卡尔积 (Cartesian Product)

我包含这个运算符主要是为了完整性；正如我们所见，它实际上只是`join`的一个特例，并且事实上**`Tutorial D`**并不直接支持它——但为了讨论起见，假设它支持。显然，操作数不能有任何共有的`属性`名（否则会发生什么？），且结果`关系头`是输入`关系头`的集合论`并集`。以下是一个示例：

```
( S RENAME ( CITY AS SCITY ) )
TIMES
( P RENAME ( CITY AS PCITY ) )
```

`SQL`对应物：

```
SELECT S.SNO, S.SNAME, S.STATUS, S.CITY AS SCITY
P.PNO, P.PNAME, P.COLOR, P.WEIGHT, P.CITY AS PCITY
FROM   S, P
```

### 除 (Divide)

正如本章开头所承诺的，我将在此给出该运算符的精确定义，但出于几个原因，我不想过多讨论它。其中最重要的原因是，任何可以用`divide`表述的查询，都可以用*`关系比较`*来替代表述，而且我认为更简单（我将在本章后面讨论`关系比较`）。因此，你或许想在初次阅读时跳过以下讨论。

我不想过多深入细节的另一个原因是，无论如何存在多种不同的“`divide`”运算符！也就是说，不幸的是，有多个不同的运算符都叫“`divide`”，我当然不想解释它们全部。相反，我将在此将注意力限制在原始且最简单的那个，其定义如下。设`关系`*`r`*和*`s`*满足*`s`*的`关系头`是*`r`*的`关系头`的子集。则*`r`*除以*`s`*，即*`r`* `DIVIDEBY` _`s`_，是以下内容的简写：

```
r { X } MINUS ( ( r { X } TIMES s ) MINUS r ) { X }
```

这里的*`X`*是*`r`*的`关系头`与*`s`*的`关系头`的（集合论）差集。因此，例如，表达式：

```
SP { SNO, PNO } DIVIDEBY P { PNO }
```

（给定我们常用的示例数据值）产生：

**图5-2**

![Figure Five-2](httpatomoreillycomsourceoreillyimages247886.png)

（大致是供应所有零件的供应商编号；我将在本章后面的“关系比较”一节中解释使用“大致”这一限定词的原因。）以下是`SQL`对应物：

```
SELECT DISTINCT SPX.SNO
FROM   SP AS SPX
WHERE  NOT EXISTS
( SELECT P.PNO
FROM   P
WHERE  NOT EXISTS
( SELECT SPY.SNO
FROM   SP AS SPY
WHERE  SPY.SNO = SPX.SNO
AND  SPY.PNO = P.PNO ) )
```

顺便问一下，你是否想过为什么`divide`被称为“除”？原因是，如果*`r`*和*`s`*是没有共有`属性`的`关系`，并且我们形成`笛卡尔积`_`r`_ `TIMES` _`s`_，然后将结果除以*`s`*，我们会回到*`r`*；换句话说，`笛卡尔积`和`divide`在某种意义上是互逆的。

### 哪些运算符是基本的？

正如我已经或多或少提到过的，并非我们讨论的所有运算符都是基本的——其中几个可以用其他运算符来定义。一个可能的基本集合（并非唯一）包含`restrict`、`project`、`join`、`union`和`semidifference`。_`注：`_ 你可能会惊讶于列表中看不到`rename`。事实上，`rename`并不是基本的，尽管我还没有覆盖足够的基础来展示原因。然而，这个例子表明，是基本的和有用的之间存在区别！我绝对不想缺少我们非常有用的`rename`运算符，即使它不是基本的。

## 求值 SQL 表达式

除了自然连接，Codd最初还定义了一个他称为*`theta`*连接（_`theta`_-join）的运算符，其中*`theta`*表示任何常用的标量比较运算符（“=”，“≠”，“<”等）。该运算符不是基本的；事实上，它等价于`笛卡尔积`的`限制`。以下是供应商和零件按城市（此处*`theta`*为“≠”）进行“不等于”连接的**`Tutorial D`**表述示例：

```
( ( S RENAME ( CITY AS SCITY )
TIMES
( P RENAME ( CITY AS PCITY ) )
WHERE SCITY ≠ PCITY
```

请注意，结果有两个“`city`”`属性`，SCITY 和 PCITY。`SQL`对应物：

```
SELECT S.SNO, S.SNAME, S.STATUS, S.CITY AS SCITY,
P.PNO, P.PNAME, P.COLOR, P.WEIGHT, P.CITY AS PCITY
FROM   S, P
WHERE  S.CITY <> P.CITY
```

我们可以认为这个`SQL表达式`分三步实现，如下所示：

1. 执行`FROM`子句，生成表 S 和 P 的`笛卡尔积`。（如果我们以关系方式执行此操作，我们将*`在`*计算该`笛卡尔积`*`之前`*重命名城市`属性`。`SQL`之所以能在之后重命名它们并成功，是因为其表的列具有从左到右的顺序，这意味着它可以通过序数位置区分两个城市列。为简化起见，我们忽略此细节。）
2. 接着，执行`WHERE`子句，通过对该`笛卡尔积`进行`限制`，消除两城市值相等的行。
3. 最后，执行`SELECT`子句，对该`限制`结果在`SELECT`子句指定的列上进行`投影`。（当然，在这个特定示例中，`SELECT`子句也在进行一些重命名——稍后我们将看到它还提供类似于`关系扩展`运算符的功能。此外，除非指定了`DISTINCT`，否则它通常不会消除重复项。但为了简化，我也想忽略所有这些细节。）

至少作为初步近似，`FROM`子句对应于`笛卡尔积`，`WHERE`子句对应于`限制`，`SELECT`子句对应于`投影`，且整体`SELECT - FROM - WHERE`表达式表示`笛卡尔积`的`限制`的`投影`。由此可知，我刚刚给出了`SQL`的`SELECT - FROM - WHERE`表达式*`语义`*的宽松但相当严谨的定义；等价地，我给出了求值此类表达式的*`概念算法`*。当然，这并不意味着实现必须使用该算法来求值此类表达式；_`恰恰相反`_，它可以使用任何喜欢的算法，只要无论使用什么算法，都保证给出与概念算法相同的结果。并且通常有很好的理由（通常是性能原因）使用不同的算法，从而（例如）以不同的顺序执行子句或以其他方式重写原始查询。然而，实现只有在*`能够证明它使用的算法在逻辑上等价于概念算法`*的情况下，才可以自由地这样做。事实上，表征系统优化器工作的一种方式就是找到一个保证等价于概念算法但性能更好的算法。

## 扩展与汇总

你可能已经注意到，到目前为止我描述的代数没有常规的计算能力。现在，`SQL`有；例如，我们可以编写类似 `SELECT A+B AS . . .` 或 `SELECT SUM(C) AS . . .`（等等）的查询。然而，一旦我们写下那个“+”或那个 SUM，我们就超出了最初定义的`关系代数`的界限。因此，我们需要向代数添加一些新运算符以提供此类功能。这就是`extend`和`summarize`的用途。粗略地说，`extend`支持“_`跨元组`_”计算，`summarize`支持“_`沿属性向下`_”计算。让我们仔细看看。

### 扩展 (Extend)

我将从一个示例开始，因为这个运算符对你来说可能是新的。假设零件重量以磅为单位，我们想以克为单位查看这些重量。一磅等于 454 克，因此我们可以写：

```
EXTEND P ADD              | SELECT P.*,
( WEIGHT * 454 AS GMWT )  |    ( P.WEIGHT * 454 ) AS GMWT
                          | FROM   P
```

给定我们常用的示例值，结果如下所示：

**图5-3**

![Figure Five-3](httpatomoreillycomsourceoreillyimages247839.png)

> **注**
>
> _`重要提示：`_ `关系变量` P在`数据库`中*`不会`*被更改！`EXTEND`不是`SQL`风格的`ALTER TABLE`；`EXTEND表达式`只是一个表达式，并且像任何表达式一样，它仅仅表示一个值。

继续这个例子，现在考虑查询“_`获取克重大于7000克的零件的零件号和克重。`_”以下是**`Tutorial D`**表述：

```
( ( EXTEND P
ADD ( WEIGHT * 454 AS GMWT ) )
WHERE GMWT > 7000.0 ) { PNO, GMWT }
```

`SQL`对应物：

```
SELECT P.*, ( P.WEIGHT * 454 ) AS GMWT
FROM   P
WHERE  ( P.WEIGHT * 454 ) > 7000.0
```

正如你所见，表达式`P.WEIGHT * 454`在`SQL`表述中出现了两次，我们只能希望实现足够智能，能够识别它只需对每个`元组`求值一次而不是两次。相比之下，在**`Tutorial D`**中，该表达式只出现一次。

此示例说明的问题是，`SQL`的`SELECT - FROM - WHERE`模板过于僵化。正如**`Tutorial D`**表述所示，我们需要做的是执行一个`扩展`的`限制`；用`SQL`术语来说，我们需要将`WHERE`子句应用于`SELECT`子句的结果。但`SELECT - FROM - WHERE`模板强制`WHERE`子句应用于*`FROM`*子句的结果，而不是`SELECT`子句的结果。换句话说：在许多方面，代数的全部意义在于（得益于`闭包`）`关系运算`可以以任意方式组合；但`SQL`的`SELECT - FROM - WHERE`模板实际上意味着查询*`必须`*表述为`笛卡尔积`，后跟`限制`，再后跟`投影`和/或`扩展`和/或`重命名`的某种组合——而许多查询根本不符合此模式。

顺便说一下，你可能会疑惑为什么我没有把`SQL查询`表述成这样：

```
SELECT P.*, ( P.WEIGHT * 454 ) AS GMWT
FROM   P
WHERE  GMWT > 7000.0
```

（更改在最后一行。）原因是 GMWT 是*`最终结果`*的列名；表 P 没有这样的列，因此`WHERE`子句没有意义，且表达式会因语法错误而失败。

实际上，`SQL标准`确实允许以更接近**`Tutorial D`**的风格来表述所讨论的查询：

```
SELECT TEMP.PNO, TEMP.GMWT
FROM ( SELECT P.PNO, ( P.WEIGHT * 454 ) AS GMWT
FROM P ) AS TEMP
WHERE  TEMP.GMWT > 7000.0
```

如前所述，然而，并非所有`SQL`产品都允许嵌套子查询以这种方式出现在`FROM`子句中。

现在给出定义。设*`r`*为一个`关系`。则*`扩展`*：

```
EXTEND r ADD ( exp AS X )
```

是一个`关系`，其 (a) `关系头`等于*`r`*的`关系头`扩展了`属性`_`X`_，以及 (b) `关系体`由所有`元组`*`t`*组成，使得*`t`*是*`r`*的一个`元组`扩展了`属性`*`X`*的值，该值是通过对*`r`*的那个`元组`求值*`exp`*计算得出的。`关系`*`r`*不能具有名为*`X`*的`属性`，且*`exp`*不能引用*`X`*。注意，结果的`基数`等于*`r`*的基数，且`度`等于*`r`*的度加一。结果中*`X`*的类型是*`exp`*的类型。

### 汇总 (Summarize)

同样，我将从一个示例开始（查询是“_`对于每个供应商，获取供应商编号以及该供应商供应的零件数量`_”）：

```
SUMMARIZE SP PER ( S { SNO } ) ADD ( COUNT () AS P_COUNT )
```

给定我们常用的示例值，结果如下所示：

**图5-4**

![Figure Five-4](httpatomoreillycomsourceoreillyimages247899.png)

特别注意供应商 S5 的`元组`；`PER`规范表明`汇总`将“_`按S在SNO上的投影`_”进行，这意味着它产生一个包含五个`元组`的结果（该`投影`中的每个`元组`对应一个）。相比之下，以下`SQL表达式`（可能被认为等价于**`Tutorial D`**表述）：

```
SELECT SP.SNO, COUNT(*) AS P_COUNT
FROM   SP
GROUP  BY SP.SNO
```

产生的结果仅包含供应商 S1、S2、S3 和 S4 的`元组`。原因当然是，它从 SP 中提取供应商编号，而供应商 S5 根本没有出现在 SP 中。与**`Tutorial D`**表述*`等价`*的`SQL表达式`是：

```
SELECT S.SNO, TEMP.P_COUNT
FROM   S, LATERAL ( SELECT COUNT(*) AS P_COUNT
FROM   SP
WHERE  SP.SNO = S.SNO ) AS TEMP
```

正如我已经指出的，然而，并非所有`SQL`产品都支持此类表达式。

> **注**
>
> 标准要求此处使用关键字 LATERAL，因为子查询“横向”引用了同一`FROM`子句中的另一个元素。

现在给出定义。设*`r`*和*`s`*为`关系`，使得*`s`*的类型与*`r`*的某个`投影`的类型相同，并设*`s`*的`属性`为*`A1, A2, . . ., An`*。则*`汇总`*：

```
SUMMARIZE r PER ( s ) ADD ( summary AS X )
```

是一个`关系`，其 (a) `关系头`等于*`s`*的`关系头`扩展了`属性`_`X`_，以及 (b) `关系体`由所有`元组`*`t`*组成，使得*`t`*是*`s`*的一个`元组`扩展了`属性`*`X`*的值。该*`X`*值是通过对所有与`元组`*`t`*在`属性`*`A1, A2, . . . , An`*上具有相同值的*`r`*的`元组`求值*`summary`*计算得出的。`关系`*`s`*不能具有名为*`X`*的`属性`，且*`summary`*不能引用*`X`*。注意，结果的`基数`等于*`s`*的基数，且`度`等于*`s`*的度加一。结果中*`X`*的类型是*`summary`*的类型。

支持哪些“`汇总`”函数？好吧，集合是开放式的，但肯定包括常用的`COUNT`、`SUM`、`AVG`、`MAX`和`MIN`。以下是一个涉及`MAX`和`MIN`的示例：

```
SUMMARIZE SP PER ( SP { SNO } ) ADD ( MAX ( QTY ) AS MAXQ ,
MIN ( QTY ) AS MINQ )
```

此示例说明了另外两点：

- 可以在单个`SUMMARIZE`内执行“_`多重汇总`_”。（我之前没有提到这一点，但类似的评论也适用于`RENAME`和`EXTEND`。）
- 此示例中的 PER 操作数不仅“_`类型与`_”待汇总`关系`的某个`投影`相同”，它实际上就是这样一个`投影`。在这种情况下，表达式可以稍微简化，如下所示：

```
SUMMARIZE SP BY { SNO } ADD ( MAX ( QTY ) AS MAXQ ,
MIN ( QTY ) AS MINQ )
```

其他合法的“`汇总`”包括`COUNTD`、`SUMD`和`AVGD`（其中“`D`”代表“`distinct`”，意为“_`在汇总前消除冗余重复值`_”）；`AND`、`OR`和`XOR`（用于类型为`BOOLEAN`的`属性`）；`INTERSECT`、`UNION`和`D_UNION`（用于关系值属性）；等等。

顺便提一下，这里的`COUNT`等*`不是`* _`聚合运算符`_，尽管它们中的大多数与`聚合运算符`同名（`SQL`混淆了这两个概念，导致了不幸的结果）。*`聚合运算符`*调用是一个标量表达式，它返回一个标量值。以下是一个示例：

```
VAR N INTEGER ;
N := COUNT ( S WHERE CITY = 'London' ) ;
```

相比之下，`汇总`仅仅是`SUMMARIZE`调用的操作数；它们在该上下文之外没有意义，实际上也不能在该上下文之外使用。

这里再举一个`SUMMARIZE`示例（“_`伦敦有多少供应商？`_”）：

```
SUMMARIZE ( S WHERE CITY = 'London' ) ADD ( COUNT () AS N )
```

同样，此示例说明了两点：

- 此`SUMMARIZE`没有`PER`规范。因此，默认情况下，汇总是*`按TABLE_DEE`*进行的——也就是说，所示表达式是以下内容的简写：

```
SUMMARIZE ( S WHERE CITY = 'London' )
PER ( TABLE_DEE ) ADD ( COUNT () AS N )
```

我再次提醒你，`TABLE_DEE`是没有`属性`且有一个`元组`的`关系`（因此绝对与“_`每个可能关系的某个投影`_”类型相同！——即所讨论`关系`在空`属性`集上的`投影`）。因此，此`SUMMARIZE`的输出具有*`一个`*`属性`和一个`元组`，给定我们常用的示例数据值，它看起来如下：

当然，我们需要一种在无`RVA`关系和有`RVA`关系之间进行映射的方法，这就是`GROUP`和`UNGROUP`运算符的用途。我不想在此过多深入这些运算符的细节；让我只是说，给定如图 5-1 所示的关系 R1 和 R4，表达式：

```
R1 GROUP ( { PNO } AS PNO_REL )
```

将产生 R4，而表达式：

```
R4 UNGROUP ( PNO_REL )
```

将产生 R1。`SQL`没有这些运算符的直接对应物。

顺便说一下：如果 R4 中恰好包含供应商编号 S*`x`*的一个`元组`，并且该`元组`中的`PNO_REL值`为空，则上述`UNGROUP`的结果将根本不包含供应商编号 S*`x`*的任何`元组`。有关更多细节，请参阅我的著作《数据库系统导论》第八版（Addison-Wesley, 2004）或 Hugh Darwen 和我合著的《数据库、类型与关系模型：第三宣言》第三版（Addison-Wesley, 2006）。

另外：你可能会好奇对带有`RVA`的`关系`进行运算是什么样子。好吧，对*`任何`*`关系`进行运算时，当它们引用该关系的某个类型为*`T`*的`属性`*`A`*时，通常涉及我们可以称为对该`属性`*`A`*的值进行的“_`子运算`_”，而这些子运算正是针对该类型*`T`*定义的运算。因此，如果*`T`*是`关系类型`，那么这些“_`子运算`_”就是针对`关系类型`定义的运算——也就是说，它们本质上就是本章描述的`关系运算`（`join`及其他）！见本章末尾的练习 5-27 至 5-30。

## 表达式转换

我现在已经介绍了我想详细讨论的所有代数运算符，但本章还有一些相关主题值得注意。首先是可能将给定的`关系表达式`转换为另一个逻辑等价的表达式。我在第 3 章的“为什么禁止重复元组”一节中提到了这种可能性，当时我解释说此类转换是优化器所做的事情之一；事实上，此类转换构成了`关系优化`核心的两大思想之一（另一个超出了本书范围，是使用“_`数据库统计信息`_”进行所谓的*`基于成本的优化`*）。在本节中，我想对*`表达式转换`*（有时也称为*`查询重写`*）的过程多说几句。我将从一个微不足道的例子开始。考虑以下表达式（查询是“_`获取供应零件P2的供应商及其相应数量`_”）：

```
( ( S JOIN SP ) WHERE PNO = PNO('P2') ) { ALL BUT PNO }
```

假设有 100 个供应商和 1,000,000 条发货记录，其中 500 条是针对零件 P2 的。如果表达式只是按字面蛮力求值，完全不做任何优化，事件序列如下：

1. _`连接S和SP：`_ 此步骤涉及读取 100 个供应商`元组`；将 1,000,000 条发货`元组`读取 100 次，每次对应 100 个供应商中的一个；构建一个包含 1,000,000 个`元组`的中间结果；并将那 1,000,000 个`元组`写回磁盘。（为简化起见，我假设`元组`在物理上就是这样存储的，并且我假设我可以将“_`元组I/O次数`_”作为合理的性能度量。这两个假设都不现实，但这一事实并不实质性地影响我的论点。）
2. _`限制步骤1的结果：`_ 此步骤涉及读取 1,000,000 个`元组`，但产生的结果仅包含 500 个`元组`，我假设可以将其保留在主存中。
3. _`投影步骤2的结果：`_ 此步骤根本不涉及`元组I/O`，因此我们可以忽略它。

以下过程与刚才描述的过程等价，在产生相同最终结果的意义上，但显然效率高得多：

1. _`将SP限制为仅包含零件P2的元组：`_ 此步骤涉及读取 1,000,000 条发货`元组`，但产生的结果仅包含 500 个`元组`，可以保留在主存中。
2. _`连接S和步骤1的结果：`_ 此步骤涉及读取 100 个供应商`元组`（仅一次，而不是每次针对 P2 发货读取一次，因为所有 P2 发货都在内存中）。结果包含 500 个`元组`（仍在主存中）。
3. _`投影步骤2的结果：`_ 同样，我们可以忽略此步骤。

这两个过程中的第一个总共涉及 102,000,100 次`元组I/O`，而第二个仅涉及 1,000,100 次；因此，显然第二个过程比第一个快 100 倍以上。同样显然的是，我们希望实现使用第二个过程而不是第一个！如果它这样做，那么它（实际上）将原始表达式：

```
( S JOIN SP ) WHERE PNO = PNO('P2')
```

（我现在忽略最终`投影`，因为它与论点并非真正相关）转换为表达式：

```
S JOIN ( SP WHERE PNO = PNO('P2') )
```

这两个表达式逻辑等价，但性能特征截然不同，正如我们所见。因此，如果系统呈现第一个表达式，我们希望它在求值之前将其转换为第二个——当然它可以做到。重点是，`关系代数`作为一种高级形式化语言，服从各种形式化的*`转换定律`*；例如，有一条定律指出`连接`后跟`限制`可以转换为`限制`后跟`连接`（我在示例中使用了那条特定定律）。_`而优秀的优化器会了解这些定律`_，并会应用它们——因为，当然，查询的性能不应取决于最初用于表述该查询的具体语法。（当然，并非所有代数运算符都是基本的这一事实直接意味着某些表达式可以转换为其他表达式——例如，涉及`intersect`的表达式可以转换为涉及`join`的表达式——但远不止于此，我们将很快看到。）

现在，有许多可能的转换定律，这里不是详尽讨论的地方。我只想强调几个重要案例和关键点。首先，上一段提到的定律实际上是一个更广泛定律的特例，称为*`分配律`*。一般而言，如果对于所有*`a`*和*`b`*有*`f(g(a,b))`* = _`g(f(a),f(b))`_，则一元运算符*`f`*在二元运算符*`g`*上*`可分配`*。例如，在普通算术中，SQRT（平方根）在乘法上可分配，因为：

```
SQRT ( a * b ) = SQRT ( a ) * SQRT ( b)
```

对于所有*`a`*和*`b`*（将*`f`*设为 SQRT，*`g`*设为“\*”）；因此，数值表达式优化器在进行数值表达式转换时，总是可以用另一个替换这些表达式中的任何一个。作为反例，SQRT 在加法上*`不`*可分配，因为*`a+b`*的平方根通常不等于*`a`*和*`b`*的平方根之和。

在`关系代数`中，`restrict`在`intersect`、`union`和`difference`上可分配。它也在`join`上可分配，前提是`限制`条件在最复杂的情况下由两个独立条件的 AND 组成，每个条件对应两个`连接`操作数之一。在上面讨论的示例中，满足了此要求——事实上，`限制`条件非常简单且仅应用于其中一个操作数——因此我们能够使用分配律用更高效的等价物替换该表达式。净效果是我们能够“_`提前执行限制`_”。提前执行限制几乎总是个好主意，因为它减少了在序列中下一个操作中要扫描的`元组`数量，并且可能也减少了该操作输出中的`元组`数量。

以下是分配律的其他一些具体案例，这次涉及`projection`。首先，`project`在`intersect`和`union`上可分配，但在`difference`上不可分配。其次，它也在`join`上可分配，只要所有连接`属性`都包含在`投影`中。这些定律可用于“_`提前执行投影`_”，这通常也是个好主意，原因与前面关于限制的原因类似。

另外两个重要的一般定律是*`交换律`*和*`结合律`*：

- 如果对于所有*`a`*和*`b`*有*`g(a,b)`* = _`g(b,a)`_，则二元运算符*`g`*是*`可交换的`*。例如，在普通算术中，加法和乘法是可交换的，但除法和减法不是。在`关系代数`中，`intersect`、`union`和`join`都是可交换的，但`difference`和`division`不是。因此，例如，如果查询涉及两个`关系`*`r`*和*`s`*的`连接`，交换律告诉我们，将*`r`*和*`s`*中的哪一个作为“_`外`_”`关系`，哪一个作为“_`内`_”`关系`无关紧要。因此，系统可以自由地选择（例如）将较小的`关系`作为计算`连接`时的“_`外`_”`关系`。
- 如果对于所有*`a, b, c`*有*`g(a,g(b,c))`* = _`g(g(a,b),c)`_，则二元运算符*`g`*是*`可结合的`*。在算术中，加法和乘法是可结合的，但减法和除法不是。在`关系代数`中，`intersect`、`union`和`join`都是可结合的，但`difference`和`division`不是。因此，例如，如果查询涉及三个`关系`*`r, s`*和*`u`*的`连接`，结合律和交换律共同告诉我们，我们可以以任何喜欢的顺序两两连接这些`关系`。（它们还告诉我们，定义运算符的*`n`*元或前缀版本是合法的，正如**`Tutorial D`**所做的那样。）因此，系统可以自由地决定各种可能序列中哪一个最有效。

请注意，所有这些转换都可以在不考虑`数据库`中实际存储的实际数据值或物理存储结构（索引等）的情况下执行。换句话说，此类转换代表了几乎保证良好的优化，无论`数据库`物理上看起来如何。

## 关系比较

在第 2 章中，我提到相等比较运算符“`=`”适用于每种类型。特别是，因此它适用于*`关系`*类型；也就是说，给定两个相同`关系类型`*`T`*的`关系`*`r`*和*`s`*，我们至少必须能够测试这两个`关系`是否相等。其他比较也可能有用；例如，我们可能想测试*`r`*是否是*`s`*的子集（意味着*`r`*中的每个`元组`也在*`s`*中），或者*`r`*是否是*`s`*的*`真`*子集（意味着*`r`*中的每个`元组`也在*`s`*中，但*`s`*包含至少一个不在*`r`*中的`元组`）。

现在，我必须立即说明，这些运算符本身不是`关系运算符`——也就是说，它们不属于`关系代数`——因为它们的结果是真值，而不是`关系`。但为了方便，在本章讨论它们，我也会这么做。以下是一个简单示例：

```
S { CITY } = P { CITY }
```

左比较项是供应商在 CITY 上的`投影`，右比较项是零件在 CITY 上的`投影`，如果这两个投影相等，则比较返回 TRUE，否则返回 FALSE。换句话说，该比较（当然是一个`布尔表达式`）意思是：“_`供应商城市的集合是否等于零件城市的集合？`_”

以下是另一个示例：

```
S { SNO } ⊃ SP { SNO }
```

_`解释：`_ 这里的符号“`⊃`”表示“_`是真超集`_”。此比较的含义（经过大幅意译）是：“_`是否存在根本不供应任何零件的供应商？`_”

其他有用的`关系比较运算符`包括“`⊇`”（“_`是超集`_”）、“`⊆`”（“_`是子集`_”）和“`⊂`”（“_`是真子集`_”）。

`关系比较`明显有用的地方是在`限制`方面。让我们看一些例子。首先考虑查询“_`获取供应所有零件的供应商。`_”以下是一种可能的表述：

```
WITH ( SP RENAME ( SNO AS X ) ) AS R :
S WHERE ( R WHERE X = SNO ) { PNO } = P { PNO }
```

_`解释：`_ 表达式 WITH . . . AS R 实际上等价于一个赋值，将表达式 `SP RENAME (SNO AS X)` 的值赋给某个临时的、系统生成的`关系变量` R；赋值后，R 的值是一个与`关系变量` SP 的当前值完全相同的`关系`，只是`属性` SNO 被重命名为 X。（引入名称 R 和 X 的目的仅仅是为了避免随后可能出现的命名冲突。）然后，对于`关系变量` S 中的给定供应商 S*`x`*（比如说），表达式：

```
( R WHERE X = SNO ) { PNO }
```

求值为一个具有一个`属性`（PNO）的`关系`，给出供应商 S*`x`*供应的所有零件的零件号。请特别注意，如果供应商 S*`x`*不供应任何零件，该`关系`将不包含任何`元组`。最后，将该度为一的`关系`与`关系变量` P 在 PNO 上的`投影`进行相等性测试。显然，该测试当且仅当 S*`x`*供应的零件号集合等于`关系变量` P 中的零件号集合时，才会返回 TRUE。因此，整体结果精确包含`关系变量` S 中表示供应`关系变量` P 中提到的所有零件的供应商的那些`元组`。

当然，如果我们愿意，可以将整个查询写成一个单一表达式：

```
S WHERE ( ( SP RENAME ( SNO AS X ) ) WHERE X = SNO ) { PNO } = P { PNO }
```

但使用 WITH 为子表达式的结果引入名称通常有助于简化复杂查询的构建。以下是一个定义：如果*`rx`*是一个提及某个`关系变量`*`R`*的`关系表达式`，则 WITH _`ry`_ AS _`R: rx`_（其中*`ry`*是另一个`关系表达式`）也是一个`关系表达式`。当然，*`rx`*可能也是一个 WITH 表达式，如下所示：

```
WITH ry AS R1 : WITH rz AS R2 : rx
```

后一个表达式可以简写为：

```
WITH ry AS R1, rz AS R2 : rx
```

我们稍后会看到大量这种简写形式的示例。

顺便说一下，上述例子（“_`获取供应所有零件的供应商`_”）与常用于说明`关系除运算符`的例子非常相似。具体而言，表达式：

```
SP { SNO, PNO } DIVIDEBY P { PNO }
```

（我在前面的“原始运算符”一节中作为`divide`的示例给出）通常被描述为查询“_`获取供应所有零件的供应商编号。`_”的关系表述。但它不是！相反，它是查询“_`获取供应至少一个零件且实际上供应所有零件的供应商编号。`_”的关系表述。（如果你好奇这里的逻辑区别是什么，请参见本章末尾的练习 5-25。）我认为，涉及`关系比较`的表述不仅比`divide`表述更容易理解——而且它具有*`正确`*的优势。

这里还有另一个例子。查询是：“_`获取供应商编号对(Sx, Sy)，使得Sx和Sy供应完全相同的零件集合。`_” 没有`关系比较`，这个查询非常困难！以下是它：

```
WITH ( SP RENAME ( SNO AS SX ) ) { SX, PNO } AS R1 ,
( SP RENAME ( SNO AS SY ) ) { SY, PNO } AS R2 ,
R1 { SX } AS R3 ,
R2 { SY } AS R4 ,
( R1 JOIN R4 ) AS R5 ,
( R2 JOIN R3 ) AS R6 ,
( R1 JOIN R2 ) AS R7 ,
( R3 JOIN R4 ) AS R8 ,
SP { PNO } AS R9 ,
( R8 JOIN R9 ) AS R10 ,
( R10 MINUS R7 ) AS R11 ,
( R6 JOIN R11 ) AS R12 ,
( R5 JOIN R11 ) AS R13 ,
R12 { SX, SY } AS R14 ,
R13 { SX, SY } AS R15 ,
( R14 UNION R15 ) AS R16 ,
R7 { SX, SY } AS R17 :
R17 MINUS R16
```

但有了`关系比较`，它就相当直接了：

```
WITH ( S RENAME ( SNO AS SX ) ) { SX } AS RX ,
( S RENAME ( SNO AS SY ) ) { SY } AS RY :
( RX JOIN RY ) WHERE ( SP WHERE SNO = SX ) { PNO } =
( SP WHERE SNO = SY ) { PNO }
```

顺便提一下，我指出在此处的`WHERE`子句末尾附加“`AND SX < SY`”会产生略微更整洁的结果；具体而言，它将 (a) 消除 (S*`x`*,S*`x`*) 形式的对，以及 (b) 确保 (S*`x`*,S*`y`*) 和 (S*`y`*,S*`x`*) 对不会同时出现。

实践中经常需要的一种特定比较是给定`关系`*`r`*与同类型的空`关系`之间的“`=`”比较——换句话说，测试*`r`*是否为空。因此，让我引入以下简写：

```
IS_EMPTY ( rx )
```

此表达式定义为：如果`关系表达式`*`rx`*表示的`关系`*`r`*为空，则返回 TRUE，否则返回 FALSE。我将在下一章 heavily 依赖此结构。

另一个常见需求是能够测试给定`元组`*`t`*是否出现在给定`关系`*`r`*中：

```
t
∊
rx
```

此表达式定义为以下`关系比较`的简写：

```
RELATION { t } ⊆ rx
```

这里的左操作数是`关系选择器`调用，它返回仅包含指定`元组`*`t`*的`关系`。因此，如果*`t`*出现在`关系表达式`*`rx`*表示的`关系`*`r`*中，该比较返回 TRUE，否则返回 FALSE（“`∊`”是*`集合成员`*运算符；表达式*`t`* ∊ *`r`*可读作“_`t`_[属于] _`r`_”）。事实上，正如你可能已经意识到的，“`∊`”运算符本质上是`SQL`的`IN`运算符——只是`SQL`的`IN`的左操作数通常是标量而不是行，这意味着发生了某种强制转换。无论如何，`SQL`肯定不支持除这个相当特殊的比较之外的任何`关系比较`。（好吧，它支持`NOT IN`；**`Tutorial D`**也支持，形式为“`∉`”。）

## 关于关系赋值的进一步探讨

赋值运算符“`:=`”类似于相等比较运算符“`=`”，因为它适用于每种类型，`关系类型`也不例外。特别是，`关系赋值`在另一个方面也与“`=`”和前一节的其他比较运算符相似：它不是`关系代数`的一部分。为什么？因为它的目标必须非常具体地是一个`关系变量`，而不是一个`关系`（`关系变量`也不属于`关系代数`——`关系代数`本身没有更新的概念，而“_`变量`_”_`意味着`_“_`可更新的`_”）。尽管如此，我想在本章对*`关系赋值`*多说几句；具体而言，我想更仔细地检查`INSERT`、`DELETE`和`UPDATE`简写形式。正如你所知，这些运算符*`确实`*只是某些`关系赋值`的简写——但现在我能够解释“_`长格式`_”版本在适当的代数运算符方面究竟是什么样子。我将通过展示一些简单示例来做到这一点。

首先，一般的`关系赋值`如下所示：

```
R := rx
```

这里的*`R`*是一个`关系变量`，*`rx`*是一个与*`R`*类型相同的`关系表达式`。效果是将表达式*`rx`*表示的`关系`*`r`*赋值给`关系变量`_`R`_（我提醒你来自第 2 章的练习，赋值后，布尔表达式*`R`* = *`r`*必须求值为 TRUE：_`赋值原则`_）。以下是一个简单示例：

```
S := RELATION { TUPLE { SNO SNO('S1'), SNAME NAME('Smith'),
STATUS 20, CITY 'London' } ,
TUPLE { SNO SNO('S2'), SNAME NAME('Jones'),
STATUS 10, CITY 'Paris'  } ,
TUPLE { SNO SNO('S3'), SNAME NAME('Blake'),
STATUS 30, CITY 'Paris'  } ,
TUPLE { SNO SNO('S4'), SNAME NAME('Clark'),
STATUS 20, CITY 'London' } ,
TUPLE { SNO SNO('S5'), SNAME NAME('Adams'),
STATUS 30, CITY 'Athens' } } ;
```

（正如我们在第 3 章看到的，这里的右侧表达式是另一个`关系选择器`调用；事实上，它是一个`关系字面量`。）

现在我可以转向`INSERT`、`DELETE`和`UPDATE`。设`关系变量` PQ 定义如下：

```
VAR PQ BASE RELATION { PNO PNO, QTY QTY } KEY { PNO } ;
```

以下是此`关系变量`上可能的`INSERT`：

```
INSERT PQ ( SUMMARIZE SP PER ( P { PNO } )
ADD ( SUM ( QTY ) AS QTY ) ) ;
```

以下是“_`长格式`_”赋值等价形式：

```
PQ := PQ UNION ( SUMMARIZE SP PER ( P { PNO } )
ADD ( SUM ( QTY ) AS QTY ) ) ;
```

换句话说，`INSERT`通过形成`关系变量` PQ 的旧值与`SUMMARIZE`调用表示的`关系`的`并`（比如说*`pq`*），然后将该`关系`*`pq`*赋值给`关系变量` PQ 来工作。（顺便提一下，我在此假设插入目标中已存在的`元组`不是错误。如果是，则展开中的那个`UNION`应替换为`D_UNION`。）

接下来，一个`DELETE`示例：

```
DELETE S WHERE CITY = 'Athens' ;
```

长格式等价形式：

```
S := S WHERE NOT ( CITY = 'Athens' ) ;
```

最后，一个`UPDATE`示例：

```
UPDATE P WHERE CITY = 'London'
( WEIGHT := 2 * WEIGHT , CITY := 'Oslo' ) ;
```

长格式等价形式：

```
P := WITH ( P WHERE CITY = 'London' ) AS R1 ,
( EXTEND R1 ADD ( 2 * WEIGHT AS NEW_WEIGHT,
'Oslo' AS NEW_CITY ) ) AS R2 ,
R2 { ALL BUT WEIGHT, CITY } AS R3,
R3 RENAME { NEW_WEIGHT AS WEIGHT, NEW_CITY AS CITY } AS R4,
P MINUS R1 AS R5 :
R5 UNION R4;
```

这个需要稍微多解释一下。首先，R1 是要更新的`元组`集合（粗略地说——见第 4 章）。接下来，我们用适用的新 WEIGHT 和 CITY 值扩展 R1 中的每个`元组`；那是 R2。然后我们丢弃旧的 WEIGHT 和 CITY 值（那是 R3）。最后，我们将 NEW*WEIGHT 和 NEW_CITY 分别重命名为 WEIGHT 和 CITY（那是 R4）；然后我们识别出不需要更新的`元组`集合（那是 R5），并将 R5 和 R4 的`并`赋值给`关系变量` P。注意此示例中使用了`EXTEND`和`RENAME`的“*`多重`\_”形式。

## ORDER BY 运算符

本章我想讨论的最后一个主题是`ORDER BY`。`ORDER BY`还不是`关系代数`的一部分；正如我在第 1 章指出的，事实上，它根本不是`关系运算符`，因为它产生的结果不是`关系`（它确实接受一个`关系`作为输入，但它产生其他东西——即一个*`元组序列`*——作为输出）。当然，我不是说`ORDER BY`没有用；然而，我*`确实`*说它不能合法或合理地出现在`关系表达式`中。因此，根据定义，以下表达式虽然合法，但本身不是`关系表达式`：

```
S MATCHING SP        | SELECT DISTINCT S.*
ORDER ( ASC SNO )  | FROM   S, SP
                   | WHERE  S.SNO = SP.SNO
                   | ORDER  BY S.SNO ASC
```

话虽如此，我想指出，由于几个原因，`ORDER BY`（在**`Tutorial D`**中仅为`ORDER`）实际上是一个相当奇怪的运算符。首先，它通过对`元组`进行排序以某种指定序列来工作——然而，“`<`”和“`>`”并未为`元组`定义（见第 3 章的练习 3-11）。其次，它不是*`函数`*。`关系代数`的所有运算符——事实上，所有只读运算符（在该术语的通常意义上）——*`都是`*函数，意味着对于任何给定输入始终只有一个可能的输出。相比之下，`ORDER BY`可以从相同的输入产生几个不同的输出。为了说明这一点，考虑操作`ORDER BY CITY`对我们常用的供应商`关系`的效果。显然，此操作可以返回以下四个不同结果中的任何一个，对应于以下序列（为简化起见，我将仅显示供应商编号）：

```
S5, S1, S4, S2, S3
S5, S4, S1, S2, S3
S5, S1, S4, S3, S2
S5, S4, S1, S3, S2
```

_`关于SQL的说明：`_ 我必须顺便提及，大多数代数运算符在`SQL`中都有也不是函数的对应物。这种状态是由于，如第 2 章练习中所示，`SQL`允许比较*`v1`* = *`v2`*即使*`v1`*和*`v2`*是不同的。例如，考虑字符字符串 'Paris' 和 'Paris '（注意后者中的尾随空格）；这些值显然是不同的，但`SQL`有时将它们视为相等。因此，某些查询会产生标准所称的“_`可能非确定性结果`_”。以下是一个简单示例：

```
SELECT DISTINCT S.CITY FROM S
```

如果一个供应商的 CITY 值为 'Paris'，另一个为 'Paris '（等等，文本说分别是 'Paris' 和 'Paris '），那么结果将包含 'Paris' 或 'Paris '中的一个或两个，但我们得到哪个结果可能是未定义的。我们甚至可能合法地在一天得到一个结果，在另一天得到另一个结果，即使`数据库`在此期间根本没有改变。你或许喜欢思考一下这种状况的一些含义。

## 总结

几乎不可避免地，这是本书最长的一章。即便如此，我还没有涵盖“_`所有已知`_”代数运算符，也没有解释我涵盖的每个最后细节；但我希望我*`确实`*涵盖了足够的内容，让你对代数到底是什么有很好的感觉。简要回顾如下，这是我讨论的运算符列表：`rename`（重命名）、`restrict`（限制）、`project`（投影，包括`ALL BUT`形式）、`join`（连接）、`semijoin`（半连接，`MATCHING`）、`intersection`（交）、`union`（并）、`disjoint union`（不相交并，`D_UNION`）、`difference`（差）、`semidifference`（半差，`NOT MATCHING`）、`product`（积）、`divide`（除）、`theta-join`（theta连接，包括等值连接）、`extend`（扩展）、`summarize`（汇总）、`group`（分组）和`ungroup`（取消分组）。我还讨论了某些非代数运算符：`关系比较`、`INSERT`、`DELETE`和`UPDATE`（以及`关系赋值`），以及`ORDER BY`。我涵盖的其他主题包括：

- _`闭包：`_ 我解释了需要一组`关系类型推断规则`的原因，以便我们始终知道任何给定`关系表达式`结果的类型。
- _`基本运算符：`_ 我展示了实际上许多运算符是其他运算符某些组合的简写。特别是，`intersect`和`product`是`join`的特例，而`difference`是`semidifference`的特例。
- _`连接：`_ 我还展示了——得益于`join`既是可交换的又是可结合的——可以定义该运算符的*`n`*元或前缀版本（其中*`n`*是任何大于或等于零的整数）。零个`关系`的`连接`是`TABLE_DEE`。
- _`SQL表达式：`_ 我给出了求值`SQL SELECT - FROM - WHERE`表达式的概念算法。我还建议`SELECT - FROM - WHERE`模板有时过于僵化。
- _`汇总与聚合运算符：`_ 我强调了（没有过多深入细节）这些在逻辑上是不同的结构。
- _`表达式转换：`_ 我简要解释了分配律、交换律和结合律，并描述了它们在优化（“_`查询重写`_”）中的作用。
- _`WITH：`_ 我讨论了使用 WITH 通过为子表达式的结果引入名称来简化表达式构建。
