# 第四章 关系变量

我们在第一章中看到，`关系变量`（简称 `relvar`）是一个其允许的 `值` 为 `关系` 的 `变量`，并且 `INSERT`、`DELETE` 和 `UPDATE` `操作` 的目标具体来说是 `关系变量`，而不是 `关系`。我们还看到，`INSERT`、`DELETE` 和 `UPDATE` 都只是某些 `关系赋值` 的简写。我还要提醒你：(a) 如果 _`R`_ 是一个 `关系变量` 且 _`r`_ 是要赋给 _`R`_ 的 `关系`，那么 _`R`_ 和 _`r`_ 必须具有相同的 `关系类型`；(b) 第三章为 `关系` 正式定义的术语 _`关系头`_（heading）、_`体`_（body）、_`属性`_（attribute）、_`元组`_（tuple）、_`基数`_（cardinality）和 _`度`_（degree），都可以以显而易见的方式解释为同样适用于 `关系变量`。现在是时候仔细研究这些问题了。作为示例的基础，我将使用以下 **`Tutorial D`** 定义，用于供应商-零件 `数据库` 中的 `基关系变量`：

```
VAR S BASE RELATION
  { SNO SNO, SNAME NAME, STATUS INTEGER, CITY CHAR }
    KEY { SNO } ;

VAR P BASE RELATION
  { PNO PNO, PNAME NAME, COLOR COLOR, WEIGHT WEIGHT, CITY CHAR }
    KEY { PNO } ;

VAR SP BASE RELATION
  { SNO SNO, PNO PNO, QTY QTY }
    KEY { SNO, PNO }

    FOREIGN KEY { SNO } REFERENCES S
    FOREIGN KEY { PNO } REFERENCES P ;
```

## 更新是集合级操作

我要强调的第一点是，无论我们使用什么语法来表达，`关系赋值` 都是一种 _`集合级`_ `操作`。（事实上，正如我们将在第五章看到的，`关系模型` 中的 _`所有`_ `操作` 都是 `集合级` 的。）因此，`INSERT` 向目标 `关系变量` 中插入一个 `元组` `集合`；`DELETE` 从目标 `关系变量` 中删除一个 `元组` `集合`；`UPDATE` 更新目标 `关系变量` 中的一个 `元组` `集合`。确实，我们经常（例如）谈论更新某个单独的 `元组`，但你需要明白：

- 这种说法实际上意味着我们要更新的 `元组` `集合` 恰好具有 `基数` 一。
- 此外，更新 `基数` 为一的 `元组` `集合` 有时根本就是不可能的。

例如，假设 `关系变量` `S` 受到一条 `完整性约束`（见第六章）的约束，即供应商 `S1` 和 `S4` 必须始终位于同一城市。那么，任何试图仅更改这两家供应商中一家的城市的“`单条元组 UPDATE`”都必然会失败。相反，我们必须同时更新两者，可能像这样（`SQL`）：

```
UPDATE S
SET    CITY = 'New York'
WHERE  S.SNO = SNO('S1') OR S.SNO = SNO('S4') ;
```

显然，这里更新的是一个包含两个 `元组` 的 `集合`。

> **注**
>
> 出于兴趣，这里是用 **`Tutorial D`** 表达的相同更新（可以看出，看起来非常相似）：
>
> ```
> UPDATE S WHERE SNO = SNO('S1') OR SNO = SNO('S4')
>        ( CITY := 'New York' ) ;
> ```

上述情况的一个结果是，`关系模型` 中没有任何东西类似于 `SQL` 的“`定位更新`”（即 `UPDATE` 或 `DELETE` “`WHERE CURRENT OF cursor`”），因为根据定义，这些 `操作` 是 `元组级` 的，而不是 `集合级` 的。它们在今天的产品中大多数时候确实能工作，但这是因为这些产品在支持 `完整性约束` 方面做得并不好。如果产品在这方面有所改进，这些“`定位更新`”可能 _`不再`_ 有效；也就是说，今天成功的应用程序明天可能会失败——在我看来，这不是一种理想的状态。

现在我需要坦白一件事。事实上，像我之前那样谈论“`更新一个元组`”——或者更准确地说是更新 `元组` 的 `集合`——本身就是非常不精确的（甚至可以说是粗糙的）。如果 _`V`_ 是可更新的，那么根据定义 _`V`_ 必须是一个 _`变量`_，而不是一个 `值`；并且 `元组`，就像 `关系` 一样，是 _`值`_，根据定义同样不能被更新。当我们谈论（例如）在某个 `关系变量` _`R`_ 中将 `元组` _`t1`_ 更新为 _`t2`_ 时，我们真正的意思是我们正在用另一个 `元组` _`t2`_ _`替换`_ _`R`_ 中的 `元组` _`t1`_。而这种说法仍然很粗糙！我们 _`真正`_ 的意思是，我们正在用另一个 `关系` _`r2`_ 替换作为 _`R`_ 原始 `值` 的 `关系` _`r1`_。那么这里的 `关系` _`r2`_ 究竟是什么？好吧，设 _`s1`_ 和 _`s2`_ 分别为仅包含 `元组` _`t1`_ 和 `元组` _`t2`_ 的 `关系`；那么 _`r2`_ 就是 (_`r1`_ `MINUS` _`s1`_) `UNION` _`s2`_。换句话说，“在 `关系变量` _`R`_ 中“将 _`t1`_ 更新为 _`t2`_`” 可以被看作是先删除 _`t1`_ 然后插入 _`t2`_——如果（尽管我前面说了那么多）允许我以这种松散的方式谈论删除和插入单个 `元组` 的话。

同理，谈论“更新 `元组` _`t`_ 中的属性 _`A`_`”——或者在 `关系` _`r`_ 中，甚至在 `关系变量` _`R`\_ 中——实际上并没有多大意义。当然，我们还是会这么说，因为这很方便（省去了大量迂回的说法）；但这就像我在第一章中讨论的“用户友好型术语”一样——只有在我们都明白这种说法只是对真相的近似，并且它确实往往会掩盖事情的本质时，才可以这么说。

## 关于候选键的更多讨论

我在第一章中解释了 `候选键` 的基本思想，但现在我想让这个概念更加精确。定义如下：

> _`定义：`_ 设 _`K`_ 是 `关系变量` _`R`_ 的 `关系头` 的一个 _`子集`_。那么当且仅当 _`K`_ 具备以下两个属性时，它才是 _`R`_ 的 _`候选键`_（或简称 _`键`_）：

**`唯一性`**
: _`R`_ 的任何可能 `值` 都不会包含两个在 _`K`_ 上具有相同 `值` 的不同 `元组`。

**`不可约性`**
: _`K`_ 的任何 _`真子集`_ 都不具有 `唯一性`。

> **注**
>
> 按照惯例，在本书中我将形如“`*`B`* 是 *`A`* 的 `子集`”和“`_`A`_ 是 _`B`_ 的 `超集`”的语句理解为包含 _`A`_ 和 _`B`_ 可能相等的情况。如果我想排除这种可能性，我会明确地使用 _`真`_ `子集` 和 `超集` 来表述。

现在，`唯一性` 属性是不言自明的，但我需要对 `不可约性` 属性多说几句。考虑 `关系变量` `S` 和 `属性` `集合` {`SNO`,`CITY`}——我们称它为 `SK`——它显然是 S 的 `关系头` 的一个具有 `唯一性` 的 `子集`（任何作为可能 `S 值` 的 `关系` 都不会有两个在 `SK` 上具有相同 `值` 的不同 `元组`）。但它不具有 `不可约性`，因为我们可以丢弃 `CITY` `属性`，剩下的 {`SNO`} `集合` 仍然具有 `唯一性`。因此我们不把 `SK` 视为 `键`，因为它“太大了”。相比之下，{`SNO`} _`是`_ `不可约` 的，它是一个 `键`。

为什么我们希望 `键` 是 `不可约` 的？一个原因是，如果我们指定了一个 _`不`_ `不可约` 的“`键`”，`DBMS` 将无法正确执行 `唯一性约束`。例如，假设我们撒谎告诉 `DBMS` {`SNO`,`CITY`} 是一个 `键`。那么它将无法强制执行供应商编号“`全局`”唯一的 `约束`；相反，它只能执行较弱的 `约束`，即供应商编号在给定城市内“`局部`”唯一。因此，这是（并非唯一原因）我们要求 `键` 不包含任何非唯一标识所需 `属性` 的一个原因。

现在，我们目前看到的所有 `关系变量` 都只有一个 `键`。相比之下，下面是一些具有两个或更多 `键` 的示例（它们应该是自解释的）。注意第二个和第三个示例中 `键` 的重叠性质。

```
VAR TAX_BRACKET BASE RELATION
  { LOW MONEY, HIGH MONEY, PERCENTAGE INTEGER }
    KEY { LOW }
    KEY { HIGH }
    KEY { PERCENTAGE } ;

VAR ROSTER BASE RELATION
  { DAY DAY_OF_WEEK, TIME TIME_OF_DAY, GATE GATE, PILOT NAME }
    KEY { DAY, TIME, GATE }
    KEY { DAY, TIME, PILOT } ;

VAR MARRIAGE BASE RELATION
  { SPOUSE_A NAME, SPOUSE_B NAME, DATE_OF_MARRIAGE DATE }
    /* assume no polygamy and no couple marrying */
    /* each other more than once ...             */
    KEY { SPOUSE_A, DATE_OF_MARRIAGE }
    KEY { DATE_OF_MARRIAGE, SPOUSE_B }
    KEY { SPOUSE_B, SPOUSE_A } ;
```

我在本节末尾补充几点杂项。首先，请注意 `键` 的概念适用于 `关系变量`，而不是 `关系`。为什么？因为说某个东西是 `键` 就是说存在某种 `完整性约束`——具体来说是某种 `唯一性约束`——而 `完整性约束` 适用于 `变量`，而不是 `值`。（`完整性约束` 约束 `更新`，而 `更新` 适用于 `变量`，而不是 `值`。见第六章的进一步讨论。）

其次，如果 _`R`_ 是一个 `关系变量`，那么 _`R`_ 肯定至少有一个 `键`。原因是 _`R`_ 的每个可能 `值` 都是一个 `关系`，因此根据定义不包含重复的 `元组`；至少，_`R`_ 的所有 `属性` 的组合肯定具有 `唯一性` 属性。[^1] 因此，要么该组合也具有 `不可约性`，要么该组合的某个 _`真子集`_ 具有 `不可约性`。无论如何，总存在 _`某个`_ 既唯一又不可约的东西。

第三，请注意 `键值` 是 _`元组`_。例如，对于 `关系变量` `S`，其唯一 `键` 为 {`SNO`}，该 `键` 对于某个特定 `元组`——比如供应商 `S1` 的 `元组`——的 `值` 是：

```
TUPLE { SNO SNO('S1') }
```

（回想第三章，`元组` 的任何 `子集` 本身也是一个 `元组`。）当然，在实践中我们通常会非正式地说，这个示例中的 `键值` 只是 `S1`——或者更准确地说是 `SNO('S1')`——但它实际上并不是。

承接上一点：现在应该清楚，`键` 的概念，就像 `关系模型` 中的许多其他概念一样，依赖于 _`元组相等`_ 的基本概念。也就是说，为了强制执行 `唯一性约束`，我们需要能够判断两个 `键值` 何时相等，而这正是 `元组相等` 的问题——即使像在 `关系变量` `S` 的情况下，所讨论的 `元组` 的 `度` 为一且“看起来像”简单的 `标量值`。

我的最后一点与 _`函数依赖`_ 的概念有关。我不想在这里深入讨论该概念——我将在第七章中讨论——但你可能已经熟悉它了；我在这里只想提醒你注意以下几点。设 _`K`_ 是 `关系变量` _`R`_ 的 `键`，且 _`A`_ 是 _`R`_ 的任何 `属性`。那么 _`R`_ 必然满足 `函数依赖`：

```
      K
      →
      A
```

简要阐述：一般而言，`函数依赖` _`K`_ → _`A`_ 意味着每当 _`R`_ 的两个 `元组` 在 _`K`_ 上具有相同的 `值` 时，它们在 _`A`_ 上也具有相同的 `值`。但如果两个 `元组` 在 _`K`_ 上具有相同的 `值`（其中 _`K`_ 是一个 `键`），那么根据定义它们就是完全相同的 `元组`！——因此它们 _`必然`_ 在 _`A`_ 上具有相同的 `值`。换句话说，粗略地说：我们总是有从 `键` 指向 `关系变量` 中其他所有内容的“`函数依赖箭头`”。同样，我将在第七章重新讨论这些想法。

## 关于外键的更多讨论

我在第一章中解释了 `外键` 的基本思想，但这里给出一个精确定义（注意再次依赖 `元组相等`）：

> _`定义：`_ 设 _`R1`_ 和 _`R2`_ 为 `关系变量`（不一定不同），且 _`K`_ 是 _`R1`_ 的 `键`。设 _`FK`_ 是 _`R2`_ 的 `关系头` 的一个 `子集`，在经过可能的 `属性重命名` 后，它恰好包含与 _`K`_ 相同的 `属性`。那么当且仅当在任何时刻，_`R2`_ 中的每个 `元组` 在 _`R1`_ 的某个（必然唯一的）`元组` 中都有一个相等的 _`FK`_ `值`（与 _`K`_ `值` 比较）时，_`FK`_ 才是一个 _`外键`_。

如我们所知，在供应商-零件 `数据库` 中，{`SNO`} 和 {`PNO`} 是 `关系变量` `SP` 中的 `外键`，分别引用 `关系变量` `S` 和 `关系变量` `P` 中的唯一 `候选键`——实际上是 `主键`。这是另一个示例：

```
VAR EMP BASE RELATION
  { ENO ENO, ..., MNO ENO, ... }
    KEY { ENO }
    FOREIGN KEY { RENAME ( MNO AS ENO ) } REFERENCES EMP ;
```

这里的 `属性` `MNO` 表示由 `ENO` 标识的员工的经理的员工编号；因此，定义中的“`引用关系变量`”（_`R2`_）和“`被引用关系变量`”（_`R1`_）在此示例中是同一个。例如，员工 `E3` 的 `EMP 元组` 可能包含 `MNO 值` `E2`，这构成了对员工 `E2` 的 `EMP 元组` 的 `引用`。但是 `外键值`，就像 `候选键值` 一样，是 _`元组`_；从概念上讲，因此我们必须在 `外键` 规范中重命名 `MNO` `属性`，以便 `元组相等比较` 有效。（什么 `元组相等比较`？就是隐含在检查 `外键约束` 过程中的那个比较——回想一下，如果要对 `元组` 进行相等性测试，它们当然必须属于相同的 `类型`，而“相同类型”意味着它们必须具有相同的 `属性名`。）

顺便提一下，我应该提到最初制定的 `关系模型` 要求 `外键` 匹配的不仅是某个 `候选键`，而且非常具体地是 `_`主\_` `键`。然而，我在第一章中给出了不坚持总是选择某个 `候选键`并将其设为主键的理由；因此，我也不想坚持`外键`必须总是专门匹配`主键`。（在这一点上我同意 `SQL`。）

现在，`SQL` 不仅支持 `外键` 本身，还支持某些相关的 _`引用操作`_（referential actions），如 `CASCADE`（可在 `ON DELETE` 或 `ON UPDATE` 子句中指定）。例如，装运表的 `CREATE TABLE` 语句可能包含以下内容：

```
FOREIGN KEY ( SNO ) REFERENCES S ( SNO ) ON DELETE CASCADE
```

根据此规范，尝试删除特定供应商将 `级联` 删除该供应商的所有装运记录。我提到这一点的原因如下：

- 首先，此类规范在实践中可能很有用，但它们本身并不是 `关系模型` 的一部分。
- 但这不一定是问题！`关系模型` 是 `数据库` 领域的基础，但它 _`只是`_ 基础。没有理由不能在该基础之上或旁边构建附加功能——只要这些附加功能不违反模型的规定即可（当然，我认为还应补充：符合模型的精神并被证明有用）。详细说明：
  1. `类型理论` 提供了最明显的例子。我们在第二章中看到“`类型` 与 `表` 正交”，但我们也看到 `关系系统` 中完整且正确的 `类型支持` 是非常可取的，至少可以说如此。
  2. 作为第二个例子，`关系模型` 对 `恢复` 和 `并发控制` 几乎一无所知，但这显然并不意味着 `关系系统` 不应该提供这些控制。（实际上可以认为 `关系模型` 隐式地涉及了这些问题，因为它依赖 `DBMS` 正确实现 `更新` 且不丢失数据——但它没有规定任何具体内容。）

本节最后一点：我讨论了 `外键`，因为它们具有相当大的实用重要性，并且也是最初定义中模型的一部分。但我认为应该强调，它们并非真正基础性的——它们实际上只是某些在实践中常用的 `完整性约束` 的简写，正如我们将在第六章看到的。[^2]（事实上，`候选键` 也是如此，但在这种情况下，提供简写的实际好处是压倒性的。）

## 关于视图的更多讨论

`视图`[^3][^4]，也称为 `虚关系变量`，是一个本身没有独立存在，但对用户而言看起来有的 `关系变量`。定义如下：

> _`定义：`_ _`视图`_ 或 _`虚关系变量 V`_ 是一个 `关系变量`，其在时间 _`t`_ 的 `值` 是在该时间 _`t`_ 评估某个 `关系表达式` 的结果。该 `表达式` 在定义 _`V`_ 时指定，并且必须至少提及一个 `关系变量`。

以下是几个示例，“伦敦供应商”和“非伦敦供应商”（左侧为 **`Tutorial D`**，右侧为 `SQL`）：

```
VAR LS VIRTUAL                    |  CREATE VIEW LS AS
( S WHERE CITY = 'London' ) ;     |  ( SELECT S.*
                                  |  FROM   S
                                  |  WHERE  S.CITY = 'London' ) ;

VAR NLS VIRTUAL                   |  CREATE VIEW NLS AS
( S WHERE CITY ≠ 'London' ) ;  |                  ( SELECT S.*
                        FROM   S    |
                                  |   WHERE  S.CITY <> 'London' ) ;
```

所有这些示例中的括号都是不必要的，但并非错误。我包含它们是为了清晰。

### 视图检索

重申一下，`视图` 旨在让用户觉得它们有自己独立的存在；换句话说，就用户而言，它们应该“看起来和感觉起来”就像 `基关系变量`。特别是，用户应该能够像操作 `基关系变量` 一样操作 `视图`，并且 `DBMS` 应该能够将这些用户 `操作` 映射到定义 `视图` 的底层 `基关系变量` 上的合适 `操作`。我这里说“底层”，是因为（当然）如果 `视图` 真的表现得像 `基关系变量`，我们可以做的一件事就是在它们之上定义进一步的 `视图`，如下 `SQL` 示例所示：

```
CREATE VIEW LS_STATUS
  AS ( SELECT LS.SNO, LS.STATUS
       FROM   LS ) ;
```

映射只读 `操作` 是直接的。例如，假设我们对 `视图` `LS` 发出此 `SQL 查询`：

```
SELECT LS.SNO
FROM   LS
WHERE  LS.STATUS > 10
```

首先，`DBMS` 将 `FROM` 子句中对 `视图` 的 `引用` 替换为定义该 `视图` 的 `表达式`，得到：

```
SELECT LS.SNO
FROM ( SELECT S.*
       FROM   S
       WHERE  S.CITY = 'London' ) AS LS
WHERE  LS.STATUS > 10
```

现在可以将此 `表达式` 简化为：

```
SELECT S.SNO
FROM   S
WHERE  S.CITY = 'London'
AND    S.STATUS > 10
```

上述过程之所以有效，正是由于 `关系代数` 的 _`闭包`_ 属性。`闭包` 意味着，除了其他内容之外，只要我们允许放置某物的 _`名称`_——例如在 `查询` 中——我们总是可以放置一个更一般的 `表达式`，该表达式会评估为适当类型的 `事物`。[^5] 例如，在 `FROM` 子句中，我们可以有一个 `SQL 表名`；因此我们也可以有一个更一般的 `SQL 表表达式`，这就是为什么在示例中允许用定义 `视图` `LS` 的 `表达式` 替换名称 `LS`。

顺便提一下，值得一提的是上述过程在早期版本的 `SQL` 中并不总是有效（具体而言，是在 1992 年之前的 `SQL 标准` 版本中），因为那些早期版本未能正确支持 `闭包`。结果，针对某些看似无害的 `表`（实际上是 `视图`）的某些看似无害的 `查询` 会失败——而且失败的方式很难解释。这是一个简单示例：

```
CREATE VIEW V
  AS ( SELECT S.CITY, SUM ( S.STATUS ) AS ST
       FROM   S
       GROUP  BY S.CITY ) ;


SELECT V.CITY
FROM   V
WHERE  V.ST > 25
```

此示例在 1992 年之前的 `SQL 标准` 中失败了。尽管 `标准` 现已修复，但并不意味着所有产品都已修复！事实上，至少在撰写本文时（2005 年初），至少有一个主要产品仍然没有修复。

### 视图更新

现在转向 `更新操作`。在深入细节之前，我想再次看看伦敦和非伦敦供应商 `视图`（现在我将切换到 **`Tutorial D`**）：

```
VAR LS  VIRTUAL ( S WHERE CITY = 'London' ) ;

VAR NLS VIRTUAL ( S WHERE CITY ≠ 'London' ) ;
```

这里的重要一点如下：与其让 `S` 作为 `基关系变量`，`LS` 和 `NLS` 作为 `视图`，_`LS 和 NLS 也可以是基关系变量，而 S 可以是视图`_，如下所示：

```
VAR LS BASE RELATION
  { SNO SNO, SNAME NAME, STATUS INTEGER, CITY CHAR }
    KEY { SNO } ;

VAR NLS BASE RELATION
  { SNO SNO, SNAME NAME, STATUS INTEGER, CITY CHAR }
    KEY { SNO } ;

VAR S VIRTUAL ( LS UNION NLS ) ;
```

> **注**
>
> 为了实现完全等价，我们还必须指定某些 `约束`——特别是规定 `LS` 中的每个 `CITY 值` 都是 London，且 `NLS` 中的每个 `CITY 值` 都 _`不是`_ London 的 `约束`——但我在此省略了这些细节。见第六章对此类问题的进一步讨论。

此示例传达的信息是，在很大程度上，_`哪些关系变量是基变量，哪些是虚变量是任意的`_——由此可以推断，在 `基` 和 `虚关系变量` 之间绝不能存在任意且不必要的区别。这种状态被称为 _`可互换性原则`_（基与虚关系变量之间）。以下是一些含义：

- 与 `基关系变量` 一样，`视图` 也受 `完整性约束` 的约束。（我们通常认为 `完整性约束` 仅适用于 `基关系变量`，但 _`可互换性原则`_ 表明这种立场实际上站不住脚。）
- 特别是，`视图` 具有 `候选键`（因此我或许应该在此之前的 `视图` 示例中包含一些 `键` 规范；**`Tutorial D`** 允许此类规范，但 `SQL` 不允许）。它们也可能具有 `外键`，并且 `外键` 可能引用它们。
- 我在第一章中没有提到这一点，但“`实体完整性`”规则被认为专门适用于 `基关系变量`，而不是 `视图`。这违反了 _`可互换性原则`_。（当然，我无论如何都拒绝该规则，因为它与 `空值` 有关。）
- 许多 `SQL 产品` 和 `SQL 标准` 提供了某种“`行 ID`”功能。如果该功能适用于 `基表` 而不适用于 `视图`——这在实践中相当常见——它违反了 _`可互换性原则`_。[^6] 当然，`行 ID` 本身并不是 `关系模型` 的一部分，但这一事实本身并不意味着它们不应该被支持。但我作为一个重要的补充观察到，如果这些 `行 ID` 被视为某种 _`对象`_ ID（在 `面向对象` 的意义上）——正如它们不幸地在 `SQL 标准` 以及大多数主要 `SQL 产品` 中所做的那样——那么它们绝对是禁止的！`对象 ID` 实际上是 _`指针`_，而 `关系模型` 明确禁止 `指针`。

并且，回到本讨论的主要点，我们 _`必须`_ 能够更新 `视图`——因为如果我们不能，那么这一事实本身就构成了对 _`可互换性原则`_ 最明显的违反。

如你所知，`SQL` 对此要求的支持相当薄弱，无论是在 `标准` 中还是在主要商业产品中。它通常至少包含对定义为单个底层 `基关系变量` 的简单 `限制` 和/或 `投影` 的 `视图` 上的 `更新` 的支持（尽管即使在这里也存在一些问题）。例如，考虑以下 `视图`（本质上与我们在第一章中看到的相同）：

```
CREATE VIEW SST_PARIS
  AS ( SELECT S.SNO, S.STATUS
       FROM   S
       WHERE  S.CITY = 'Paris' ) ;
```

此 `视图` 是 `基表` `S` 的 `限制` 的 `投影`，因此我们例如可能能够对其执行以下 `DELETE`：

```
DELETE
FROM   SST_PARIS
WHERE  SST_PARIS.STATUS > 15 ;
```

此 `DELETE` 映射为：

```
DELETE
FROM   S
WHERE  S.CITY = 'Paris'
AND    S.STATUS > 15 ;
```

但很少有产品提供对远比此复杂的 `视图` 的更新支持。

不幸的是，我现在正涉足一个仍存在一定争议的领域。我个人的观点是，`视图更新` 问题已基本解决（即理论已存在）；然而，并非每个人都同意我的观点，而且无论如何，对该主题的详细讨论需要比本书所能包含的更多的背景知识。因此，恐怕我所能做的就是将你引向另一本书——C. J. Date 和 Hugh Darwen 合著的《Databases, Types, and the Relational Model: The Third Manifesto》第三版（Addison-Wesley, 2006），该主题在其中得到了深入探讨——如果你想要更多具体细节。

### 杂项说明

为了结束本节，我还需要说几件事。首先，众所周知但值得一提，`视图` 服务于两个相当不同的目的：

- 实际定义 `视图` _`V`_ 的用户显然知道 _`V`_ 是根据 `表达式` _`X`_ 定义的。该用户可以在任何打算使用 `表达式` _`X`_ 的地方使用名称 _`V`_，但这种使用基本上只是简写。
- 相比之下，仅仅被告知 _`V`_ 存在并可供使用的用户应该（至少在理想情况下）_`不知道`_ `表达式` _`X`_；对该用户而言，_`V`_ 应该看起来和感觉起来就像一个 `基关系变量`，正如我们已经看到的。而 `视图` 的这第二种用途才是真正重要的，也是我在此之前的本节中一直 tacitly 集中讨论的。

其次，当我在本节开头解释什么是 `视图` 时，我说定义 `视图` 的 `关系表达式` 必须至少提及一个 `关系变量`。为什么？因为如果它不提及，那么“`虚关系变量`”根本就不会是一个 `关系变量`！——我的意思是，它不会是一个 _`变量`_，并且肯定不可更新。相反，它将是一个 _`关系常量`_，或者我们可以称之为“`relcon`”。例如（临时发明一些语法）：

```
CONST PERIODIC_TABLE ( RELATION {
      TUPLE { ELEMENT 'Hydrogen', SYMBOL 'H' , ATOMICNO  1 },
      TUPLE { ELEMENT 'Helium'  , SYMBOL 'He', ATOMICNO  2 },
      ...
      TUPLE { ELEMENT 'Uranium' , SYMBOL 'U' , ATOMICNO 92 }
                                } ) ;
```

虽然提供上述某种“`relcon`”功能可能确实可取，但我认为我们不应将此类事物视为 `关系变量`。我认为假装 `常量` 是 `变量` 无助于理解。

第三，正如我撰写本文时出现的一个不幸的术语冲突，当然在学术界，在某种程度上也在商业界。回想第一章，`视图` 可以被认为是一种 `派生关系变量`。好吧，还有另一种 `派生关系变量`，称为 _`快照`_（snapshot）。顾名思义，`快照` 虽然是派生的，但是真实的，而不是虚拟的——这意味着它不仅通过其在其他 `关系变量` 方面的定义来表示，而且至少在概念上，还通过其自身单独的数据副本来表示。例如（再次临时发明一些语法）：

```
VAR LSS SNAPSHOT ( S WHERE CITY = 'London' )
    REFRESH EVERY DAY ;
```

定义 `快照` 就像执行 `查询`，只不过：

- `查询` 的结果以指定名称（示例中的 `LSS`）保存在 `数据库` 中，作为只读 `关系变量`（只读，即除了定期刷新之外；见下一要点）。
- 定期（示例中的 `EVERY DAY`）`快照` 会被刷新——其当前 `值` 被丢弃，`查询` 再次执行，该新执行的结果成为新的 `快照值`。

因此，在示例中，`快照` `LSS` 表示最多 24 小时前的数据。

`快照` 在 `数据仓库`、`分布式系统` 和许多其他上下文中很重要。在所有情况下，基本原理是许多应用程序可以容忍，甚至可能需要“`截至`”某个特定时间点的数据。报告和会计应用程序就是一个例证；此类应用程序通常要求数据在适当时刻冻结（例如，在会计期间结束时），而 `快照` 允许在不阻塞其他应用程序的情况下进行此类冻结。

到目前为止，一切顺利。问题是 `快照` 已经变得（至少在某些圈子里）不再被称为 `快照`，而是被称为 _`物化视图`_（materialized views）。但 `快照` _`不是`_ `视图`！事实上，就 `关系模型` 而言，`视图` 的全部要点是它们 _`不会`_ 被物化；正如我们所见，对 `视图` 的 `操作` 是通过将它们映射到底层 `基关系变量` 上的合适 `操作` 来实现的。因此，“`物化视图`”本身就是自相矛盾的。更糟糕的是，无修饰的术语 _`视图`_ 现在经常被专门用来表示“`物化视图`”——同样，至少在某些圈子里——因此我们面临着不再有一个好术语来表示原始意义上的 `视图` 的危险。当然，在本书中，我确实使用术语 _`视图`_ 表示其原始意义，但请注意它在其他地方并不总是具有该含义。_`Caveat lector`_（读者请留心）。

## 关系变量与谓词

现在我们来到本章在许多方面最重要的部分。其核心是这样的：还有另一种思考 `关系变量` 的方式。我的意思是，大多数人认为 `关系变量` 就像是传统计算意义上的文件——也许是相当抽象的文件（也许用“有纪律的”比抽象更好），但仍然是文件。但有一种不同的看待它们的方式，我认为这种方式可以导致对真正发生的事情的更深层次理解。它是这样的。

考虑供应商 `关系变量` `S`。像所有 `关系变量` 一样，该 `关系变量` 应该代表现实世界的某个部分。实际上，我可以更精确：==该 `关系变量` 的 `关系头` 代表某个 _`谓词`_（predicate），意味着它是关于现实世界某个部分的某种通用陈述（它是通用的，因为它是 _`参数化`_ 的，我稍后会解释）。该 `谓词` 看起来如下：==

> 供应商 `SNO` 处于合同下，名为 `SNAME`，状态为 `STATUS`，位于城市 `CITY`。

此 `谓词` 是 `关系变量` `S` 的 _`预期解释`_——换句话说，是它的 _`含义`_，也称为 _`内涵`_（intension，注意拼写）。

你可以将 `谓词` 视为一个 _`真值函数`_。像所有函数一样，它有一组参数，在被调用时返回结果，并且（因为它是真值的）该结果是 `TRUE` 或 `FALSE`。例如，对于刚刚显示的 `谓词`，参数是 `SNO`、`SNAME`、`STATUS` 和 `CITY`（当然对应于 `关系变量` 的 `属性`），它们代表适用 `类型` 的 `值`（分别是 `SNO`、`NAME`、`INTEGER` 和 `CHAR`）。当我们调用该函数时——正如逻辑学家所说的 _`实例化谓词`_——我们用 `参数` 替换参数。假设我们分别替换参数 `S1`、`Smith`、`20` 和 `London`。那么我们得到以下 _`命题`_（proposition）：

> 供应商 `S1` 处于合同下，名为 `Smith`，状态为 `20`，位于城市 `London`。

一般而言，逻辑中的 `命题` 是无条件为真或为假的陈述。以下是几个示例：

> Edward Abbey 写了《monkey Wrench Gang》。

> William Shakespeare 写了《monkey Wrench Gang》。

第一个为真，第二个为假。不要落入认为 `命题` 必须总是为真的常见陷阱！然而，我在与 `关系变量` 相关的上下文中谈论的那些 _`是`_ 专门指为真的那些，正如我现在解释的：

- 首先，_`每个`_ `关系变量` 都有一个关联的 `谓词`，称为该 `关系变量` 的 _`关系变量谓词`_。
- 设 `关系变量` _`R`_ 具有 `谓词` _`P`_。那么在给定时间出现在 _`R`_ 中的每个 `元组` _`t`_ 都可以被视为代表某个 `命题` _`p`_，该 `命题` 是通过在该时间用来自 _`t`_ 的 `属性值` 作为 `参数` 调用（或实例化）_`P`_ 而派生的。
- 并且（非常重要！）我们按照惯例假设以这种方式获得的每个 `命题` _`p`_ 都评估为 `TRUE`。

因此，给定我们通常的 `关系变量` `S` 的示例 `值`，我们假设以下 `命题` 都评估为 `TRUE`：

> 供应商 `S1` 处于合同下，名为 `Smith`，状态为 `20`，位于城市 `London`。

> 供应商 `S2` 处于合同下，名为 `Jones`，状态为 `10`，位于城市 `Paris`。

> 供应商 `S3` 处于合同下，名为 `Blake`，状态为 `30`，位于城市 `Paris`。

依此类推。不仅如此，我们更进一步：如果某个 `元组` 在某个时间合理地可能出现在某个 `关系变量` 中但实际上没有，那么我们假设相应的 `命题` 在该时间为假（换句话说，我们采用所谓的 _`封闭世界假设`_）。例如，`元组`：

```
TUPLE { SNO SNO('S6'), SNAME NAME('Lopez'),
                       STATUS 30, CITY 'Madrid' }
```

是——我们同意——一个合理的供应商 `元组`，但此时并未出现在 `关系变量` `S` 中，因此我们假设 _`并非`_ 以下 `命题` 在此时为真：

> 供应商 `S6` 处于合同下，名为 `Lopez`，状态为 `30`，位于城市 `Madrid`。

换句话说，给定的 `关系变量` 在任何给定时间都包含 _`所有`_ 且 _`仅`_ 代表该时间为真 `命题`（`谓词` 的真实例化）的 `元组`。

> _`更多术语：`_ 同样，设 _`P`_ 是 `关系变量` _`R`_ 的 `关系变量谓词` 或 `内涵`，且 _`R`_ 在某个给定时间的 `值` 为 `关系` _`r`_。那么 _`r`_——或者更准确地说是 _`r`_ 的 `体`——在该时间构成 _`P`_ 的 _`外延`_（extension）。因此请注意，`外延` 随时间变化，但 `内涵` 不变化。

### 关系表达式

我在本节中讨论的所有思想都以自然的方式扩展，以适用于任意的 `关系表达式`。例如，考虑这个 `表达式`，它表示供应商在所有 `属性` 上除 `CITY` 外的 `投影`：

```
S { SNO, SNAME, STATUS }
```

结果包含所有以下形式的 `元组`：

```
TUPLE { SNO s, SNAME n, STATUS t }
```

使得以下形式的 `元组`：

```
TUPLE { SNO s, SNAME n, STATUS t, CITY c }
```

当前出于某个 `CITY 值` _`c`_ 出现在 `S` 中。换句话说，结果代表以下 `谓词` 的当前 `外延`：

> 存在某个城市 `CITY`，使得供应商 `SNO` 处于合同下，名为 `SNAME`，状态为 `STATUS`，位于城市 `CITY`。

观察到该 `谓词` 只有三个参数，相应的 `关系`（供应商在所有但 `CITY` 上的 `投影`）也只有三个 `属性`——`CITY` _`不是`_ 参数，而是逻辑学家所谓的 _`约束变量`_（bound variable），这是因为它被短语 _`存在某个城市`_ “`量化`”了。（有关 `约束变量` 和 `量词` 的进一步解释，见附录 A。）可能更清晰地说明同一点——该 `谓词` 只有三个参数，而不是四个——是观察到所讨论的 `谓词` 在逻辑上等价于以下这个：

> 供应商 `SNO` 处于合同下，名为 `SNAME`，状态为 `STATUS`，位于某个城市（_`换句话说，在某处，但我们不知道在哪里`_）。

从以上所有内容可以推断，特别是 `视图` 代表某些 `谓词`。例如，如果 `视图` `SST` 定义如下：

```
VAR SST VIRTUAL ( S { SNO, SNAME, STATUS } ) ;
```

那么 `SST` 的 `关系变量谓词` 正是：

> 供应商 `SNO` 处于合同下，名为 `SNAME`，状态为 `STATUS`，位于某个城市。

关于 `谓词` 和 `命题`，我还有最后一点要说明。我说过 `谓词` 有一组参数。像往常一样，该 `集合` 可能是空的——如果它是空的，`谓词` 就变成了 `命题`！（它当然是一个无条件为真或为假的陈述。）换句话说，`命题` 是一个 _`退化`_ 的 `谓词`；所有 `命题` 都是 `谓词`，但大多数 `谓词` 不是 `命题`。

## 关于关系与类型的更多讨论

第二章的 _`标题是关系与类型`_。然而，我在该章中无法解释这两个概念之间最重要的区别——但现在我可以了，我将解释。

我已经表明，`数据库` 在任何给定时间都可以被视为一组真 `命题`：例如，_`命题`_ _`供应商 S1 处于合同下，名为 Smith，状态为 20，位于城市 London`_。更具体地说，我已经表明出现在此类 `命题` 中的 `参数值`（示例中的 `S1`、`Smith`、`20` 和 `London`）正是相应 `元组` 中的 `属性值`，并且当然每个此类 `属性值` 都是关联 `类型` 的 `值`。由此可以得出：

> **类型是我们能够谈论的事物的集合；**

> **关系是关于这些事物的（真）陈述。**

换句话说，`类型` 给我们提供了词汇表——我们可以谈论的事物——而 `关系` 赋予我们谈论我们可以谈论的事物的能力。（这里有一个可能有所帮助的类比：_`类型之于关系，犹如名词之于句子`_。）例如，如果我们为了简化，将注意力仅限于供应商，我们会看到：

- 我们可以谈论的事物是供应商编号、名称、整数和字符串——没有别的。
- 我们可以说的事物是形如“`具有指定供应商编号的供应商处于合同下，具有指定名称，具有由指定整数表示的状态，位于由指定字符串表示的城市`”的事物——没有别的。（即，除了由我们可以说的事物 _`逻辑蕴含`_ 的事物之外。例如，给定我们已经知道可以说的关于供应商 `S1` 的事物，我们也可以说像 _`供应商 S1 处于合同下，名为 Smith，状态为 20，位于某个城市`_ 这样的事物——其中城市未指定。如果你认为我刚刚告诉你的内容非常 reminiscent of，并且可能与关系 `投影` 有某种深刻联系……嗯，你完全正确。）

上述情况至少有三个重要的推论。具体而言，为了表示“现实世界的某个部分”（正如我在上一节中所说）：

1. `类型` 和 `关系` 都是 _`必要`_ 的——没有 `类型`，我们就没有什么可谈论的；没有 `关系`，我们就不能说任何话。
2. `类型` 和 `关系` 合在一起是 _`充分`_ 的，以及必要的——逻辑上我们不需要其他任何东西。（好吧，我们确实需要 `关系变量`，以便反映现实世界随时间变化的事实，但我们不需要它们来表示任何给定时间的情况。）
3. `类型` 和 `关系` _`不是同一件事`_。小心任何试图假装它们是的人！事实上，假装 `类型` 只是一种特殊类型的 `关系` 正是某些商业产品试图做的事情（尽管它们通常不以这种术语谈论）——我希望很明显，任何建立在这种逻辑错误之上的产品注定最终会失败。我想到的产品当然不是 `关系产品`；通常，它们是支持 `面向对象` 意义上的 `对象` 的产品，或者试图以某种方式将此类 `对象` 和 `SQL 表` 结合的产品。（事实上，我想到的至少一种产品确实已经失败了。）更多细节超出了本书的范围。

我想通过提供对我所说内容的稍微更正式的视角来结束本节。我说过 `数据库` 可以被认为是一组真 `命题`。事实上，`数据库` 连同适用于该 `数据库` 中表示的 `命题`（或更确切地说是此类 `命题` 的 `集合`）的 `操作符`，是 _`一个逻辑系统`_。当我说“`逻辑系统`”时，我的意思是形式系统——例如欧几里得几何——它具有 _`公理`_（“给定真理”）和 _`推理规则`_，我们可以通过它们从这些 `公理` 证明 _`定理`_（“推导真理”）。事实上，当 Codd 在 1969 年首次发明 `关系模型` 时，他非常伟大的洞察是，（尽管名字如此）`数据库` 并不仅仅是一组 _`数据`_；相反，它是一组 _`事实`_，换句话说，真 `命题`。这些 `命题`——即给定的那些，也就是由 `基关系变量` 表示的那些——是所讨论的逻辑系统的 `公理`。`推理规则` 本质上是新 `命题` 可以从给定命题推导出来的规则；换句话说，它们是告诉我们如何应用 `关系代数` 的 `操作符` 的规则。因此，当系统评估某个 `关系表达式` 时（特别是当它响应某个 `查询` 时），它实际上是从给定真理中推导新真理；实际上，它在证明一个定理！

一旦我们理解了上述内容，我们就可以看到形式逻辑的整个 apparatus 变得可用于解决“`数据库问题`”。换句话说，诸如以下问题：

- `数据库` 对用户应该看起来像什么？
- `完整性约束` 应该看起来像什么？
- `查询语言` 应该看起来像什么？
- 我们如何最好地实现 `查询`？
- 更一般地说，我们如何最好地评估 `数据库表达式`？
- 结果应该如何呈现给用户？
- 我们最初应该如何设计 `数据库`？

（以及其他类似问题）都实际上成为逻辑问题， susceptible to logical treatment，并且可以给出逻辑答案。

当然，不言而喻的是，`关系模型` 非常直接地支持上述认知——这就是为什么，在我看来，该模型是坚如磐石的、“正确的”，并且将长存。这也是为什么，同样在我看来，其他“`数据模型`”根本不在同一个水平线上。事实上，我严重质疑这些其他“`模型`”是否配得上被称为模型，就像 `关系模型` 可以被称为模型一样。当然，它们大多数在一定程度上是临时拼凑的，而不是像 `关系模型` 那样牢固地建立在集合论和谓词逻辑之上。我将在第八章详细阐述这些问题。

## 总结

本章最重要的部分是关于 `谓词` 的部分（以及随后的部分，讨论 `关系` 与 `类型` 的相关问题）。基本上，每个 `关系变量` _`R`_ 都有一个关联的 `谓词` _`P`_（_`R`_ 的 _`关系变量谓词`_）；_`P`_ 是 _`R`_ 的 _`预期解释`_ 或 _`内涵`_，并且它不随时间变化。如果 _`R`_ 的当前 `值` 是 `关系` _`r`_，那么 _`r`_ 代表 _`P`_ 的 _`当前外延`_；它由一组 `元组` 组成，每个 `元组` 代表一个真 `命题`，换句话说，_`P`_ 的一个真实例化。`外延` 确实随时间变化。因此，`数据库` 连同其 `操作符` 可以被视为一个 _`逻辑系统`_。

以下是本章正文中的一些其他重要点：

- 只有 `关系变量` 是可更新的；谈论“`更新一个元组`”或“`更新一个属性`”是方便的但粗糙的。`更新` 始终是 `集合级` 的。
- 每个 `关系变量` 至少有一个（`候选`）`键`。`键` 具有 `唯一性` 和 `不可约性`。`键值` 是 `元组`。
- 某些 `关系变量` 具有 `外键`。`SQL` 支持某些相关的“`引用操作`”，如 `CASCADE`；此类操作在实践中可能非常有用，但它们不是 `关系模型` 的一部分。此外，`外键` 本身也并非真正基础性的。
- 对 `视图` 的 `操作` 是通过将它们映射到适当的底层 `基关系变量` 上的 `操作` 来实现的。（映射过程基本上由于 `闭包` 而有效，对于只读 `操作` 是直接的，但对于 `更新操作` 则不然。）_`可互换性原则`_ 指出在 `基` 和 `虚关系变量` 之间绝不能存在任意且不必要的区别。
- “`类型之于关系，犹如名词之于句子`。”
- `类型` 和 `关系` 都是必要且充分的，足以表示我们喜欢的任何数据。（即在逻辑层面上；当然，我们知道其他构造在物理层面上很有用，但这是因为该层面的目标不同。物理层面故意不在 `关系模型` 的管辖范围内。）
