再次考虑我们常用的供应商`关系变量` `S`。由于 `{SNO}` 是一个`键`，该`关系变量` 肯定受制于 `FD` `{SNO}` → `{STATUS}`。因此，令 _X_ 为 `{SNO}`，_Y_ 为 `{STATUS}`，_Z_ 为 `{SNAME,CITY}`，`Heath 定理` 告诉我们，可以将该`关系变量` 分解为`关系变量` `SNC` 和 `ST`，其中 `SNC` 的`标题` 为 `{SNO,SNAME,CITY}`，`ST` 的`标题` 为 `{SNO,STATUS}`。图 6-1 显示了与图 3-1 中 `S` 的值相对应的 `SNC` 和 `ST` 的示例值：

**图 6-1**：`关系变量` `SNC` 和 `ST`——示例值

（此处为图示：显示包含 `SNO`、`SNAME`、`CITY` 属性的 `SNC` 关系，以及包含 `SNO`、`STATUS` 属性的 `ST` 关系）

在此分解中：

- `关系变量` `SNC` 和 `ST` 都处于 `BCNF`——`{SNO}` 是两者的`键`，且在这些`关系变量` 中成立的唯一`非平凡` `FD` 都是"从`超键` 出发的箭头"。
- 更重要的是，该分解肯定是`无损` 的（这实际上由 `Heath 定理` 保证）——如果我们将 `SNC` 和 `ST` 连接在一起，就能恢复 `S`。
- 然而，`FD` `{CITY}` → `{STATUS}` **已经丢失**——我的意思是，当然，它已被某个`多关系变量约束` 所取代，正如前一章所解释的那样。<sup>1</sup> 相关的`多关系变量约束` 可表述如下：

```
CONSTRAINT ... WITH ( SNCT := JOIN { SNC , ST } ) :
                      COUNT ( SNCT { CITY } ) =
                      COUNT ( SNCT { CITY , STATUS } ) ;
```

_解释_：此约束表示，如果我们将 `SNC` 和 `ST` 连接，会得到一个结果——称之为 `SNCT`——其中不同`城市` 的数量等于不同`城市/状态` 对的数量。而该后一属性成立的事实等价于说 `FD` `{CITY}` → `{STATUS}` 在该连接中成立，因此也在原始`关系变量` `S` 中成立（因为分解是`无损` 的，意味着 `S` 与该连接 `SNCT` 在逻辑上是相同的）。

因此，我们"丢失"了一个 `FD`。这意味着什么？当然，取代它的`多关系变量约束` 更难表述，正如我们刚才所见。也许更重要的是，它更难强制执行：即，比采用图 3-2 中所示的、投影到 `SNC` 和 `CT` 的优选分解更难。<sup>2</sup> 例如，假设我们更新`关系变量` `SNC`，将供应商 `S1` 的城市从伦敦改为雅典；那么我们也必须更新`关系变量` `ST`，将供应商 `S1` 的状态从 20 改为 30——因为如果不这样做，那么将 `SNC` 和 `ST` 重新连接在一起将产生一个不是`关系变量` `S` 的合法值的结果。（相比之下，如果我们更新`关系变量` `SNC`，将供应商 `S2` 的城市从巴黎改为雅典，那么我们*不必* 也更新`关系变量` `ST`——但我们仍然需要检查`关系变量` `ST` 才能确定这一事实。）

> **重要提示**：给定一个架构良好的 `DBMS`，或许可以让系统"自动"执行对`关系变量` `ST` 的必要检查，而无需用户手动操作。甚至可能让系统也"自动"执行任何必要的额外更新。然而，即使在这样的系统中，该约束仍然更难强制执行（即，仍然需要做更多的工作，即使是由系统而非用户完成）。无论如何，在撰写本文时，这些可能性都只是空想——当今的商业产品大多甚至不允许一般性地表述`多关系变量约束`，更不用说强制执行它们了；上述可能性在今天遥不可及，因此处理（尤其是强制执行）此类约束是用户的责任。

因此，传达的信息是：尽量选择保持 `FD` 的分解，而不是丢失它们的分解。（在当前情况下，将投影 `{SNO,STATUS}` 替换为 `{CITY,STATUS}` 即可解决问题。）松散地说，换句话说，如果 `FD` _X_ → _Y_ 在原始`关系变量` 中成立，尽量不要选择 _X_ 最终在一个`关系变量` 中而 _Y_ 在另一个`关系变量` 中的分解。_注意_：当然，我在这里假设分解不是基于该 `FD` _X_ → _Y_ 本身进行的——因为如果是，我们将有效地得到两个 _X_，其中一个必然与 _Y_ 在同一个`关系变量` 中，而另一个则不会。我还隐式地假设 _X_ → _Y_ 是所谓`不可约覆盖` 的一部分，该覆盖适用于原始`关系变量` 中成立的全部 `FD` `集合`。我将在本章后面讨论`不可约覆盖`。

### 一个不幸的冲突

保持 `FD` 的基本思想相当简单；然而不幸的是，存在一些复杂情况，事实证明关于这个主题还有相当多的内容需要说明。为了温和地介绍其中一些复杂情况，我将提出一个可能被视为病理性的示例。我们给定一个`关系变量` `SJT`，具有属性 `S`（学生）、`J`（科目）和 `T`（教师），谓词为*学生 `S` 由教师 `T` 教授科目 `J`*。以下`业务规则` 适用：<sup>3</sup>

- 对于每个科目，该科目的每个学生仅由一位教师教授。
- 每位教师仅教授一个科目。
- 每位学生学习多个科目，因此（通常）由多位教师教授。
- 每个科目（通常）由多位学生学习。
- 每个科目（通常）由多位教师教授。
- 同一科目的不同学生可能（也可能不）由同一位教师教授该科目。

符合这些规则的该`关系变量` 的示例值如图 6-2 所示：

**图 6-2**：`关系变量` `SJT` 的示例值

`关系变量` `SJT` 的 `FD` 是什么？根据第一条业务规则，我们有 `{S,J}` → `{T}`。根据第二条，我们有 `{T}` → `{J}`。对其余规则的仔细分析表明，除了`平凡` 的或`可约` 的（或两者兼有）`FD` 之外，没有其他 `FD` 成立。因此，成立的唯一`非平凡`、`不可约` 的 `FD` 是以下两个：

```
{ S , J } → { T }
{ T }     → { J }
```

那么`键` 是什么？`{S,J}` 是一个`键`，因为整个`标题` 显然在函数上依赖于 `{S,J}` 而不依赖于 `{S,J}` 的任何真子集。此外，`{S,T}` 也是一个`键`，因为：

1.  鉴于 `FD` `{T}` → `{J}` 成立，整个`标题` 肯定在函数上依赖于 `{S,T}`。
2.  鉴于 `FD` `{S}` → `{J}` 和 `{T}` → `{S}` _不_ 成立，整个`标题` 不在函数上依赖于 `{S,T}` 的任何真子集。

因此有两个`键`，`{S,J}` 和 `{S,T}`。<sup>4</sup> 也许更重要的是，`{T}` _不是_ 一个`键`，因此`关系变量` `SJT` 受制于一个不是"从`键` 出发的箭头"的 `FD`（更正式地说，它不是由`键` 蕴含的）。因此，该`关系变量` 不处于 `BCNF`，尽管它处于 `3NF`。（_练习_：验证此说法！）并且它遭受`冗余` 问题；例如，鉴于图 6-2 中显示的示例值，`Professor White` 教授 `Math` 这一事实出现了两次。正如您所料，因此它也遭受更新异常；例如，再次参考图 6-2，我们不能删除 `Jones` 正在学习 `Physics` 这一事实，而不会丢失 `Professor Brown` 教授 `Physics` 的信息。

现在，我们可以通过适当地分解该`关系变量` 来克服这些问题。将 `Heath 定理` 应用于 `FD` `{T}` → `{J}`（令 _X_、_Y_ 和 _Z_ 分别为 `{T}`、`{J}` 和 `{S}`），我们得到以下`无损` 分解：

```
TJ { T , J }
   KEY { T }
TS { T , S }
   KEY { T , S }
```

我将另一个练习留给您：(a) 显示这两个`关系变量` 对应于图 6-2 中 `SJT` 值的值，(b) 显示这些`关系变量` 处于 `BCNF`，以及 (c) 检查该分解确实避免了上述`冗余` 和更新异常。特别要注意，`FD` `{T}` → `{J}` 在此分解中变成了一个`键约束`；而在原始设计中，情况并非如此，因此它必须单独表述和强制执行（我的意思是，除了`键约束` 之外）。

到目前为止，一切顺利。但还有另一个问题。事实是，尽管分解为 `TJ` 和 `TS` 确实解决了某些问题，但不幸的是，它也引入了其他问题。具体来说，`FD`

```
{ S , J } → { T }
```

丢失了（当然，它不是由 `FD` `{T}` → `{J}` 蕴含的，而这是分解结果中成立的唯一`非平凡` `FD`）。因此，`关系变量` `TJ` 和 `TS` 不能独立更新。例如，尝试将`元组`

```
( Smith , Prof. Brown )
```

插入 `TS` 必须被拒绝，因为 `Professor Brown` 教授 `Physics`，而 `Smith` 已经由 `Professor Green` 教授 `Physics`；然而，如果不检查 `TJ`，就无法检测到这一事实。

总结来说，该示例表明如下：在`无损` 分解中，我们通常追求两个目标，`BCNF` 投影和 `FD` 保持，而遗憾的是，这些目标可能相互冲突。换句话说，并不总是可能同时实现两者。

现在，在本章早期草稿的这一点上，我写了以下内容：

> 那么我们放弃哪个目标呢？嗯，如果我能告诉您，我会的，但我不能。`SJT` 示例表明，`规范化` 理论尽管很重要，但就其本身而言还不够；我的意思是，有些问题它没有回答。因此，传达的信息是：_我们需要更多的科学！_ `规范化` 理论当然是科学的，但它并不能解决所有设计问题。

然而，在我的一位审稿人的敦促下，我得出的结论是，上述段落可能被夸大了。这里我们需要的与其说是更多的科学，不如说是更好的实现！也就是说，在当前这种情况下，容忍一个未正确`规范化` 的设计的主要论点是，当今的 `DBMS` 使得处理像示例中"丢失"的 `FD` 这样的`多关系变量约束` 相当棘手。因此，让我在此表明立场并正式声明，在存在冲突的情况下，`依赖保持` 可能是应该放弃的目标。<sup>5</sup>

### 另一个示例

我在上一节开头提出，`SJT` 示例可能被认为是病理性的。然而，现在我要声称它并非完全如此；我将给出更多示例，这些示例（就我而言）表明 `FD` 保持问题出现的频率可能比您想象的要高。

通常理解的`规范化` 是一个按顺序从 `1NF` 到 `2NF` 再到 `3NF`（等等）的过程。让我们同意将这种通常理解的过程——即按顺序从 `1NF` 到 `2NF` 再到 `3NF`（等等）——称为"常规`规范化过程`"。因此，在本节及接下来的两节中，我想展示一系列示例，以证明如果过于盲目地遵循常规`规范化过程`，它不一定是一个好主意。我的第一个示例涉及一个看起来像这样的`关系变量`：

```
RX1 { SNO , PNO , CITY , STATUS , QTY }
```

名称 `RX1` 代表"`关系变量` 示例 1"；谓词是*供应商 `SNO` 位于城市 `CITY`，该城市具有状态 `STATUS`，并供应零件 `PNO`，数量为 `QTY`*。（为简单起见，我忽略供应商名称。）让我们合理地假设，以下 `FD` 在该`关系变量` 中成立：

```
{ SNO }       → { CITY }
{ CITY }      → { STATUS }
{ SNO , PNO } → { QTY }
```

现在，直观上很明显，以下 `FD` 也隐式成立：<sup>6</sup>

```
{ SNO }       → { STATUS }
{ SNO , PNO } → { CITY , STATUS }
```

事实上，最后一个 `FD` 可以扩展为 `{SNO,PNO}` → _H_，其中 _H_ 是整个`标题`；换句话说，`{SNO,PNO}` 是`关系变量` `RX1` 的一个`键`。

现在回想一下，`关系变量` _R_ 处于 `2NF` 当且仅当，对于每个`键` _K_ 和每个`非键属性` _A_，`FD` _K_ → {_A_} 是`不可约` 的。显然，`RX1` 不处于 `2NF`，因为 `FD` `{SNO,PNO}` → `{CITY}` 是"从 `RX1` 的`键` 出发的 `FD`"，但不是`不可约` 的；具体来说，它不是`不可约` 的，因为 `FD` `{SNO}` → `{CITY}` 也在该`关系变量` 中成立。因此，常规`规范化过程` 会建议我们通过将 `Heath 定理` 应用于该 `FD` `{SNO}` → `{CITY}` 来分解该`关系变量`。但如果我们这样做，我们会得到：

```
RX1A { SNO , CITY }
     KEY { SNO }
RX1B { SNO , PNO , STATUS , QTY }
     KEY { SNO , PNO }
```

现在请注意，`FD` `{CITY}` → `{STATUS}` 在此分解中丢失了。因此，从这个示例中立即得出的一个教训是，`FD` 保持问题可能与从 `1NF` 到 `2NF` 的步骤相关——而不仅仅是与从 `3NF` 到 `BCNF` 的步骤相关，后者是上一节中 `SJT` 示例所说明的步骤。

> **重要提示**：此处的`关系变量` `RX1A` 肯定处于 `2NF`。相比之下，`关系变量` `RX1B` 不处于 `2NF`，因为 `FD` `{SNO,PNO}` → `{STATUS}` 是`可约` 的。因此，我们可以再次应用 `Heath 定理` 将其分解为在 `{SNO,STATUS}` 和 `{SNO,PNO,QTY}` 上的投影，两者都处于 `2NF`；然而，损害已经造成，可以说——`FD` `{CITY}` → `{STATUS}` 已经丢失。

如何在此示例中保持该 `FD`？一个答案是：不是基于 `FD` `{SNO}` → `{CITY}`，而是基于 `FD` `{SNO}` → `{CITY,STATUS}` 进行分解。但请仔细注意，此 `FD` 不是最初明确列出的 `FD` 之一，也不是我说过显然由那些明确 `FD` 蕴含的 `FD` 之一；因此，在常规`规范化过程` 中，它不太可能被选为分解的基础。然而，假设我们确实选择它并执行相应的分解。结果如下：

```
RX1A′ { SNO , CITY , STATUS }
      KEY { SNO }
RX1B′ { SNO , PNO , QTY }
      KEY { SNO , PNO }
```

在此分解中，`STATUS` 出现在`键` 为 `{SNO}` 的`关系变量` 中，而不是`键` 为 `{SNO,PNO}` 的`关系变量` 中，因此 `FD` `{CITY}` → `{STATUS}` 得以保持。_注意_：当然，此处的`关系变量` `RX1A′` 仍然不处于 `3NF`，因此我们可能希望进一步分解它。然而，同样，我们需要稍微小心一点；具体来说，我们需要基于 `FD` `{CITY}` → `{STATUS}` 而不是 `{SNO}` → `{STATUS}` 进行分解，否则我们将再次丢失一个 `FD`。但 `{CITY}` → `{STATUS}` 是常规`规范化过程` 会告诉我们要使用的 `FD`，因此这里应该没有问题。

现在，上述内容的一个替代方案是基于 `FD` `{CITY}` → `{STATUS}` 分解原始`关系变量` `RX1`，得到：

```
RX1A′′ { CITY , STATUS }
       KEY { CITY }
RX1B′′ { SNO , PNO , CITY , QTY }
       KEY { SNO , PNO }
```

此分解也保持了 `FD` `{CITY}` → `{STATUS}`。但请注意，此 `FD` 不是导致原始`关系变量` `RX1` 违反 `2NF` 的那个 `FD`（它不是"从真`子键` 出发的箭头"）；因此，同样，如果我们遵循常规`规范化过程`，在实践中我们不太可能在此阶段选择它作为分解的基础。另请注意，此处的`关系变量` `RX1B′′` 仍然不处于 `3NF`，因此我们可能希望进一步分解它。我将该进一步分解的细节留给您思考。

### …… 还有另一个

让我们看另一个示例。假设供应商被划分为类别（`C1`、`C2` 等），因此我们有一个`关系变量` `RX2`，看起来像这样（为简单起见，我忽略供应商名称，就像我在 `RX1` 示例中所做的那样）：

```
RX2 { SNO , CLASS , CITY , STATUS }
    KEY { SNO }
```

谓词是*供应商 `SNO` 属于类别 `CLASS`，位于城市 `CITY`，并具有状态 `STATUS`*。还假设 (a) 每个类别只有一个状态，(b) 每个城市也只有一个状态，但 (c) 类别和城市在其他方面完全独立。那么以下 `FD` 成立：

```
{ CLASS } → { STATUS }
{ CITY }  → { STATUS }
```

_注意_：我还假设存在一个生效的`业务规则`，规定对于任何给定的供应商，城市状态等于类别状态（这就是为什么我们只需一个 `STATUS` 属性而不是两个的原因）。<sup>7</sup>

现在回想一下，`关系变量` _R_ 处于 `3NF` 当且仅当，对于在 _R_ 中成立的每个`非平凡` `FD` _X_ → _Y_，要么 _X_ 是`超键`，要么 _Y_ 是`子键`（或两者兼有）。显然，`RX2` 不处于 `3NF`，因为在 `FD` `{CITY}` → `{STATUS}` 中，`{CITY}` 不是`超键`，`{STATUS}` 也不是`子键`。因此，常规`规范化过程` 会建议我们通过将 `Heath 定理` 应用于该 `FD` `{CITY}` → `{STATUS}` 来分解该`关系变量`。但如果我们这样做，我们会得到（两个投影`关系变量` 都处于 `3NF`）：

```
RX2A { CITY , STATUS }
     KEY { CITY }
RX2B { SNO , CLASS , CITY }
     KEY { SNO }
```

现在请注意，`FD` `{CLASS}` → `{STATUS}` 在此分解中丢失了。（当然，如果我们基于该 `FD` 而不是 `FD` `{CITY}` → `{STATUS}` 进行分解，那么后一个 `FD` 就会丢失。）因此，现在我们看到 `FD` 保持问题也可能与从 `2NF` 到 `3NF` 的步骤相关。

现在，我们可以通过基于 `FD` `{SNO}` → `{CLASS,CITY}` 进行分解来保持此示例中的 `FD`——尽管同样，此 `FD` 不太可能被选为分解的基础，因为它没有明确表述。<sup>8</sup> 无论如何，结果如下：

```
RX2A′ { CLASS , CITY , STATUS }
      KEY { CLASS , CITY }
RX2B′ { SNO , CLASS , CITY }
      KEY { SNO }
```

在此分解中，`{CLASS,CITY}` 是 `RX2B′` 中的（复合）`外键`，引用 `RX2A′`。`关系变量` `RX2B′` 处于 `3NF`。然而，`关系变量` `RX2A′` 甚至不处于 `2NF`，因为 `FD` `{CLASS,CITY}` → `{STATUS}` 显然是`可约` 的。因此，如果我们决定保留该`关系变量`，`FD` `{CLASS}` → `{STATUS}` 和 `{CITY}` → `{STATUS}` 将必须单独表述和强制执行。<sup>9</sup> 或者，我们可以将该`关系变量` 分解为在 `{CLASS,STATUS}` 和 `{CITY,STATUS}` 上的投影，在这种情况下，将必须单独表述和强制执行一个适当的`多关系变量约束`。_练习_：该约束会是什么样子？

### …… 还有另一个

现在考虑上一节示例的一个修订版本，其中 (a) 供应商再次被划分为类别，但 (b) 现在每个类别只有一个城市（而每个城市又只有一个状态，如前所述）。因此，我们有一个`关系变量` `RX3`，看起来像这样（再次为简单起见，我忽略供应商名称）：

```
RX3 { SNO , CLASS , CITY , STATUS }
```

事实上，当然，`RX3` 与 `RX2` 具有相同的`标题`，但谓词不同：_供应商 `SNO` 属于类别 `CLASS`，该类别具有城市 `CITY`，该城市具有状态 `STATUS`_。以下 `FD` 成立（以及其他）：

```
{ SNO }   → { CLASS }
{ CLASS } → { CITY }
{ CITY }  → { STATUS }
```

`关系变量` `RX3` 不处于 `3NF`，因为在 `FD` `{CLASS}` → `{CITY}` 中，`{CLASS}` 不是`超键`，`{CITY}` 也不是`子键`。（对于 `{CITY}` → `{STATUS}`，情况类似，_mutatis mutandis_。）因此，常规`规范化过程` 会建议我们通过将 `Heath 定理` 应用于该 `FD` `{CLASS}` → `{CITY}` 来分解该`关系变量`。但如果我们这样做，我们会得到：

```
RX3A { CLASS , CITY }
     KEY { CLASS }
RX3B { SNO , CLASS , STATUS }
     KEY { SNO }
```

在此分解中，`RX3A` 处于 `3NF`，但 `RX3B` 仅处于 `2NF`——并且如您所见，`FD` `{CITY}` → `{STATUS}` 丢失了。事实上，基于该 `FD` `{CITY}` → `{STATUS}` 进行分解会更好：

```
RX3A′ { CITY , STATUS }
      KEY { CITY }
RX3B′ { SNO , CLASS , CITY }
      KEY { SNO }
```

`RX3A′` 处于 `3NF`，而 `RX3B′` 仅处于 `2NF`，但至少 `FD` `{CITY}` → `{STATUS}` 得以保持。更重要的是，我们现在可以基于 `FD` `{CLASS}` → `{CITY}` 继续分解 `RX3B′`，得到：

```
RX3BA′ { CLASS , CITY }
       KEY { CLASS }
RX3BB′ { SNO , CLASS }
       KEY { SNO }
```

这些`关系变量` 都处于 `3NF`。

因此，现在我们看到了四个不同的分解示例，其中 `FD` 可能丢失或已经丢失。关于这个主题还有很多可说的，但一个明确的信息是：常规`规范化过程`——事实上，实践中经常教授的那个！——在几个方面是不充分的。具体来说：

- 传统观点认为，`FD` 保持仅与从 `3NF` 到 `BCNF` 的步骤相关，但正如我们所见，情况未必如此。
- 常规`规范化过程` 通常建议作为分解基础的 `FD` 不一定是最佳选择。
- 该过程还假设最佳设计可以通过按顺序从 `1NF` 到 `2NF` 再到 `3NF`（等等）逐步找到，这同样未必是事实。

当然，"第一"、"第二"等的命名法本身就强化了这最后一种认知；但这种命名法在某种程度上真的只是一个历史偶然事件。我的意思是，如果首先定义的`范式` 是 `BCNF`——这很容易实现，因为其定义在概念上非常简单，不涉及 `FD` `不可约性`、`非键属性`、`子键`、`1NF`、`2NF` 或 `3NF` 的提及——那么实际上就根本不需要将 `2NF` 和 `3NF` 作为特定的`范式` 单独提出来。<sup>10</sup>

### 一个有效的过程

现在这里有一个过程，它*保证* 产生一个分解，其中所有`关系变量` 都处于 `3NF`（尽管不一定是 `BCNF`）并且所有 `FD` 都得以保持。<sup>11</sup> 为方便起见，我在下文中将其称为 _`3NF 过程`_。输入是一个`关系变量` _R_ 和所谓的*不可约覆盖* _C_（假设），适用于在 _R_ 中成立的 `FD`。我将在稍后解释什么是`不可约覆盖`——顺便说一句，又是`不可约` 这个词——但让我先陈述该过程：

1.  令 _S_ 为一个`标题` `集合`。将 _S_ 初始化为空`集合` `{}`。
2.  令 _X_ 为 _C_ 中某个 `FD` 的左侧（`决定因素`）；令 _C_ 中左侧为 _X_ 的完整 `FD` `集合` 为 _X_ → _Y1_, ..., _X_ → _Yn_；并令 _Y1_, ... _Yn_ 的并集为 _Y_。将 _X_ 和 _Y_ 的并集添加到 _S_。对每个不同的 _X_ 执行此步骤。
3.  令 _U_ 为 _R_ 的属性`集合` 中未包含在 _S_ 的任何元素中的属性。如果 _U_ 非空，则将 _U_ 添加到 _S_。
4.  如果 _S_ 的任何元素都不是 _R_ 的`超键`，则将 _R_ 的某个`键` _K_ 添加到 _S_。

在此过程结束时，_S_ 的元素是一组 `3NF` `关系变量` 的`标题`，_R_ 可以`无损` 分解为这些`关系变量` 而不会丢失任何 `FD`。特别要注意，该过程没有明确提及 `2NF`，甚至没有将其作为某种垫脚石。

那么它是如何工作的呢？显然，`不可约覆盖` 的概念很重要。为了解释这个概念，让我首先指出我之前几次顺便提到的内容：即，_某些 `FD` 蕴含其他 `FD`_。作为一个简单的例子，`FD` _X_ → _Y_ 和 _Y_ → _Z_ 共同蕴含 `FD` _X_ → _Z_——我的意思是，如果前两个 `FD` 被`关系` _r_ 满足，那么第三个也必须被 _r_ 满足。或者，也许更重要的是：如果前两个在`关系变量` _R_ 中成立，那么第三个也必须在 _R_ 中成立。我们在上一节中看到了一个说明，关于`关系变量` `RX3`，其中 `FD` `{CLASS}` → `{CITY}` 和 `{CITY}` → `{STATUS}` 都成立，因此 `FD` `{CLASS}` → `{STATUS}` 也成立。

因此，某些 `FD` 蕴含其他 `FD`。给定一个 `FD` `集合` _F_，那么我们可以有意义地谈论 _F_ 的一个*覆盖*。以下是定义：

> **`定义`（`覆盖`）**：`FD` `集合` _F_ 的一个`覆盖` 是一个 `FD` `集合` _C_，使得 _F_ 中的每个 `FD` 都由 _C_ 中的 `FD` 所蕴含。

作为一个平凡的例子，令 _F_ 为`集合`：

```
{ X → Y , Y → Z , X → Z }
```

那么以下两者都是 _F_ 的`覆盖`：

```
{ X → Y , Y → Z }
{ X → Y , Y → Z , X → Z }
```

这个例子立即说明了两点：首先，`覆盖` 通常不是唯一的；其次，任何 `FD` `集合` 肯定是其自身的`覆盖`，因为除其他事项外，每个 `FD` 都蕴含自身。第三个也是更重要的一点如下：强制执行给定`集合` _F_ 的`覆盖` _C_ 中的 `FD` 将"自动"强制执行该`集合` _F_ 中的那些 `FD`。因此，给定需要强制执行的一些 `FD` `集合` _F_，只需找到 _F_ 的某个`覆盖` _C_ 并强制执行 _C_ 中的 `FD` 即可。（事实上，强制执行 _F_ 的一个*不可约*`覆盖` 中的 `FD` 就足够了，这将很快变得清楚。）

现在我可以定义`覆盖` 为`不可约` 的含义：

> **`定义`（`不可约覆盖`）**：`FD` `集合` _F_ 的一个`覆盖` _C_ 是`不可约` 的，当且仅当它具有以下所有属性：
>
> 1.  _`单一依赖项`_：_C_ 中的每个 `FD` 在右侧只有一个属性。
> 2.  _`不可约决定因素`_：_C_ 中的每个 `FD` 本身是`不可约` 的。<sup>12</sup>
> 3.  _`无冗余 FD`_：不能从 _C_ 中丢弃任何 `FD` 而不失去 _C_ 是 _F_ 的`覆盖` 这一属性。

当然，以下问题立即出现：给定某个特定的 `FD` `集合`，我们如何找到该`集合` 的`不可约覆盖`？我将在下一章正确回答这个问题。现在，让我只给出一个示例——即，以下 `FD` `集合`，它们构成了在我们常用的供应商`关系变量` `S` 中成立的 `FD` 的`不可约覆盖`：

```
{ SNO }  → { SNAME }
{ SNO }  → { CITY }
{ CITY } → { STATUS }
```

简要阐述：当然，在 `S` 中成立的每个 `FD` 都由这三个 `FD` 共同蕴含，因此这三个确实构成了一个`覆盖`。此外，这三个中的每一个都有一个`单一依赖项`；不能从这三个`决定因素` 中的任何一个中删除任何属性；并且这三个 `FD` 中的任何一个都不能被丢弃。因此，可以得出结论，该`覆盖` 实际上是一个`不可约` 的`覆盖`。相比之下，以下 `FD` `集合` 也是在 `S` 中成立的 `FD` 的`覆盖`，但它们不是`不可约` 的（在每种情况下，为什么不是？）：

```
1.  { SNO }  → { SNAME , CITY }
    { CITY } → { STATUS }

2.  { SNO , SNAME } → { CITY }
    { SNO }         → { SNAME }
    { CITY }        → { STATUS }

3.  { SNO }  → { SNAME }
    { SNO }  → { CITY }
    { CITY } → { STATUS }
    { SNO }  → { STATUS }
```

现在让我们回到 `3NF` 过程。特别是，让我们看看它如何适用于 `SJT` 示例。<sup>13</sup> 提醒您一下，该`关系变量` 具有属性 `S`、`J` 和 `T`；`键` `{S,J}` 和 `{S,T}`；并且受制于 `FD` `{T}` → `{J}`。因此，以下 `FD` 成立：

```
{ S , J } → { T }
{ S , T } → { J }
{ T }     → { J }
```

然而，很容易看出，`FD` `{S,T}` → `{J}` 在这里是`冗余` 的——事实上，当我在本章前面首次讨论这个示例时，我实际上就假设了这一点——因此，另外两个 `FD` 共同形成一个`不可约覆盖`（我称之为 _C_）：

```
{ S , J } → { T }
{ T }     → { J }
```

现在我们可以应用 `3NF` 过程。我们从一个空的`标题` `集合` _S_ 开始。第二步做两件事：它将 _C_ 中具有相同左侧的 `FD` 聚集在一起——这在示例中实际上已经完成了——然后将`集合`（实际上是`标题`）

```
{ S , J , T }
{ T , J }
```

添加到 _S_。第三步没有效果，因为原始`关系变量` 的每个属性现在都包含在 _S_ 的至少一个元素中。最后一步也没有效果，因为 _S_ 的元素 `{S,J,T}` 是原始`关系变量` 的`超键`。总体而言，因此，`3NF` 过程告诉我们，`关系变量` `S` 可以`无损` 分解，以`保持 FD` 的方式，分解为在 `{S,J,T}` 和 `{T,J}` 上的投影。出现的问题：

- 在 `{S,J,T}` 上的投影当然与原始`关系变量` 相同！——换句话说，它是一个*`恒等投影`*（见下一节），这里实际上没有多少`分解` 在进行。
- 实际上，似乎没有太多理由同时维护第二个`关系变量`（即，在 `{T,J}` 上的投影）和原始`关系变量`，除非我们希望能够说，例如，`Professor Black` 教授 `Physics`，而同时不存在实际上正在被 `Professor Black` 教授的学生。如果我们不想要这种能力，我们可能不想维护该`关系变量`。因此，`3NF` 过程产生的分解不一定是*推荐* 的——但是，重申一遍，它是一个所有`关系变量` 都处于 `3NF` 且所有 `FD` 都得以保持的分解。

我将把它作为一个练习（练习 6.3），展示当 `3NF` 过程应用于`关系变量` `RX1`、`RX2` 和 `RX3` 时会发生什么。同时，我想用几句关于 `BCNF` 和可能的 _`BCNF 过程`_ 的话来结束本节。

首先，我们可以向 `3NF` 过程添加另一个（第五个）步骤，如下所示：

5.  令 _Z_ 为 _S_ 的一个元素，使得`关系变量` _R_ 在 _Z_ 的属性上的投影 _P_ 不处于 `BCNF`；令 _X_ → _Y_ 为 _C_ 的一个在 _P_ 中成立的 `FD`；并令 _X_ 不是 _P_ 的`超键`。用 (a) _X_ 和 _Y_ 的并集以及 (b) _Z_ 和 _Y_ 之间的差 _Z_ – _Y_（按此顺序）替换 _S_ 中的 _Z_。对每个不同的 _Z_ 和每个不同的 _X_ 执行此步骤。

现在，应用于`关系变量` `SJT` 的 `3NF` 过程产生了一个由`标题` `{T,J}` 和 `{S,J,T}` 组成的`集合` _S_。`SJT` 在 `{T,J}` 上的投影处于 `BCNF`，但 `SJT` 在 `{S,J,T}` 上的（恒等）投影不处于 `BCNF`，因为 `FD` `{T}` → `{J}` 在此后一个投影中成立，而 `{T}` 不是`超键`。因此，应用步骤 5，我们删除`标题` `{S,J,T}` 并插入 (a) `{T}` 和 `{J}` 的并集——但此插入没有效果，因为该并集已经是 _S_ 的一个元素——以及 (b) `{S,J,T}` 和 `{J}` 之间的差，按此顺序。因此，_S_ 最终包含以下`标题` 作为元素：

```
{ S , T }
{ T , J }
```

这两个`标题` 是一组 `BCNF` `关系变量` 的`标题`，`SJT` 可以`无损` 分解为这些`关系变量`（这些`关系变量` 的`键` 分别是 `{S,T}` 和 `{T}`）。因此，向 `3NF` 过程添加步骤 5 将其转换为一个 _`BCNF 过程`_，尽管不能保证 `FD` 会得以保持。（事实上，当然，不可能提供任何此类保证，因为我们已经知道 `BCNF` 和 `FD` 保持可能是相互冲突的目标。）然而，任何丢失的 `FD` 都是那些*无法* 在不违反 `BCNF` 的情况下得以保持的 `FD`。

实际上，我们可以稍微简化事情，直接使用以下过程直接转到 `BCNF`，完全绕过 `3NF`（输入与 `3NF` 过程相同——即，它由一个`关系变量` _R_ 和一个`不可约覆盖` _C_ 组成，适用于在 _R_ 中成立的 `FD`）：

1.  将 _S_ 初始化为仅包含 _R_ 的`标题`。
2.  （_与 `3NF` 过程的步骤 2 相同_。）令 _X_ 为 _C_ 中某个 `FD` 的左侧（`决定因素`）；令 _C_ 中左侧为 _X_ 的完整 `FD` `集合` 为 _X_ → _Y1_, ..., _X_ → _Yn_；并令 _Y1_, ... _Yn_ 的并集为 _Y_。将 _X_ 和 _Y_ 的并集添加到 _S_。对每个不同的 _X_ 执行此步骤。
3.  （_与上述步骤 5 相同_。）令 _Z_ 为 _S_ 的一个元素，使得 _R_ 在 _Z_ 的属性上的投影 _P_ 不处于 `BCNF`；令 _X_ → _Y_ 为 _C_ 的一个在 _P_ 中成立的 `FD`；并令 _X_ 不是 _P_ 的`超键`。用 (a) _X_ 和 _Y_ 的并集以及 (b) _Z_ 和 _Y_ 之间的差 _Z_ – _Y_（按此顺序）替换 _S_ 中的 _Z_。对每个不同的 _Z_ 和每个不同的 _X_ 执行此步骤。

在此过程结束时，_S_ 的元素是一组 `BCNF` `关系变量` 的`标题`，_R_ 可以`无损` 分解为这些`关系变量`，尽管不一定不会丢失 `FD`。注意，该过程没有提及 `2NF` 或 `3NF`。

### 恒等分解

这稍微偏离了本章的主题，但我想简要阐述一下`恒等投影` 的概念。以下是定义（我为`关系变量` 定义它，但当然，一个完全类似的定义也适用于`关系`——以及`元组`，顺便提一下）：

> **`定义`（`恒等投影`）**：给定`关系变量` 的`恒等投影` 是该`关系变量` 在其所有属性上的投影。

现在，应该很明显，任何`关系变量` 都可以`无损` 分解，尽管是平凡地，分解为其`恒等投影`。然而，有些人不喜欢将这样的分解视为一种`分解` 本身（正如我在 `SJT` 示例中所说，这里实际上没有多少`分解` 在进行）。如果您恰好是那些人之一，那么您可能更喜欢以下看待此事的方式。令`关系变量` _R_ 具有`标题` _H_。那么，`FD` `{}` → `{}` 肯定在 _R_ 中成立，其中 `{}` 是空属性`集合`（此 `FD` 当然是`平凡` 的，并且在*每个*`关系变量` 中成立）。因此，根据 `Heath 定理`——令 _X_、_Y_ 和 _Z_ 分别为 `{}`、`{}` 和 _H_——_R_ 可以`无损` 分解为其投影 _R1_ 和 _R2_，其中：

1.  _R1_ 的`标题` _XY_ 是 `{}` 和 `{}` 的并集，简化为只是 `{}`；即，_R1_ 是 _R_ 在没有任何属性上的投影，其值要么是 `TABLE_DUM`（如果 _R_ 为空），要么是 `TABLE_DEE`。<sup>14</sup>
2.  _R2_ 的`标题` _XZ_ 是 `{}` 和 _H_ 的并集，简化为只是 _H_；即，_R2_ 是 _R_ 在其所有属性上的投影，或者换句话说，是 _R_ 的`恒等投影`。

现在，我希望很清楚，此分解是`无损` 的——_R_ 肯定等于 _R1_ 和 _R2_ 的连接。（另一方面，_R1_ 和 _R2_ 的组合确实未能满足通常的要求，即两个投影都应在重建过程中需要。）

当我谈到所谓的 _`恒等分解`_ 时，<sup>15</sup> 让我指出，任何`关系变量` 也可以总是（再次平凡地，但这次是"水平地"而不是"垂直地"）分解为相应的`恒等限制`。<sup>15</sup> 以下是定义（再次，我为`关系变量` 定义它，但当然，类似的概念也适用于`关系`，尽管这次不适用于`元组`）：

> **`定义`（`恒等限制`）**：给定`关系变量` _R_ 的`恒等限制` 是 _R_ 的任何`限制`，其中`限制条件` 恒为真——换句话说，_R_ 的任何`限制`，它在逻辑上等价于以下形式之一：
>
> ```
> R WHERE TRUE
> ```

_注意_：在逻辑中，恒为真的东西，例如`布尔表达式` `CITY = CITY`，称为*`重言式`*。因此，我们可以说`关系变量` _R_ 的`恒等限制` 是任何`限制条件` 为`重言式` 的`限制`。

我顺便指出，任何给定的`关系变量` _R_ 也总是有一个*`空限制`*，<sup>16</sup> 我们可以这样表示：

```
R WHERE FALSE
```

给定`关系变量` _R_ 的`恒等限制` 和`空限制` 的（不相交的！）并集当然恒等于 _R_。_注意_：在逻辑中，恒为假的东西，例如`布尔表达式` `CITY ≠ CITY`，称为*`矛盾式`*；因此，我们可以说`关系变量` _R_ 的`空限制` 是任何`限制条件` 为`矛盾式` 的`限制`。<sup>17</sup>

### 关于冲突的更多讨论

回到本章的主题：到目前为止，我们已经看到了几个 `FD` 可能丢失的示例。在大多数这些示例中，我们可以通过小心来避免丢失 `FD`；然而，在一个案例中（`SJT` 示例），保持 `FD` 和 `BCNF` 分解的目标确实相互冲突。因此，明显的问题出现了：我们能否描述那些真正存在冲突的情况？答案是肯定的；事实上，很容易做到。

令 _R_ 为我们正在处理的`关系变量`，并令 _C_ 为在 _R_ 中成立的 `FD` 的`不可约覆盖`。构建一个 _`FD 图`_ 如下：

1.  为 _R_ 的每个属性创建一个节点。
2.  令 _X_ → _Y_ 为 _C_ 中的一个 `FD`，其中 _X_ 涉及两个或多个属性；创建一个"超节点"，仅包含 _X_ 中命名的属性的节点。（如果您在纸上执行此操作，可以画一个圆圈包围各个属性节点。）超节点被视为节点。对 _C_ 中`决定因素` 涉及两个或多个属性的每个 `FD` 重复此步骤。
3.  令 _X_ → _Y_ 为 _C_ 中的一个 `FD`。从 _X_ 的节点向 _Y_ 的节点绘制一条有向弧。对 _C_ 中的每个 `FD` 重复此步骤。
4.  当且仅当完成的图包含任何`环`（其中`环` 是从一个节点到其自身的两个或多个有向弧的序列）时，_R_ 不能`无损` 分解为 `BCNF` 投影而不丢失 `FD`。

作为一个练习，尝试将上述过程应用于本章前面讨论的各种示例。当您这样做时，您会很快理解（如果您还没有的话）这里真正发生了什么。明确说明这一点：只有当`关系变量` 涉及类似于 `SJT` 示例中存在的 `FD` 模式时，才存在真正的冲突。

### 独立投影

为了结束本章，我想回到我开篇提到的示例。提醒您一下，该示例涉及我们常用的供应商`关系变量` `S` `无损` 分解为其在 `{SNO,SNAME,CITY}` 上的投影 `SNC` 和在 `{SNO,STATUS}` 上的投影 `ST`。该分解丢失了 `FD` `{CITY}` → `{STATUS}`，导致对任一投影的更新有时需要对另一个投影进行更新，以强制执行每个城市只有一个状态的约束。相比之下，将 `S` "合理地"分解为在 `{SNO,SNAME,CITY}` 上的投影 `SNC` 和在 `{CITY,STATUS}` 上的投影 `CT` 则没有这样的问题——可以对任一投影进行更新，而无需考虑另一个。<sup>18</sup>

为了当前讨论的目的，让我将像 `SNC` 和 `ST` 这样的分解称为 _`坏`_ 分解，将像 `SNC` 和 `CT` 这样的分解称为 _`好`_ 分解。正如我们所见，`好` 分解中的投影可以独立于彼此更新；因此，它们有时被明确称为 _`独立投影`_。相比之下，`坏` 分解中的投影在同一意义上不是独立的。因此，我们可以说，为了保持 `FD`，我们希望一个分解，其中投影是独立的。并且有一个定理，由 `Jorma Rissanen` 提出，可以帮助解决这个问题。在我陈述该定理之前，让我先给出两个投影独立的精确定义：

> **`定义`（`独立投影`）**：`关系变量` _R_ 的投影 _R1_ 和 _R2_ 是`独立` 的，当且仅当在 _R_ 中成立的每个 `FD` 也在 _R1_ 和 _R2_ 的连接中成立。

现在以下是该定理：

> **`Rissanen 定理`**：令`关系变量` _R_，具有`标题` _H_，具有投影 _R1_ 和 _R2_，分别具有`标题` _H1_ 和 _H2_；此外，令 _H1_ 和 _H2_ 都是 _H_ 的真子集，令它们的并集等于 _H_，并令它们的交集非空。<sup>19</sup> 那么投影 _R1_ 和 _R2_ 是`独立` 的，当且仅当 (a) 它们的公共属性构成其中至少一个的`超键`，并且 (b) 在 _R_ 中成立的每个 `FD` 都由在至少其中一个中成立的那些 `FD` 所蕴含。

考虑将 `S` "好"分解为其投影 `SNC` 和 `CT`。这两个投影是`独立` 的，因为 (a) 公共属性`集合` 只是 `{CITY}`，并且 `{CITY}` 是 `CT` 的`超键`——实际上是一个`键`——并且 (b) 在 `S` 中成立的每个 `FD` 要么在两个投影之一中成立，要么由在其中成立的那些 `FD` 所蕴含（见下一章）。相比之下，考虑"坏"分解为投影 `SNC` 和 `ST`。这里的投影不是`独立` 的，因为 `FD` `{CITY}` → `{STATUS}` 不能从那些投影中成立的 `FD` 推断出来（尽管至少公共属性`集合` `{SNO}` 是两者的`超键`——实际上是一个`键`——这是真的）。

作为一个历史注释，我指出，正是 `Rissanen` 关于`独立投影` 的工作（这项工作是在 1977 年完成或至少发表的）为我们现在称为 `FD` 保持的理论奠定了基础。

### 练习

6.1 来自"一个不幸的冲突"一节的`关系变量` `SJT` 受制于 `FD` `{S,J}` → `{T}`。编写一个 **`Tutorial D`** `CONSTRAINT` 语句来表达如果我们基于 `{T,J}` 和 `{T,S}` 分别将 `SJT` 分解为其投影 `TJ` 和 `TS` 时取代此 `FD` 的`多关系变量约束`。

6.2 （_重复自本章正文_。）假设来自"... 还有另一个"一节的`关系变量` `RX2A′` 被分解为在 `{CLASS,STATUS}` 和 `{CITY,STATUS}` 上的投影。正如该节所指出的，现在必须单独表述和强制执行一个适当的`多关系变量约束`。该约束是什么样子？

6.3 以下`关系变量` 旨在表示一组美国街道地址：

```
ADDR { STREET , CITY , STATE , ZIP }
```

使用 **`Tutorial D`** 语法，该`关系变量` 的典型`元组` 可能看起来像这样：

```
TUPLE { STREET '1600 Pennsylvania Ave.' ,
                CITY 'Washington' , STATE 'DC' , ZIP '20500' }
```

假设，并非完全不现实，以下 `FD` 在该`关系变量` 中成立并且是`不可约` 的：

```
{ STREET , CITY , STATE } → { ZIP }
{ ZIP }                   → { CITY , STATE }
```

您将如何分解此`关系变量`？

6.4 展示将 `3NF` 过程应用于本章正文中的`关系变量` `RX1`、`RX3` 和 `RX2`（注意此处的顺序！）的效果。

6.5 以下是一个谓词：

> _电影明星 `S` 在电影 `M` 中扮演角色 `R`，该电影由导演 `D` 执导并于年份 `Y` 发行；此外，明星 `S` 出生于日期 `B`，因此具有星座 `Z` 和生肖 `C`，并且 `Z` 和 `C` 共同决定 `S` 的星运 `H`。_

给出捕获上述情况的一组 `FD`。陈述您关于可能生效的`业务规则` 所做的任何假设。此外，应用 `BCNF` 过程以获得一组适当的 `BCNF` `关系变量`。该过程会丢失任何 `FD` 吗？

### 答案

6.1

```
CONSTRAINT ... WITH ( SJT := JOIN { TJ , TS } ) :
                      COUNT ( SJT ) =
                      COUNT ( SJT { S , J } ) ;
```

或者，使用第 4 章练习 4.8 答案中描述的约束替代样式：

```
CONSTRAINT ... JOIN { TJ , TS } KEY { S , J } ;
```

6.2 令 `LT` 和 `CT` 分别为 `RX2A′` 在 `{CLASS,STATUS}` 和 `{CITY,STATUS}` 上的投影。那么 (a) `{CLASS}` 和 `{CITY}` 将是 `RX2B′` 中的`外键`，分别引用 `LT` 和 `CT`，并且 (b) 以下`多关系变量约束` 也将成立：

```
CONSTRAINT ... WITH ( LTX := LT RENAME { STATUS AS X } ,
                      CTY := CT RENAME { STATUS AS Y } ) :
               AND ( JOIN { RX2B′ , LTX , CTY } , X = Y ) ;
```

6.3 给定的第一个 `FD` 意味着 `{STREET,CITY,STATE}` 是一个`键`；第二个意味着该`关系变量` 不处于 `BCNF`。然而，如果我们使用 `Heath 定理` 将其分解（基于`函数依赖` `{ZIP}` → `{CITY,STATE}`）为 `BCNF` 投影，如下：

```
ZCT { ZIP , CITY , STATE }
    KEY { ZIP }
ZR  { ZIP , STREET }
    KEY { ZIP , STREET }
```

——那么我们丢失了 `FD` `{STREET,CITY,STATE}` → `{ZIP}`。因此，`关系变量` `ZCT` 和 `ZR` 不能独立更新。（_子练习_：开发 `ZCT` 和 `ZR` 的一些示例值来说明这一点。）当然，如果我们不执行此分解，将会有一些`冗余`；具体来说，给定邮政编码对应特定城市和州这一事实将出现多次。但这种`冗余` 会导致问题吗？鉴于给定城市和州的邮政编码不经常变化，答案是"可能，但不经常"。（另一方面，邮政编码*从不* 变化是不正确的。）

6.4 以下是 `RX1` 的一个`不可约覆盖`：

```
{ SNO , PNO } → { QTY }
{ SNO }       → { CITY }
{ CITY }      → { STATUS }
```

`3NF` 过程产生 `{SNO,PNO,QTY}`、`{SNO,CITY}` 和 `{CITY,STATUS}`。

接下来，`RX3`。一个`不可约覆盖`：

```
{ SNO }   → { CLASS }
{ CLASS } → { CITY }
{ CITY }  → { STATUS }
```

`3NF` 过程产生 `{SNO,CLASS}`、`{CLASS,CITY}` 和 `{CITY,STATUS}`。

最后 `RX2`。`不可约覆盖`：

```
{ SNO }   → { CLASS }
{ SNO }   → { CITY }
{ CLASS } → { STATUS }
{ CITY }  → { STATUS }
```

`3NF` 过程产生 `{SNO,CLASS,CITY}`、`{CLASS,STATUS}` 和 `{CITY,STATUS}`。此示例有趣之处在于，正如本章正文所示，如果我们基于 `FD` `{SNO}` → `{CLASS,CITY}` 进行分解，我们得到 `{SNO,CLASS,CITY}` 和 `{CLASS,CITY,STATUS}` 作为 `3NF` 投影`标题`，而这*不是* 我们从 `3NF` 过程得到的结果。事实上，`3NF` 过程的结果需要维护以下相当复杂的`多关系变量约束`（一个`等式依赖`）：

```
CONSTRAINT ... JOIN { SLC , LT } = JOIN { SLC , CT } ;
```

（"对于给定供应商，类别状态 = 城市状态"；此处的 `SLC`、`LT` 和 `CT` 分别表示 `RX2` 在 `{SNO,CLASS,CITY}`、`{CLASS,STATUS}` 和 `{CITY,STATUS}` 上的投影）。因此，该示例说明了这一点：尽管 `3NF` 过程肯定保证产生 `3NF` 投影并且不会丢失任何 `FD`，但可能不应该过于盲目地遵循它。

_注意_：假设我们将`关系变量` `LT` 和 `CT` 中的状态属性命名为不同的名称，如下：

```
LT { CLASS , CLASS_STATUS }
CT { CITY , CITY_STATUS }
```

那么对于任何给定供应商，两个状态值必须相等的约束可以这样表述：

```
CONSTRAINT ... IS_EMPTY ( ( JOIN { SLC , LT , CT } )
                          WHERE CLASS_STATUS ≠ CITY_STATUS ) ;
```

（**`Tutorial D`** 表达式 `IS_EMPTY` (_rx_) 如果`关系表达式` _rx_ 表示的`关系` _r_ 为空则返回 `TRUE`，否则返回 `FALSE`。）或者：

```
CONSTRAINT ...
   AND ( JOIN { SLC , LT , CT } , CLASS_STATUS = CITY_STATUS ) ;
```

此示例的整体信息可以这样表述：整个关于丢失或保持 `FD` 的事情实际上只是一个更普遍现象的特例：即，如果我们从某个设计 _D1_ 开始并将其映射到某个逻辑上等价的 _D2_，那么该过程通常将涉及对约束以及`关系变量` 的某种重组，这是必然的（事实上应该很明显）。

6.5 _假设_：没有明星在任何给定电影中扮演超过一个角色；没有电影有超过一位导演。（这些假设合理吗？）`FD`：

```
{ S , M } → { R }
{ M }     → { D , Y }
{ S }     → { B }
{ B }     → { Z , C }
{ Z , C } → { H }
```

如果所有九个属性组合在一个`关系变量` 中，那么该`关系变量` 仅处于 `1NF`（`{S,M}` 是一个`键`）。`BCNF` 过程产生 `{S,M,R}`、`{M,D,Y}`、`{S,B}`、`{B,Z,C}` 和 `{Z,C,H}`。没有 `FD` 丢失。
