## 第 4 章 `FDs` 与 `BCNF`（非正式讲解）

> _在教授抽象概念之前先教授具体内容是极其错误的。_
>
> — Z. A. `Melzak`：《`Companion to Concrete Mathematics`》（1973）

如前一章所述，`Boyce/Codd normal form`（简称 `BCNF`）是根据 `functional dependencies`（`FDs`）定义的；事实上，它确实是关于 `functional dependencies` 的`范式`，正如——稍作前瞻——`5NF` 确实是关于 `join dependencies`（`JDs`）的`范式`。本章的总体目的是详细解释 `BCNF` 和 `FDs`；不过，正如本章标题所示，此处的各种解释及相关定义在此阶段都特意保持一定的非正式性。（非正式，但不失准确；我不会故意误导。）下一章将对这些内容进行更形式化的处理。

### 第一范式

设`关系` _r_ 具有属性 _A1_, ..., _An_，其类型分别为 _T1_, ..., _Tn_。那么，根据`定义`，如果`元组` _t_ 出现在`关系` _r_ 中，则属性 _Ai_ 在 _t_ 中的值必须是类型 _Ti_ 的值（_i_ = 1, ..., _n_）。例如，如果 _r_ 是`关系变量` `SP` 的当前值（参见第 1 章图 1-1），那么 _r_ 中的每个`元组`都具有类型为 `CHAR` 的 `SNO` 值、类型同样为 `CHAR` 的 `PNO` 值，以及类型为 `INTEGER` 的 `QTY` 值。

现在，换一种方式表述上一段前两句的内容，简单来说就是`关系` _r_ 处于`第一范式`（`1NF`）。因此，_每个_`关系`都处于`1NF`！——因为如果某个"关系" _r_ 不满足上述两个条件，那么它根本就不是`关系`。因此，请原谅我重复说明，以下是正式的`定义`：<sup>1</sup>

> **`定义`（`第一范式`）**：设`关系` _r_ 具有属性 _A1_, ..., _An_，其类型分别为 _T1_, ..., _Tn_。那么，_r_ 处于`第一范式`（`1NF`）当且仅当，对于出现在 _r_ 中的所有`元组` _t_，属性 _Ai_ 在 _t_ 中的值是类型 _Ti_ 的值（_i_ = 1, ..., _n_）。

换句话说，`1NF` 仅意味着所讨论`关系`中的==每个`元组`为每个属性恰好包含一个适当类型的值。==_特别需要注意的是，`1NF` 对允许的属性类型没有任何限制_。<sup>2</sup> 这些类型甚至可以是`关系`类型！也就是说，具有`关系值属性`（`relation valued attributes`，简称 `RVAs`）的`关系`是合法的（您可能会对此感到惊讶，但这确实是真的）。图 4-1 下方给出了一个示例。

关于 `RVAs`，我稍后会进一步说明，但首先我需要澄清几个小问题。首先，我需要定义`关系`处于`规范化`（`normalized`）状态的含义：

> **`定义`（`规范化`）**：`关系` _r_ 是`规范化`的，当且仅当它处于`1NF`。

换句话说，`规范化`和`第一范式`的含义完全相同——所有`规范化`的`关系`都处于`1NF`，所有处于`1NF` 的`关系`都是`规范化`的。这种略显奇怪的情况的原因是，`规范化`是原始的（历史性）术语；直到人们开始讨论 `2NF` 及更高`范式`时，才引入`1NF` 这个术语，以便描述那些不属于更高`范式` 的`关系`。<sup>3</sup> 当然，如今`规范化`一词常被用来表示某个更高`范式`（通常是 `3NF`，或者可能是 `BCNF`）；事实上，我自己也曾这样使用过，尽管我通常尽量避免——因为严格来说，这种用法是含糊且不正确的，除非不存在混淆的可能，否则最好避免使用。

关于我的第二个"小问题"：请注意，==本节迄今为止的所有讨论（特别是`定义`）都是针对`关系`而非`关系变量`进行的。==但是，由于任何可以赋值给`关系变量` 的`关系`根据`定义`都处于`1NF`，因此将`1NF` 概念以显而易见的方式扩展到也适用于`关系变量`并不会造成任何问题——而且这样做是可取的，因为（正如我们将看到的）所有其他`范式` 都被`定义`为适用于`关系变量`，而非`关系`。事实上，可以认为`1NF` 之所以根据`关系`而非`关系变量` 来`定义`，原因在于区分`关系`和`关系变量` 这一概念（我是说`关系`与`关系变量` 之间的区别）被正确确立之前，确实已经过去了遗憾的许多年。

回到 `RVAs`。我实际上已经说过，具有 `RVAs` 的`关系变量` 是合法的——但现在我需要补充的是，至少从设计角度来看，这类`关系变量` 通常是（并非总是）不推荐的。现在，这一事实并不意味着您应该完全避免使用 `RVAs`（特别是，如果某个查询结果的某个属性碰巧是`关系值`，那没有问题）——它只是意味着我们通常不希望 `RVAs` "被设计到数据库中"，可以这么说。我不想在这本书中深入讨论这个问题的细节；让我只是说，具有 `RVAs` 的`关系变量` 往往非常类似于在 `IMS` 等较旧的非`关系` 系统中发现的`层次结构`，<sup>4</sup> 因此所有曾经与层次结构相关的老问题都会再次浮现。以下是其中一些问题供参考：

- 根本点在于层次结构是不对称的。因此，虽然它们可能使某些任务更容易，但肯定会使其他任务更加困难。
- 作为前一点的具体说明，查询本身也是不对称的，而且比它们的对称对应物更复杂。例如，考虑针对图 4-1 中的`关系` 表述查询"获取由供应商 `S2` 供应的零件编号"和"获取供应零件 `P2` 的供应商编号"所涉及的内容。这些查询的自然语言版本彼此是对称的，但它们在 `SQL`<sup>5</sup> 或 **`Tutorial D`** 或其他任何形式语言中的表述形式肯定不是对称的（参见练习 4.14）。
- 类似的评论也适用于安全性和完整性约束。
- 类似的评论也适用于更新操作，可能力度更大。
- 通常没有指导说明如何选择"最佳"层次结构。例如，在供应商和零件的情况下，我们应该使零件从属于供应商（这实际上是图 4-1 所示设计所做的），还是使供应商从属于零件？
- 即使是像组织图和物料清单结构这样的"自然"层次结构，通常也最好用非层次化的设计来表示。

> ===这个主要指的是表中套表，这就是RVAs。这样会退化成层次模型===

### 违反第一范式

到目前为止，您可能会想知道，如果所有`关系变量` 根据`定义` 都处于`1NF`，那么某事物*不*处于`1NF` 可能意味着什么。也许令人惊讶的是，这个问题确实有一个合理的答案。关键在于，当今的商业 `DBMS` 根本没有正确支持`关系变量`（或`关系`）——相反，它们支持一种为了方便我称之为`表`（`table`）的结构，尽管用这个术语我并不一定想将自己局限于 `SQL` 系统中特定类型的表。<sup>6</sup> 而`表`，与`关系变量` 不同，确实可能不处于`1NF`。详细说明如下：

> **`定义`（`规范化表`）**：一个`表`处于`第一范式`（`1NF`）——等价地，这样的`表`是`规范化`的——当且仅当它是某个`关系变量` 的直接且忠实的表示。

那么，当然问题在于：`表`是`关系变量` 的直接且忠实表示意味着什么？这个问题的答案涉及五个基本要求，所有这些要求都是以下事实的直接结果：`关系变量` 在任何给定时间的值（当然）始终是一个`关系`：

1.  `表`从不包含任何重复行。
2.  列没有从左到右的顺序。
3.  行没有从上到下的顺序。
4.  所有列都是`常规列`。
5.  每个行和列的交点始终恰好包含一个适用类型的值，且仅此而已。

要求 1-3 是不言自明的，<sup>7</sup> 但其他两个可能需要稍微多一点的解释。首先，考虑要求 4（"所有列都是`常规列`"）。为了满足此要求，所讨论的`表` 必须满足以下两个条件：

    a.  每列都有一个适当的名称（即，在 `SQL` 术语中，可以作为 `CREATE TABLE` 语句中的列名指定的名称），并且该名称在适用于所讨论`表` 的====列名中是唯一的。====
    b.  不允许任何行包含超出上述列值的任何额外内容。因此，没有只能通过特殊运算符而非通过`常规列` 引用访问的"隐藏"列（其中"`常规列` 引用"基本上就是列名），也没有导致对行的常规运算符调用产生不规则效果的列。特别是，因此，除了常规`关系` 键值之外，没有其他`标识符`（没有隐藏的"行 ID"或"对象 ID"，遗憾的是，这在某些 `SQL` 产品中今天仍然存在），也没有在某些文献中的"`时态数据库`"提案中发现的隐藏`时间戳`。==所有列要在Types中==

至于要求 5：首先请注意，==此要求意味着`空值`==（`nulls`）是被禁止的（因为`空值`，无论它们可能是什么，肯定不是值）。然而，更一般地说，该要求旨在解决`数据值原子性`（`data value atomicity`）的问题。这是`关系` 模型中关于`关系` 的"人人都知道"的一件事——即，此类`关系` 内的属性值应该是`原子` 的（对吧？）。那么，在此上下文中，`原子` 究竟是什么意思呢？嗯，在他著名的 1970 年论文中，<sup>8</sup> Codd 仅仅说它意味着"不可分解"。而在后来的著作中，他继续说`不可分解` 反过来意味着"由 `DBMS` 不可分解"，我理解这意味着用户无法要求 `DBMS` 代表他或她执行此类分解，无论是显式还是隐式。好的，那么让我们考虑几个示例：

- _`字符字符串`_：字符字符串在上述意义上是不可分解的吗？显然不是——例如，想想 `SQL` 的 `SUBSTRING`、`LIKE` 和连接运算符，所有这些显然都依赖于字符字符串通常具有某种内部结构，因此可以分解成更小的部分。然而，肯定没有人会认为字符字符串不应该被允许出现在`关系` 中。
- _`定点数`_：可以分解为整数部分和小数部分。
- _`整数`_：可以分解为其质因数。（当然，我意识到这不是我们在此上下文中通常考虑的分解类型；我只是想表明可分解性的概念本身可以有多种解释。）
- _`日期和时间`_：可以分别分解为年/月/日和时/分/秒组件。
- _`关系表达式`_：例如，考虑`目录`（`catalog`）中的视图定义。此类表达式肯定是"可分解的"——事实上，由 `DBMS` 可分解——因为如果它们不是，那么首先将它们保存在`目录` 中就没有意义了。

所有这些示例以及许多类似示例的底线是，==如果`关系` _r_ 有一个属性 _A_，那么 _A_ 在 _r_ 内的值可以是*任何内容*，只要它们是为该属性 _A_ 定义的类型 _T_ 的值即可。==而该类型 _T_ 反过来可以是任何类型！<sup>9</sup> 它甚至可以是`关系` 类型（因此就有了前面几页讨论过的`关系值属性` 的可能性）。

`1NF` 的"`原子性`"要求有时以"无重复组"的形式陈述。事实上，我在许多早期著作中——特别是在本书前一版中——也曾以这种形式陈述过，在那里我尝试（但我认为失败了）给出`重复组` 可能是什么的精确定义。经过进一步反思，我得出的结论是，最好在此上下文中根本不要考虑`重复组`，而是转而关注（然后，正如我刚才所做的那样，去驳斥！）`原子性` 的概念。换句话说，我现在认为反对`重复组` 的禁令本质上一直是毫无意义的——我特此向任何可能被我早期在这方面努力所误导的人道歉。

总之，如果五个要求中的任何一个被违反，所讨论的`表` 就不会"直接且忠实"地表示一个`关系变量`，_那么所有保证都将失效_。特别是，`关系` 运算符（如`连接`）不再保证按预期工作（如果您——正如我假设的那样——熟悉 `SQL`，您已经知道这一点）。==`关系` `模型` 处理的是`关系` 值和变量，而且*仅*处理`关系` 值和变量。==

### `函数依赖`

关于`1NF` 就说到这里；现在我可以继续并开始讨论一些更高`范式`。现在，我已经说过 `Boyce/Codd normal form`（`BCNF`）是根据`函数依赖`（`FDs`）定义的，当然`第二范式`（`2NF`）和`第三范式`（`3NF`）也是如此。所以这里是一个`定义`：

> **`定义`（`函数依赖`）**：设 _X_ 和 _Y_ 是`关系变量` _R_ 的`head` 的子集；那么`函数依赖`（`FD`）
>
> ```
> X → Y
> ```
>
> 在 _R_ 中成立，当且仅当，每当 _R_ 的两个`元组` 在 _X_ 上一致时，它们在 _Y_ 上也一致。<sup>10</sup> 这里的 _X_ 和 _Y_ 分别称为`决定因素`（`determinant`）和`依赖项`（`dependant`），整个 `FD` 可以读作"_X_ `函数决定` _Y_"，或"_Y_ `函数依赖于` _X_"，或更简单地读作"_X_ 箭头 _Y_"。

这里有几个示例：

- `FD` `{CITY} → {STATUS}` 在`关系变量` `S` 中成立，正如我们从第 2 章所知道的。顺便注意一下花括号；`定义` 中的 _X_ 和 _Y_ 是 _R_ 的`标题` 的子集，因此是（属性的）集合，即使像示例中那样，它们碰巧是单元素集合。同样，_X_ 和 _Y_ 的值是`元组`，即使像示例中那样，它们碰巧是度为 1 的`元组`。
- `FD` `{SNO} → {SNAME,STATUS}` 也在`关系变量` `S` 中成立，因为 `{SNO}` 是该`关系变量` 的`键`——事实上，是唯一的`键`——并且*总是*存在"从`键` 出发的箭头"（参见紧随其后的"`键`"一节）。_注意_：如果还不明显，我使用短语"从 _X_ 出发的箭头"来表示存在某个 _Y_，使得 `FD` _X_ → _Y_ 在相关的`关系变量` 中成立（其中 _X_ 和 _Y_ 是该`关系变量` `标题` 的子集）。

现在，这里有一件有用的事情需要记住：==如果 `FD` _X_ → _Y_ 在`关系变量` _R_ 中成立，那么对于 _X_ 的所有超集 _X''_ 和 _Y_ 的所有子集 _Y'_，`FD` _X''_ → _Y'_ 也在`关系变量` _R_ 中成立==（只要 _X''_ 仍然是`标题` 的子集，当然）。换句话说，==我们总是可以向`决定因素` 添加属性或从`依赖项` 中减去属性，而我们得到的仍然是在所讨论`关系变量` 中成立的 `FD`。==例如，这里是另一个在`关系变量` `S` 中成立的 `FD`：

```
{ SNO , CITY } → { STATUS }
```

（我从 `FD` `{SNO} → {SNAME,STATUS}` 开始，向`决定因素` 添加了 `CITY`，并从`依赖项` 中减去了 `SNAME`。）

我还需要解释 `FD` 是`平凡`（`trivial`）的含义：

> **`定义`（`平凡 FD`）**：`FD` _X_ → _Y_ 是`平凡`的，当且仅当它不可能被违反。

例如，以下 `FD` 在任何具有名为 `STATUS` 和 `CITY` 属性的`关系变量` 中都`平凡` 成立：<sup>11</sup>

```
{ CITY , STATUS } → { CITY }
{ CITY , STATUS } → { STATUS }
{ CITY }          → { CITY }
{ CITY }          → { }
```

简要阐述一下（但为了简单起见，仅考虑这些示例中的第一个）：如果两个`元组` 具有相同的 `CITY` 和 `STATUS` 值，那么它们肯定具有相同的 `CITY` 值。事实上，很容易看出 `FD` _X_ → _Y_ 是`平凡`的，当且仅当 _Y_ 是 _X_ 的子集（用符号表示，_Y_ ⊆ _X_）。现在，当我们进行`数据库设计` 时，我们通常不关心`平凡` `FD`，因为它们，嗯，是`平凡`的；但是当我们试图在这些问题上做到形式化和精确时——特别是，当我们试图发展设计的`理论` 时——那么我们需要考虑所有 `FD`，包括`平凡` 的和非`平凡` 的。

> 平凡和非平凡：他们是否是子集关系

### `键` 再探

我在第 1 章中一般性地讨论了`键` 的概念，但现在是时候更精确地讨论这个问题并引入更多术语了。首先，以下是术语`候选键`（`candidate key`）的正式`定义`——正如第 1 章所述，我在本书大部分地方将其缩写为`键`（`key`）：

> **`定义`（`候选键`，`键`）**：设 _K_ 是`关系变量` _R_ 的`标题` 的子集。那么 _K_ 是 _R_ 的`候选键`（或简称`键`），当且仅当它具有以下两个属性：
>
> 1.  _`唯一性`_：_R_ 的任何合法值都不包含两个在 _K_ 上具有相同值的不同的`元组`。
> 2.  _`不可约性`_：_K_ 的任何真子集都不具有`唯一性` 属性。

==表中任意两行在该属性集上的值不能完全相同==

==该集合删除任一列都会被破坏唯一性==

这是我们遇到的第一个涉及某种`不可约性` 的`定义`，但我们在接下来的篇幅中将遇到更多——某种或另一种的`不可约性` 在整个`设计理论` 领域中是普遍存在且重要的，正如我们将看到的。关于`键` 的`不可约性`，其中一个原因（不是唯一的原因）是，如果我们指定一个不是`不可约` 的"`键`"，那么 `DBMS` 将无法强制执行适当的`唯一性约束`。例如，假设我们告诉 `DBMS`（撒谎！）`{SNO,CITY}` 是`关系变量` `S` 的`键`，并且事实上是唯一的`键`。那么 `DBMS` 就无法强制执行供应商编号是"全局"唯一的约束；相反，它只能强制执行较弱的约束，即供应商编号是"局部"唯一的，意思是它们在相关城市内是唯一的。

我不会在这里进一步讨论上述`定义`，因为这个概念是如此熟悉<sup>12</sup>——但请注意接下来的几个`定义` 如何依赖于它：

> **`定义`（`键属性`）**：`关系变量` _R_ 的属性 _A_ 是 _R_ 的`键属性`，当且仅当它是 _R_ 的至少一个`键` 的一部分。
>
> **`定义`（`非键属性`）**：`关系变量` _R_ 的属性 _A_ 是 _R_ 的`非键属性`，当且仅当它不是 _R_ 的任何`键` 的一部分。<sup>13</sup>
>
> 例如，在`关系变量` `SP` 中，`SNO` 和 `PNO` 是`键属性`，而 `QTY` 是`非键属性`。
>
> **`定义`（"`全键`" `关系变量`）**：一个`关系变量` 是"`全键`"的，当且仅当整个`标题` 是一个`键`（在这种情况下，它必然是唯一的`键`）——等价地，当且仅当整个`标题` 的任何真子集都不是`键`。
>
> _注意_：如果一个`关系变量` 是"`全键`"的，那么它肯定没有`非键属性`，但反之则不然——一个`关系变量` 可以是其所有属性都是`键属性`，但又不一定是"`全键`"的（对吧？）。

> **`定义`（`超键`）**：设 _SK_ 是`关系变量` _R_ 的`标题` 的子集。那么 _SK_ 是 _R_ 的`超键`，当且仅当它具有以下属性：
>
> 1.  _`唯一性`_：_R_ 的任何合法值都不包含两个在 _SK_ 上具有相同值的不同的`元组`。
>
> 更简洁地说，_R_ 的`超键` 是 _R_ 的`标题` 的子集，它是`唯一` 的但不一定是`不可约` 的。换句话说，我们可以松散地说，`超键` 是`键` 的超集（"松散"，因为所讨论的超集当然仍然必须是相关`标题` 的子集）。因此，请注意，所有`键` 都是`超键`，但"大多数"`超键` 不是`键`。_注意_：不是`键` 的`超键` 有时被称为`真超键`（`proper superkey`）。
>
> 定义`子键`（`subkey`）的概念也很方便：
>
> **`定义`（`子键`）**：设 _SK_ 是`关系变量` _R_ 的`标题` 的子集。那么 _SK_ 是 _R_ 的`子键`，当且仅当它是 _R_ 的至少一个`键` 的子集。
>
> _注意_：不是`键` 的`子键` 有时被称为`真子键`（`proper subkey`）。

举例来说，考虑`关系变量` `SP`，它只有一个`键`，即 `{SNO,PNO}`。该`关系变量` 具有：

    a.  两个`超键`：
        ```
        { SNO , PNO }
        { SNO , PNO , QTY }
        ```

请注意，`标题` 始终是任何`关系变量` _R_ 的`超键`。

    b.  四个`子键`：
        ```
        { SNO , PNO }
        { SNO }
        { PNO }
        { }
        ```

请注意，空属性集始终是任何`关系变量` _R_ 的`子键`。

为了结束本节，请注意，如果 _H_ 和 _SK_ 分别是`关系变量` _R_ 的`标题` 和`超键`，那么 `FD` _SK_ → _H_ 必然在 _R_ 中成立。（等价地，对于 _H_ 的所有子集 _Y_，`FD` _SK_ → _Y_ 在 _R_ 中成立。）原因是，如果 _R_ 的两个`元组` 在 _SK_ 上具有相同的值，那么它们实际上必须是完全相同的`元组`，在这种情况下，它们显然必须在 _Y_ 上具有相同的值。当然，所有这些评论都适用于 _SK_ 不仅是`超键` 而且是`键` 的重要特殊情况；正如我早些时候所说（当然非常松散），总是存在从`键` 出发的箭头。事实上，我们现在可以做一个更一般的陈述：总是存在从`超键` 出发的箭头。

### 第二范式

在我可以继续讨论 `2NF`、`3NF` 和 `BCNF` 的`定义` 之前，我需要引入一个更多概念，即 `FD` `不可约性`（注意，这是另一种`不可约性`）：

> **`定义`（`不可约 FD`）**：`FD` _X_ → _Y_ 关于`关系变量` _R_ 是`不可约` 的（或者如果 _R_ 是已知的，则简称为`不可约`），当且仅当它在 _R_ 中成立，并且对于 _X_ 的任何真子集 _X'_，`FD` _X'_ → _Y_ 不在 _R_ 中成立。==X唯一不可变==

例如，`FD` `{SNO,PNO} → {QTY}` 关于`关系变量` `SP` 是`不可约` 的。_注意_：这种`不可约性` 有时更明确地称为`左` 不可约性（因为我们真正讨论的是 `FD` 的左侧），但我选择在这里省略"左"以简化表述。

现在——您可能已经原谅我这么想了——我终于可以`定义` `2NF` 了：

> **`定义`（`第二范式`）**：`关系变量` _R_ 处于`第二范式`（`2NF`），当且仅当，对于 _R_ 的每个`键` _K_ 和 _R_ 的每个`非键属性` _A_，`FD` _K_ → {_A_}（必然在 _R_ 中成立）是`不可约` 的。

_注意_：以下（"首选"）`定义` 与刚刚给出的`定义` 在逻辑上是等价的——参见本章末尾的练习 4.4——但有时可能更有用：

> **`定义`（`第二范式`，首选）**：`关系变量` _R_ 处于`第二范式`（`2NF`），当且仅当，对于在 _R_ 中成立的每个`非平凡` `FD` _X_ → _Y_，以下至少一项为真：
>
>     a.  *X* 是`超键`。
>     b.  *Y* 是`子键`。
>     c.  *X* 不是`子键`。

出现的问题：

- 首先，请理解，将`2NF` 视为设计过程的最终目标是非常不寻常的。事实上，`2NF` 和 `3NF` 主要具有历史意义；它们都被视为通往 `BCNF` 的垫脚石，而 `BCNF` 具有更大的实际（以及理论）意义。
- 文献中关于`2NF` 的`定义` 通常采用"_R_ 处于`2NF` 当且仅当它处于`1NF` 且……"的形式。然而，此类`定义` 通常基于对`1NF` 是什么的误解。正如我们所见，_所有_`关系变量` 都处于`1NF`，因此"它处于`1NF` 且"这些词没有添加任何内容。

让我们看一个示例。实际上，对于`范式` 来说，看一个反例通常比看一个正面示例更有启发性。因此，考虑`关系变量` `SP` 的一个修订版本——我们称之为 `SCP`——它具有一个额外的属性 `CITY`，表示适用供应商所在的城市。以下是一些示例`元组`：

这个`关系变量` 显然存在`冗余`：每个供应商 `S1` 的`元组` 都告诉我们 `S1` 在伦敦，每个供应商 `S2` 的`元组` 都告诉我们 `S2` 在巴黎，依此类推。并且（援引上述`2NF` 的第一个`定义`），该`关系变量` 不处于`第二范式`——它的唯一`键` 是 `{SNO,PNO}`，因此 `FD` `{SNO,PNO} → {CITY}` 肯定成立，但该 `FD` 不是`不可约` 的；具体来说，我们可以从`决定因素` 中删除 `PNO`，剩下的 `{SNO} → {CITY}` 仍然是在该`关系变量` 中成立的 `FD`。等价地，我们可以说 `FD` `{SNO} → {CITY}` 成立且是`非平凡` 的；此外，(a) `{SNO}` 不是`超键`，(b) `{CITY}` 不是`子键`，并且 (c) `{SNO}` _是_`子键`，因此再次（援引现在`2NF` 的第二个"首选"`定义`）该`关系变量` 不处于`第二范式`。

### 第三范式

这次我将*仅仅* 从我的首选`定义` 开始：

> **`定义`（`第三范式`，首选）**：`关系变量` _R_ 处于`第三范式`（`3NF`），当且仅当，对于在 _R_ 中成立的每个`非平凡` `FD` _X_ → _Y_，以下至少一项为真：
>
>     a.  *X* 是`超键`。
>     b.  *Y* 是`子键`。

出现的问题：

- 重复我在上一节中说过的话（可能与流行观点相反），`3NF` 主要具有历史意义——它最多应被视为通往 `BCNF` 的垫脚石。_注意_：我之所以说*与流行观点相反*，是因为许多常见（至少在流行文献中）的`3NF` "`定义`"实际上是 `BCNF` 的`定义`——而 `BCNF`，正如我已经指出的，_是_ 重要的。_读者需谨慎_。
- 文献中关于`3NF` 的`定义` 通常采用"_R_ 处于`3NF` 当且仅当它处于`2NF` 且……"的形式。我更喜欢一个不提及`2NF` 的`定义`。然而，请注意，我的`3NF` `定义` 实际上可以通过删除条件 (c)（"_X_ 不是`子键`"）从我的`2NF` 首选`定义` 推导出来。因此，`3NF` 蕴含`2NF`——也就是说，如果一个`关系变量` 处于`3NF`，那么它肯定处于`2NF`。

我们已经看到了一个处于`2NF` 但不处于`3NF` 的`关系变量` 的示例：即，供应商`关系变量` `S`（参见第 3 章图 3-1）。详细说明：`非平凡` `FD` `{CITY} → {STATUS}` 在该`关系变量` 中成立，正如我们所知；此外，`{CITY}` 不是`超键`，`{STATUS}` 不是`子键`，因此该`关系变量` 不处于`3NF`。（它肯定处于`2NF`。_练习_：验证这个说法！）

### `Boyce/Codd 范式`

正如我早些时候所说，`Boyce/Codd normal form`（`BCNF`）是关于 `FD` 的*那个*`范式`——但现在我可以精确定义它了：

> **`定义`（`Boyce/Codd 范式`）**：`关系变量` _R_ 处于 `Boyce/Codd normal form`（`BCNF`），当且仅当，对于在 _R_ 中成立的每个`非平凡` `FD` _X_ → _Y_，以下成立：
>
>     a.  *X* 是`超键`。

出现的问题：

- 从`定义` 可以得出，在 `BCNF` `关系变量` 中成立的唯一 `FD` 要么是`平凡` 的（我们显然无法摆脱它们），要么是从`超键` 出发的箭头（我们也无法摆脱它们）。或者正如有些人喜欢说的那样：_每个事实都是关于键、整个键且仅关于键的事实_——尽管我必须立即补充说，这种非正式的表征，尽管直观上吸引人，但并不是真正准确的，因为它假设（除其他事项外）只有一个`键`。
- 该`定义` 没有引用`2NF` 或`3NF`。然而，请注意，该`定义` 可以通过删除`3NF` `定义` 中的条件 (b)（"_Y_ 是`子键`"）推导出来。因此，`BCNF` 蕴含`3NF`——也就是说，如果一个`关系变量` 处于`BCNF`，那么它肯定处于`3NF`。

作为一个处于`3NF` 但不处于`BCNF` 的`关系变量` 的示例，考虑供应`关系变量` 的一个修订版本——我们称之为 `SNP`——它具有一个额外的属性 `SNAME`，表示相关供应商的名称。还假设供应商名称必然是唯一的（即，没有两个供应商在任何时候具有相同的名称）。以下是一些示例`元组`：

（此处为图示：显示了包含 `SNO`、`SNAME`、`PNO` 和 `QTY` 属性的`关系` 样本数据）

我们再次观察到一些`冗余`：每个供应商 `S1` 的`元组` 都告诉我们 `S1` 名为 Smith，每个供应商 `S2` 的`元组` 都告诉我们 `S2` 名为 Jones，依此类推；同样，每个名为 Smith 的`元组` 都告诉我们 Smith 的供应商编号是 `S1`，每个名为 Jones 的`元组` 都告诉我们 Jones 的供应商编号是 `S2`，依此类推。并且该`关系变量` 不处于`BCNF`。首先，它有两个`键`，`{SNO,PNO}` 和 `{SNAME,PNO}`。<sup>14</sup> 其次，`标题` 的每个子集——特别是子集 `{QTY}`——（当然）在函数上都依赖于这两个`键`。第三，然而，`FD` `{SNO} → {SNAME}` 和 `{SNAME} → {SNO}` 也成立；这些 `FD` 肯定不是`平凡` 的，也不是从`超键` 出发的箭头，因此该`关系变量` 不处于`BCNF`（尽管它处于`3NF`）。

最后，正如您肯定知道的，`规范化` 原则指出：如果`关系变量` _R_ 不处于`BCNF`，那么将其分解为处于`BCNF` 的`投影`。对于`关系变量` `SNP`，以下任一分解都将满足此目标：

- 在 `{SNO,SNAME}` 和 `{SNO,PNO,QTY}` 上进行`投影`
- 在 `{SNO,SNAME}` 和 `{SNAME,PNO,QTY}` 上进行`投影`

我现在可以解释为什么 `BCNF` 是那个"怪胎"，即没有简单地被称为"第 _n_ `范式`"（对于某个 _n_）。引用 Codd 首次描述这个新`范式` 的论文中的话：<sup>15</sup>

> 最近，Boyce 和 Codd 提出了以下`定义`：一个 [`关系变量`] R 处于`第三范式`，如果它处于`第一范式`，并且，对于 R 的任何属性集合 C，如果任何不在 C 中的属性在函数上依赖于 C，那么 R 中的所有属性都在函数上依赖于 C [*换句话说，C 是`超键`*]。

因此，Codd 在这里给出了他认为是对*第三*`范式` 的"新改进"`定义`。但问题是，新`定义` 严格强于旧`定义`；也就是说，任何通过新`定义` 处于`3NF` 的`关系变量` 肯定通过旧`定义` 也处于`3NF`，但反之则不然——一个`关系变量` 可以通过旧`定义` 处于`3NF` 而不通过新`定义` 处于`3NF`（上面讨论的`关系变量` `SNP` 就是一个例子）。因此，那个"新改进"`定义` 实际上`定义` 了一个新的、更强的`范式`，因此需要一个独特的名称。然而，到这一点被充分认识到时，Fagin 已经`定义` 了他所称的`第四` `范式`，所以*那个* 名称不可用。<sup>16</sup> 因此产生了异常的名称 _`Boyce/Codd normal form`_。

4.4 首先，以下是两个`定义`，为后续引用编号：

    1.  `关系变量` *R* 处于`2NF`，当且仅当，对于 *R* 的每个`键` *K* 和 *R* 的每个`非键属性` *A*，`FD` *K* → {*A*} 是`不可约` 的。
    2.  `关系变量` *R* 处于`2NF`，当且仅当，对于在 *R* 中成立的每个`非平凡` `FD` *X* → *Y*，以下至少一项为真：(a) *X* 是`超键`；(b) *Y* 是`子键`；(c) *X* 不是`子键`。
    
    设 *R* 不按`定义` 1 处于`2NF`。那么存在一个 `FD`——根据`定义` 是`非平凡` 的——*K* → {*A*}，其中 *K* 是 *R* 的`键`，*A* 是 *R* 的`非键属性`，该 `FD` 在 *R* 中成立且是`可约` 的。由于它是`可约` 的，（也是`非平凡` 的）`FD` *X* → {*A*} 在 *R* 中成立，对于某个真`子键` *X*（*X* ⊂ *K*）。因此，用 *Y* 表示 {*A*}，我们有一个在 *R* 中成立的`非平凡` `FD` *X* → *Y*，使得 *X* 不是`超键`，*Y* 不是`子键`，并且 *X* 是`子键`。因此，根据`定义` 2，R 不处于`2NF`。所以，松散地说，`定义` 2 蕴含`定义` 1。<sup>18</sup>
    
    现在，设 *R* 不按`定义` 2 处于`2NF`。那么存在一个`非平凡` `FD` *X* → *Y*（记为 *F*）在 *R* 中成立，使得 *X* 不是`超键`，*Y* 不是`子键`，并且 *X* 是`子键`。但是，如果 *X* 是`子键` 而不是`超键`，那么它必须是某个`键` *K* 的*真*`子键`。现在有两种情况需要考虑：
    
        a.  *Y* 包含一个`非键属性` *A*。在这种情况下，*K* → {*A*} 在 *R* 中成立但是`可约` 的，因此根据`定义` 1，*R* 不处于`2NF`；因此，再次松散地说，`定义` 1 蕴含`定义` 2。
        b.  不存在这样的 *F* 使得 *Y* 包含`非键属性` *A*。但是，对于每个 *F*，*Y* 中包含的每个属性 *A* 都使得 {*A*} 是`子键`。因此 *R* 处于`3NF`（因此肯定处于`2NF`）：矛盾。
    
    因此，`定义` 1 和`定义` 2 是等价的。

4.5 考虑以下（无效的！）论证。

    设`关系变量` *R* 不处于`2NF`。那么必须存在 *R* 的某个`键` *K* 和 *R* 的某个`非键属性` *A*，使得 `FD` *K* → {*A*}（必然在 *R* 中成立）是`可约` 的——意味着可以从 *K* 中删除某个属性，得到 *K'*，使得 `FD` *K'* → {*A*} 仍然成立。因此 *K* 必须是复合的。
    
    这个论证似乎表明练习的答案必须是*是*——即，如果一个`关系变量` 不处于`2NF`，那么它必须具有复合`键`。但这个论证是不正确的！这里有一个反例。设 `USA` 是一个具有属性 `COUNTRY` 和 `STATE` 的二元`关系变量`；谓词是 *STATE 是 COUNTRY 的一部分*，但每个`元组` 中的 `COUNTRY` 都是美国。现在，`{STATE}` 是该`关系变量` 的唯一`键`，因此 `FD` `{STATE} → {COUNTRY}` 肯定成立。然而，`FD` `{ } → {COUNTRY}` 显然也成立（参见下面练习 4.10 的答案）；因此 `FD` `{STATE} → {COUNTRY}` 是`可约` 的，所以该`关系变量` 不处于`2NF`，然而`键` `{STATE}` 并不是复合的。

4.6 不！作为反例，考虑上一个练习答案中的`关系变量` `USA`。该`关系变量` 受制于 `FD` `{ } → {COUNTRY}`，该 `FD` 既不是`平凡` 的，也不是从`超键` 出发的箭头，因此该`关系变量` 不处于`BCNF`。（事实上，当然，它甚至不处于`2NF`，正如我们在上一个练习的答案中所看到的。）因此，该`关系变量` 可以无损分解为其在 `{COUNTRY}` 和 `{STATE}` 上的两个一元`投影`。（请注意，重建原始`关系变量` 所需的相应`连接` 实际上简化为`笛卡尔积`。）

4.9 设 `FD` _X_ → _Y_ 在 _R_ 中成立。根据`定义`，_X_ 和 _Y_ 是 _R_ 的`标题` 的子集。鉴于一个包含 _n_ 个元素的集合有 2<sup>_n_</sup> 个可能的子集，因此 _X_ 和 _Y_ 各有 2<sup>_n_</sup> 个可能的值，因此可能在 _R_ 中成立的 `FD` 数量的上限是 2<sup>2*n*</sup>。例如，如果 _R_ 的度为五，则可能成立的 `FD` 数量的上限是 1,024（其中 243 个是`平凡` 的）。_子练习_：
a. 那个 243 的数字是从哪里来的？
_答案_：我让您自己弄清楚！
b. 假设这 1,024 个 `FD` 确实都成立。那么在这种情况下，我们可以对 _R_ 得出什么结论？
_答案_：它的`基数` 必须小于二。原因是，在这种情况下成立的一个 `FD` 是 `{ } → *H*`，其中 _H_ 是`标题`；因此 `{ }` 是一个`键`，所以 _R_ 被约束为最多包含一个`元组`，正如在下一个练习的答案中所解释的那样。

    至于 *R* 可以有多少个`键`：设 *m* 是大于或等于 *n*/2 的最小整数。如果 (a) 每个不同的 *m* 个属性集合都是一个`键`，或者 (b) *m* 是奇数且每个不同的 (*m* – 1) 个属性集合都是一个`键`，那么 *R* 将具有最大可能数量的`键`。无论哪种方式，都可以得出`键` 的最大`数量` 是 *n*! / (*m*! × (*n* – *m*)!)。<sup>21</sup> 例如，度为五的`关系变量` 最多可以有十个不同的`键`，度为三的`关系变量` 最多可以有三个不同的`键`。（后一种情况的一个示例可以在附录 C 中找到。）

4.10 设指定的 `FD` _X_ → _Y_ 在`关系变量` _R_ 中成立。现在，_每个_`元组`（无论它是否是 _R_ 的`元组`）对于该`元组` 在空属性集上的`投影` 都具有相同的值——即，0-`元组`（参见第 2 章练习 2.12 的答案）。因此，如果 _Y_ 为空，那么对于 _R_ 的属性的所有可能集合 _X_，`FD` _X_ → _Y_ 都成立；事实上，它是一个`平凡` `FD`（因此它不是很有趣），因为空集是每个集合的子集，因此在这种情况下 _Y_ 肯定是 _X_ 的子集。另一方面，如果 _X_ 为空，那么 `FD` _X_ → _Y_ 意味着，在任何给定时间，_R_ 的所有`元组` 在 _Y_ 上具有相同的值（因为它们肯定在 _X_ 上具有相同的值）。此外，如果 _Y_ 反过来是 _R_ 的整个`标题`——换句话说，如果 _X_ 是`超键`——那么 _R_ 被约束为最多包含一个`元组`（否则 _R_ 将遭受`超键` `唯一性` 违规）。_注意_：在后一种情况下，_X_ 不仅是`超键`，而且事实上是`键`，因为它肯定是`不可约` 的。此外，它是*唯一* 的`键`，因为`标题` 的每个其他子集都将其作为真子集包含。

4.15 在这里，我认为首先给出 **`Tutorial D`** 中 `<relation assign>` 的部分语法可能会有所帮助，这是 **`Tutorial D`** 中基本的`关系` 更新运算符。（请注意，为了当前目的，语法略有简化。至于各种语法类别的名称，它们应该是直观上不言自明的。）

```
<relation assign>
    ::=   <relvar name> := <relation exp>
        | <insert> | <delete> | <update>

<insert>
    ::=   INSERT <relvar name> <relation exp>

<delete>
    ::=   DELETE <relvar name> <relation exp>
        | DELETE <relvar name> [ WHERE <boolean exp> ]

<update>
    ::=   UPDATE <relvar name> [ WHERE <boolean exp> ] :
                               { <attribute assign commalist> }
```

而 `<attribute assign>`，如果相关属性碰巧是`关系值`，基本上就是一个 `<relation assign>`（只是在该 `<relation assign>` 中，相关的 `<attribute name>` 出现在目标 `<relvar name>` 的位置），这就是我们开始的地方。那么，以下是所需更新的 **`Tutorial D`** 语句：

b. `UPDATE SPQ WHERE SNO = 'S2' : { INSERT PQ RELATION { TUPLE { PNO 'P5' , QTY 500 } } } ;`

4.17 一个明显的评论是，早在 1995 年，我就和其他人一样困惑！尽管我确实（无意？）通过使用来自编程语言世界的相对体面的术语*`标量`* 来代替那个相当可疑的术语*`原子`* 来掩盖我的困惑。然而，我已经意识到，在此上下文中，_`标量`_ 实际上并不比 _`原子`_ 更形式化或更精确。事实是，这两个术语都没有绝对的含义——这完全取决于我们想用数据做什么。例如，有时我们想将整个零件编号集合作为单个事物来处理；在其他时候，我们想处理该集合内的单个零件编号——但那时我们正在下降到更低的细节级别（换句话说，更低的抽象级别）。以下类比可能有助于澄清这一点。在物理学中——毕竟，"`原子性`"术语就来源于此——情况是完全平行的：有时我们想把单个物理`原子` 视为不可分割的对象；在其他时候，我们想思考构成这些`原子` 的`质子`、`中子` 和`电子`。当然，那些`质子` 和`中子` 也不是真正不可分割的——它们包含各种称为 _`夸克`_ 的"`亚亚原子`"粒子。等等，可能（？）。

    最后一点：*可能* 会尝试挽救绝对`原子性` 的概念，如下所述。让我们同意说，一个值是*不*`原子` 的，当且仅当存在运算符可以"将值拆开"，可以这么说。（我认为这个想法，或非常类似的想法，可能就是 Codd 在说 *`不可分解`* 具体意味着"由 `DBMS` 不可分解"时脑海中的想法。）因此，字符字符串不是`原子` 的，因为它可以通过 `SUBSTRING` 运算符拆开；集合不是`原子` 的，因为它可以通过从集合中提取元素或子集的运算符拆开；等等。相比之下，定点数是`原子` 的，如果没有运算符可以提取其整数和小数部分（比如说）。但是，即使我们接受这个论点，在我看来，我们也必须接受这样一个论点：完全相同的值今天可能是`原子` 的，明天可能是复合的！例如，对于一个定点数，如果最初没有运算符可以提取其整数和小数部分，但随后引入了此类运算符，就会出现这种情况。而且，在我看来，如果`原子性` 的概念以这种方式依赖于时间，那么（如先前所声称的）它确实没有任何绝对的含义。
