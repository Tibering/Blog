**图 3-1**

回想一下，`函数依赖`（`FD`）

```
{ CITY } → { STATUS }
```

在`关系变量` `S` 中成立（我已在图中添加箭头以提示这一事实）。由于该`函数依赖`在该`关系变量`中成立<sup>1</sup>，结果表明该`关系变量`处于`第二范式`（`2NF`）但不处于`第三范式`（`3NF`）。因此，它存在`冗余`问题；具体而言，某个城市具有某个状态这一事实在一般情况下会多次出现。而`进一步规范化`的规范——请注意，从此刻起，我将大多数时候将其简称为`规范化`（不加限定）——因此建议我们将该`关系变量`分解为两个度数更小的`关系变量` `SNC` 和 `CT`，如图 3-2 所示（当然，图中显示的是对应于图 3-1 中`关系变量` `S` 示例值的这些`关系变量`的值）。

**图 3-2**  
`关系变量` `SNC` 和 `CT`——示例值

由此示例引出的要点：

- 首先，该分解（或`规范化`）确实消除了`冗余`——某个城市具有某个状态这一事实现在仅出现一次。
- 其次，==分解过程本质上是一个`取投影`的过程==——图 3-2 中显示的`关系`分别是图 3-1 中显示的`关系`的`投影`。实际上，我们可以写出几个`方程`<sup>2</sup>：

  ```
  SNC  =  S { SNO , SNAME , CITY }
  CT   =  S { CITY , STATUS }
  ```

  _注意_：其他类型的分解也是可能的，但在进一步说明之前，==我将假定术语`分解`（不加限定）特指通过`投影`进行的分解。==

- 第三，分解过程是`无损`的（也称为`无损失`）——在此过程中不会丢失任何信息，因为图 3-1 中显示的`关系`可以通过对图 3-2 中显示的`关系`进行`连接`来重建：

  ```
  S  =  SNC JOIN CT
  ```

  （再次使用`Tutorial D`语法）。因此，我们可以说图 3-1 中的`关系`与图 3-2 中的一对`关系`是`信息等价`的——或者更精确地说，对于可以对图 3-1 的`关系`执行的任何`查询`，都存在一个可以对图 3-2 的`关系`执行的相应`查询`（反之亦然），且产生相同的结果。显然，分解的这种`无损性`是一个重要属性；无论我们通过`规范化`做什么，当然都不能在执行过程中丢失任何信息。

- 由上述内容可知，==正如`投影`是（就常规理解的`规范化`而言）`分解`操作符一样，`连接`也是相应的`重组`操作符。==

## 规范化服务于两个目的

到目前为止，一切顺利；这些都是非常熟悉的内容。但现在我想指出，如果您一直仔细关注，您`可能`有理由指责我在上述内容中实施了一个小小的（？）欺骗……具体来说，==我考虑的是`关系`的`分解`为`无损`意味着什么；但`规范化`（我们本应讨论的主题）并非关于分解`关系`，而是关于分解`关系变量`。==（毕竟，`数据库设计`的定义就是关于选择哪些`关系变量`（而非哪些`关系`）应该存在于数据库中。）

假设我们确实决定执行建议的将`关系变量` `S` 分解为`关系变量` `SNC` 和 `CT` 的操作。请注意，现在我真正谈论的是`关系变量`而非`关系`；不过，为明确起见，让我们假设所讨论的`关系变量`分别具有图 3-1 和图 3-2 中显示的示例值。再次为明确起见，让我们特别关注`关系变量` `CT`。好吧，该`关系变量`确实是一个`关系变量`——我的意思是，它是一个变量——因此我们可以对其进行`更新`。例如（使用第 2 章中介绍的`元组`简写`表示法`），我们可能插入`元组`

```
( 'Rome' , 10 )
```

但在该`更新`之后，`关系变量` `CT` 包含了一个在`关系变量` `S` 中没有对应项的`元组`（顺便说一句，它在`关系变量` `SNC` 中也没有对应项）。现在，这种可能性经常被使用——事实上，`Codd` 在他关于`规范化`的最早论文中就使用过它（参见附录 `D`）——作为支持首先进行分解的论据：由此产生的双`关系变量`设计能够表示原始单`关系变量`设计无法表示的某些信息。（在当前情况下，它可以表示当前没有供应商位于其中的城市的`状态`信息。）但同一事实也意味着这两种设计实际上并非真正`信息等价`，而且`关系变量` `CT` 实际上也并非`关系变量` `S` 的`投影`<sup>3</sup>——它包含的`元组`并非`投影`自`关系变量` `S` 中的任何`元组`，也非以其他方式从中派生<sup>4</sup>。或者更确切地说（也许更切题），`CT` 也不是 `SNC` 和 `CT` 的`连接`的`投影`，因此该`连接`在某种意义上"丢失信息"；具体来说，它丢失了罗马的`状态`为`10` 这一信息<sup>5</sup>。

类似的情况也会出现在如果我们从`关系变量` `SNC` 中删除`元组`

```
( 'S5' , 'Adams' , 'Athens' )
```

时。在该`更新`之后，我们可以稍微宽松地说<sup>6</sup>，`关系变量` `S` 包含了一个在`关系变量` `SNC` 中没有对应项的`元组`（尽管它在`关系变量` `CT` 中有对应项）。因此，==这两种设计再次并非真正`信息等价`；而且这一次，`关系变量` `S` 也并非`关系变量` `SNC` 和 `CT` 的`连接`，因为它包含的`元组`在`关系变量` `SNC` 中没有对应项。==

因此，这两种设计实际上并非`信息等价`。但我之前不是说过分解的`无损性`是一个重要属性吗？我们通常不是假设如果设计 `B` 是通过`规范化`设计 `A` 产生的，那么设计 `B` 和设计 `A` 应该是`信息等价`的吗？这里到底发生了什么？

为了回答这些问题，查看`关系变量`的`谓词`是有帮助的。`SNC` 的`谓词`是：

- _供应商 `SNO` 名为 `SNAME` 且位于城市 `CITY`。_

而`CT` 的`谓词`是：

- _城市 `CITY` 具有`状态` `STATUS`。_

现在，假设一个城市即使没有供应商位于其中也可能具有`状态`；换句话说，假设`关系变量` `CT` 可能包含一个在`关系变量` `SNC` 中没有对应项的`元组`，例如 `(Rome,10)`<sup>7</sup>。==_那么，仅由`关系变量` `S` 组成的设计根本就是错误的。==_ 也就是说，如果`谓词` _城市 `CITY` 具有`状态` `STATUS`_ 的一个真实`实例`可能存在，而同时（具有相同的 `CITY` 值）`谓词` _供应商 `SNO` 名为 `SNAME` 且位于城市 `CITY`_ 的真实`实例`不存在，那么仅由`关系变量` `S` 组成的设计就不能忠实地反映现实世界中的事态（因为该设计无法表示没有供应商位于其中的城市的`状态`）。

类似地，假设一个供应商可以位于某个城市，即使该城市没有`状态`；换句话说，假设`关系变量` `SNC` 可能包含一个在`关系变量` `CT` 中没有对应项的`元组`，例如 `(S6,Lopez,Madrid)`。那么，同样，仅由`关系变量` `S` 组成的设计也是不正确的，因为它要求每个有供应商位于其中的城市都必须具有某个`状态`。

这里还有另一种看待上述论点的方式。假设仅由`关系变量` `S` 组成的设计确实忠实地反映了现实世界中的事态。==那么`关系变量` `SNC` 和 `CT` 将受以下`完整性约束`（" `SNC` 中的每个城市都出现在 `CT` 中，反之亦然"）的约束==：

```
CONSTRAINT ... SNC { CITY } = CT { CITY } ;
```

但这个`约束`<sup>8</sup>——它是我稍后将称为`相等依赖` 或 `EQD` 的一个示例<sup>9</sup>——显然在讨论的示例中并未得到满足。_注意_：为简单起见，如您所见，我没有费心给这个`约束`命名。事实上，从此刻起，除非有某些令人信服的理由，否则我将省略本书所有示例中的此类名称。

总结来说，==我们看到`规范化`可以（并且确实）用于解决两个相当不同的问题：==

1.  它可用于修复逻辑上不正确的设计，如本节前面讨论的示例所示。_练习_：第 1 章"设计理论的地位"一节中的 `STP` 示例是否存在与该示例中提出的问题类似的问题？（_答案_：是的，存在。）
2.  它可用于减少逻辑上正确的设计中的`冗余`。（显然，设计不必在上述意义上逻辑不正确才会表现出`冗余`。）

在实践中，由于这两种情况常常没有被清晰地区分，因此产生了大量混淆。事实上，大多数文献都侧重于情况 2——为明确起见，我在下文中也将假设情况 2（在有任何区别的情况下）——但请不要忽视情况 1，在实践中，情况 1 至少同样重要，甚至可能更重要。

此外，我应该指出，严格来说，`投影`和`连接`的术语仅适用于情况 2。这是因为在情况 1 中，正如我们所见，=="新"`关系变量`不一定是"旧"`关系变量`的`投影`，"旧"`关系变量`也不一定是"新"`关系变量`的`连接`（如果您明白我的意思）。==实际上，谈论`关系变量`（而非`关系`）的`投影`和`连接`究竟意味着什么呢？好吧，正如我在其他地方所写，大致如下<sup>10</sup>：

- 根据==定义，`投影`、`连接`等操作符特别适用于`关系`值。==当然，它们特别适用于恰好是`关系变量`当前值的那些值。因此，谈论例如`关系变量` `S` 在`属性` `{CITY,STATUS}` 上的`投影`显然是有意义的，这意味着对作为该`关系变量` `S` 当前值的`关系`在这些`属性`上取`投影`所得到的`关系`。然而，在某些上下文中（例如`规范化`），事实证明，以略有不同的方式使用诸如" `关系变量` `S` 在`属性` `{CITY,STATUS}` 上的`投影`"这样的表达式是方便的。具体来说，我们可能会宽松但非常方便地说，某个`关系变量` `CT` 是`关系变量` `S` 在`属性` `{CITY,STATUS}` 上的`投影`——更精确地说，这意味着`关系变量` `CT` 在任何时候的值都等于`关系变量` `S` 在相关时刻的值在这些`属性`上的`投影`。因此，在某种意义上，我们可以谈论`关系变量`本身的`投影`，而不仅仅是`关系变量`当前值的`投影`。类似的评论适用于所有`关系`操作。

换句话说，即使在情况 1 中，我们仍然使用`投影` / `连接`的术语。这种说法有些不当——甚至可以说是草率——但它至少是简洁的。但更准确的说法应该是：分解并非一个取`投影`本身的过程，而是一个`类似于`（但不完全相同于）我们取`投影`时所做的事情的过程（`重组`和`连接`的情况类似）。

## 更新异常

`更新异常`的概念经常在`规范化`的上下文中被提及。现在，应该清楚的是，任何类型的`冗余`都可能导致异常——因为`冗余`大致意味着某条信息被表示了两次，因此总是存在两种表示不一致的可能性（即，如果其中一个被`更新`而另一个没有）。更具体地说，让我们考虑`关系变量` `S` 的情况，其中以下`函数依赖`（`FD`）成立：

```
{ CITY } → { STATUS }
```

该`函数依赖`所产生的`冗余`本身——即，某个城市具有某个`状态`这一事实的重复或重复表示——已经讨论过。它会导致如下异常（这些示例假设图 3-1 中显示的`关系变量` `S` 的示例值）：

- **`插入异常`**：在罗马有供应商之前，我们无法插入罗马的`状态`为 10 这一事实。
- **`删除异常`**：如果我们删除雅典中唯一的供应商，我们将丢失雅典的`状态`为 30 这一事实。
- **`修改异常`**：我们无法更改（"修改"）给定供应商的城市而不更改（"修改"）该供应商的`状态`（一般情况下）。而且，我们无法修改给定供应商的`状态`而不对相关城市中的所有供应商进行相同的修改。

用两个`投影` `关系变量` `SNC` 和 `CT` 替换`关系变量` `S` 可以解决这些问题（具体如何解决？）。此外，让我正式声明，`关系变量` `S`（如前所述）处于`第二范式`但不处于`第三范式`，而`关系变量` `SNC` 和 `CT` 都处于`第三范式`，实际上也处于`BCNF`。一般来说，`BCNF` 是解决上述类型异常所导致问题的方案。

## 范式层次结构

如您所知，存在`许多`不同的`范式`。下面的图 3-3 是我们首次展示我所谓的`范式层次结构`（但请注意，我将在后续章节——特别是第 13-15 章——中扩展该层次结构）。

**图 3-3**  
`范式层次结构`（I）

由此引出的要点：

- 首先，您可能认为该层次结构是颠倒的，因为它将最高`范式`显示在底部，最低`范式`显示在顶部。我不想争论这一点；让我只是说，以我在图中展示的方式显示它（在我看来）更符合这样一个事实，例如，所有`2NF` `关系变量`都在`1NF`中，但某些`1NF` `关系变量`不在`2NF`中。
- 存在许多不同的`范式`：第一、第二、第三，等等。图中显示了其中六个，但如您所见，它们并非标记为第一、...、第六（不完全是）——在第三和第四之间有一个闯入者，`BCNF`。我将在第 4 章解释这种术语奇特性的原因；现在，让我只是说，名称 `BCNF` 是 `Boyce/Codd 范式` 的缩写。然而，尽管有 `BCNF` 这个例外，使用术语 `第 n 范式` 来泛指不同的`规范化`级别是方便的，我将在下文中不时采用这种用法。
- 图中还在 `BCNF` 和 `4NF` 之间显示了一个故意的间隙。然而，该间隙并不意味着在该点可能存在某些"缺失"的`范式`（实际上没有）；相反，它反映了这样一个事实，即在前四个`范式`和最后两个`范式`之间，层次结构中存在一种概念上的跳跃或转变。请参阅本书第三部分以获取进一步解释。
- 除第一个（`1NF`）之外的所有`范式`都是根据某些`依赖`来定义的——在此上下文中，这只是`完整性约束`的另一个术语。从`规范化`的角度来看，主要的`依赖`类型是`函数依赖`（`FDs`）和`连接依赖`（`JDs`）。
- `BCNF` 和 `5NF` 在图中被突出显示（设置为粗体），以表明它们相对于图中显示的其他`范式`的相对重要性。`BCNF` 是根据`函数依赖`定义的，而 `5NF` 是根据`连接依赖`定义的。事实上，正如我们将在后续章节中看到的那样，就`函数依赖`（`FDs`）而言，`BCNF` 确实是"那个"`范式`，而就`连接依赖`（`JDs`）而言，`5NF` 确实是"那个"`范式`。
- 一般来说，从设计角度来看，`规范化`的级别越高越好——因为`规范化`的级别越高，可能出现的`冗余`就越少，因此可能出现的`更新异常`也就越少。
- 一个`关系变量`可能处于`第 n 范式`但不处于`第 (n+1) 范式`。
- 相比之下，如果`关系变量` `R` 处于`第 (n+1) 范式`，那么它肯定处于`第 n 范式`。换句话说，`第五范式`（`5NF`）蕴含`第四范式`（`4NF`），依此类推。因此，说例如`关系变量` `R` 处于`BCNF` 并不排除 `R` 也处于`5NF` 的可能性。然而，在实践中，关于`关系变量` `R` 处于例如`BCNF` 的陈述通常被理解为意味着 `R` 处于`BCNF` _且不处于任何更高的范式_。因此，请仔细注意，我在本书中`不`遵循这种用法。
- 如果`关系变量` `R` 处于`第 n 范式`但不处于`第 (n+1) 范式`，那么它总是可以通过`投影`以`无损`方式分解，使得 (a) 这些`投影`通常处于`第 (n+1) 范式`，且 (b) `R` 等于这些`投影`的`连接`。
- 最后，由前一点可知，任何给定的`关系变量` `R` 总是可以特别地分解为`5NF` `投影`。换句话说，`5NF` 总是可以实现的。

_关于`冗余`概念的注释_：在第 1 章中，我说`设计理论`主要——但非完全——关于减少`冗余`，并且我在本章中多次提及`冗余`；特别是，我说过`规范化`的级别越高，防止的`冗余`就越多。但是，提出一个关于`冗余`究竟是什么的精确定义似乎相当困难！——事实上如此困难，以至于我认为在本书的这个早期阶段，甚至尝试定义它都是不合适的，因此我不会这样做。换句话说，在进一步说明之前，我将只是假设我们至少能在看到`冗余`时识别它（即`冗余`本身）——尽管实际上，即使这也是一个相当大的假设。第 17 章将深入探讨这个概念。

## 规范化与约束

在与`规范化`相关的另一个问题<sup>11</sup>中，有一个经常被忽视的问题。再次考虑将`关系变量` `S` 分解为其在 `{SNO,SNAME,CITY}` 和 `{CITY,STATUS}` 上的`投影` `SNC` 和 `CT` 的示例。那么有三种情况需要考虑：

==考虑属性对应关系 | 考虑==

1.  假设原始设计（仅由`关系变量` `S` 组成）至少在逻辑上是正确的（即，它仅仅存在`冗余`问题）。==正如我在"规范化服务于两个目的"一节中指出的那样，那么在这两个`投影`之间存在某个`约束`（一个`相等依赖`）==：

    ```
    CONSTRAINT ... SNC { CITY } = CT { CITY } ;
    ```

    （" `SNC` 中的每个城市都出现在 `CT` 中，反之亦然"）。

2.  或者，==假设 `SNC` 和 `CT` 中的一个可能包含在另一个中没有对应项的`元组`，而反之则不可能==。为明确我们的想法，再次假设 `CT` 可能包含一个在 `SNC` 中没有对应项的`元组`，例如 `(Rome,10)`，而 `SNC` 永远不能包含一个在 `CT` 中没有对应项的`元组`。那么在这两个`投影`之间存在一个`外键约束`<sup>12</sup>（在刚才提到的具体示例中，从 `SNC` 到 `CT`）：

    ```
    FOREIGN KEY { CITY } REFERENCES CT
    ```

3.  第三种可能性，也许比前两种可能性更小，是 `CT` 和 `SNC` 可能都被允许包含在对方中没有对应项的`元组`。例如，`CT` 可能包含`元组` `(Rome,10)`，但没有供应商位于罗马，而 `SNC` 可能包含`元组` `(S6,Lopez, Madrid)`，但马德里没有`状态`。=在这种情况下，显然在这两个`关系变量`之间根本不存在涉及城市的`约束`==（至少，为了示例的缘故，我们假设不存在）。

现在，稍微简化一下，我说过处于`第 n 范式`的`关系变量` `R` 总是可以`无损`分解为处于`第 (n+1) 范式`的`投影`。然而，==正如上述讨论所表明的那样，_这种分解通常意味着现在至少有一个新的`约束`需要被维护_。==使问题更复杂的是，所讨论的`约束`是一个`多关系变量约束`（即，它跨越两个`关系变量`，在某些情况下可能跨越两个以上）。==因此，存在一个权衡：我们是想要分解的好处，还是想要避免那个`多关系变量约束`<sup>13</sup>？==

至少在 `SNC` 和 `CT` 的示例中，有人可能会争辩说，分解也意味着现在有一个`约束`*不需要*被维护：即，`函数依赖` `{CITY} → {STATUS}`。==但这个论点并非完全有效——在此方面，分解所做的全部工作是将该`约束`从一个`关系变量`移动到另一个（实际上是从`关系变量` `S` 移动到`关系变量` `CT`，在那里它作为维护`{CITY}` 是一个`键`这一`约束`的副作用而被维护）。==

现在，在讨论的简单示例中，进行分解的好处几乎肯定超过不进行分解的好处。但情况并非总是如此；事实上，在更复杂的情况下，是否进行分解的问题可能相当令人烦恼。在下文中，为了避免大量重复的文本，我将通常假设我们总是想要进行分解——但请不要忘记，有时可能存在令人信服的论点反对这样做，特别是在比当前示例更复杂的示例中，例如本书第三部分和第四部分中讨论的那些。

## 相等依赖

再次考虑将`关系变量` `S` 分解为其在 `{SNO,SNAME,CITY}` 和 `{CITY,STATUS}` 上的`投影` `SNC` 和 `CT` 的示例。假设以下`约束`在这两个`投影`之间成立：

```
CONSTRAINT ... SNC { CITY } = CT { CITY } ;
```

（" `SNC` 中的每个城市都出现在 `CT` 中，反之亦然"）。如前一节所述，这个`约束`是所谓`相等依赖`的一个示例。我将在片刻后精确定义这个概念。然而，首先，我想引入一个约定，我将从此刻起使用该约定来稍微简化`投影`的`表示法`<sup>14</sup>。设`关系`表达式 `rx` 求值为`关系` `r`；设`关系` `r` 具有`标题` `H`；设 `X` 是 `H` 的子集；并设 `A1`, ..., `An` 是 `X` 的所有`属性`（即，`X` = {`A1`, ..., `An`}）。那么，我们允许将`投影`表达式 `rx`{`A1`,...,`An`} 缩写——稍微不合逻辑地——为 `rx`{`X`}。

现在我可以定义 `EQD` 概念：

- **`定义（相等依赖）`**：宽松地说，一个要求两个指定`关系`相等的`约束`。更精确地说，设 `R1` 和 `R2` 是分别具有`标题` `H1` 和 `H2` 的`关系变量`。此外，设 `X1` 和 `X2` 分别是 `H1` 和 `H2` 的子集，使得存在一个可能为空的`属性`重命名集合，使得将这些重命名应用于`关系变量` `R1` 在 `X1` 的`属性`上的`投影`所得到的结果 `R` 具有`标题` `X2`。那么，`R1` 和 `R2` 之间的`相等依赖`（`EQD`）是一个声明，声明 `R` 和`关系变量` `R2` 在 `X2` 的`属性`上的`投影`必须相等——即，`R` = `R2`{`X2`}。

事实上，`相等依赖`是称为`包含依赖`的更一般现象的一个重要特例：

- **`定义（包含依赖）`**：宽松地说，一个要求一个指定`关系`包含在另一个中的`约束`。更精确地说，设 `R1` 和 `R2` 是分别具有`标题` `H1` 和 `H2` 的`关系变量`。此外，设 `X1` 和 `X2` 分别是 `H1` 和 `H2` 的子集，使得存在一个可能为空的`属性`重命名集合，使得将这些重命名应用于`关系变量` `R1` 在 `X1` 的`属性`上的`投影`所得到的结果 `R` 具有`标题` `X2`。那么，从 `R1` 到 `R2` 的`包含依赖`（`IND`）是一个声明，声明 `R` 必须包含在（即，是子集）`关系变量` `R2` 在 `X2` 的`属性`上的`投影`中——用符号表示，`R` ⊆ `R2`{`X2`}。

由后一个定义引出的要点：

- `外键约束`是 `IND` 的一个特例。例如，在供应商 - 零件数据库中，`关系变量` `SP` 中的 `{SNO}` 是一个`外键`，引用`关系变量` `S` 中的`键` `{SNO}`；因此，存在一个从 `SP` 到 `S` 的 `IND`——`SP` 在 `{SNO}` 上的`投影`包含在 `S` 在 `{SNO}` 上的`投影`中（用符号表示，`SP`{`SNO`} ⊆ `S`{`SNO`}）。但请注意（使用上述定义的`表示法`），一般而言，`INDs`（与特定的`外键约束`不同）并不要求 `X2` 是 `R2` 的一个`键`<sup>15</sup>（甚至是一个`超键`，参见第 4 章）。
- 如前所述，`EQD` 也是 `IND` 的一个特例。更具体地说，`相等依赖` "`A` = `B`" 等价于一对 `包含依赖` "`A` 包含在 `B` 中" 和 "`B` 包含在 `A` 中"（用符号表示，`A` ⊆ `B` 且 `B` ⊆ `A`）。换句话说，`相等依赖` 是一个双向的`包含依赖`。

现在，在接下来的篇幅中，我们将看到大量`相等依赖`（`EQDs`）的示例，而不是一般的`包含依赖`（`INDs`）。事实上，这种状态应该是显而易见的：将`关系变量``无损`分解为`投影`通常至少会导致 `INDs`，并且经常导致 `EQDs`，正如我们已经知道的那样。然而，在某种意义上，那些并非由`无损分解`产生的 `EQDs` 才是有趣的。原因是，此类 `EQD` 的存在常常被证明是`冗余`的标志——因为如果（正如我早些时候所说）某条信息被表示了两次，那么 `EQD` 可能是保持这两种表示一致所需要的。

如果您以前从未听说过 `EQDs`，考虑到它们的概念重要性，您可能想知道为什么没有。当然，它们在文献中似乎没有受到太多关注。在我看来，这种不幸状态最可能的原因是 `SQL` 语言……正如您如果尝试过这个练习就会知道的那样，`EQDs` 在 `SQL` 中表述起来极其笨拙，因为 `SQL` 没有直接表达`关系`比较的方法<sup>16</sup>。支持这一论断的一个显著例子可以在本书第 17 章示例 12 的讨论中找到。

## 结束语

我想通过解决一个我在这本书中到目前为止完全没有讨论过的问题来结束本章。这是一个术语问题。具体来说，为什么 `1NF`、`2NF` 等被称为`范式`？顺便问一下，为什么`规范化`被称为`规范化`？

这些问题的答案源于数学，尽管所涉及的思想也溢出到几个相关学科，特别是计算学科。在数学中，我们经常发现自己必须处理某种类型的大量（甚至可能是无限的）对象集合：例如，所有`矩阵`的集合，或所有`有理数`的集合，或者——稍微接近我们的领域——所有`关系`的集合。在这种情况下，找到所讨论对象的`规范形式`集合是可取的<sup>17</sup>。以下是一个定义：

- **`定义（规范形式）`**：给定一个集合 `s1`，以及该集合元素之间定义的`等价`概念，`s1` 的子集 `s2` 是 `s1` 的`规范形式`集合，当且仅当 `s1` 的每个元素 `x1` 在该`等价`概念下恰好等价于 `s2` 的一个元素 `x2`（并且该元素 `x2` 是元素 `x1` 的`规范形式`）<sup>18</sup>。适用于 `x1` 的各种"有趣"属性也适用于 `x2`；因此，我们可以只研究小集合 `s2`，而不是大集合 `s1`，以证明各种"有趣"的`定理`或结果。

作为这个概念的一个琐碎说明，设 `s1` 是非负整数 `{0,1,2,...}` 的（无限）集合，并设两个这样的整数等价，当且仅当它们除以五时留下相同的余数。那么我们可以定义 `s2` 为集合 `{0,1,2,3,4}`。至于在此示例中适用的一个"有趣"`定理`，设 `x1`、`y1` 和 `z1` 是 `s1` 的任意三个元素（即，任意三个非负整数），并设它们在 `s2` 中的`规范形式`分别为 `x2`、`y2` 和 `z2`；那么乘积 `y1` × `z1` 等价于 `x1`，当且仅当乘积 `y2` × `z2` 等价于 `x2`。

现在，`范式` 只是`规范形式`的另一个术语。因此，当我们在`数据库`上下文中谈论`范式`时，我们谈论的是数据的`规范`表示。具体说明这一点：任何给定的数据集合都可以以多种不同的方式以`关系`方式表示，正如我们所知。当然，所有这些方式实际上——必须——是`信息等价`的；也就是说，`信息等价` 是我们在此特定上下文中诉诸的"定义的`等价`概念"。然而，出于各种原因，其中一些方式（即，表示给定信息的方式）优于其他方式。而那些优选的方式，当然，就是本书大部分主题所讨论的`关系范式`。

至于术语`规范化`，它仅仅指将某个给定对象映射到其`规范`等价物的一般过程。因此，特别是在`数据库`上下文中，它被（正如我们所知）用来指将某个给定的`关系变量`映射到一组`关系变量`的过程，这些`关系变量` (a) 当一起考虑时，与原始`关系变量` `信息等价`，但 (b) 各自单独处于某个优选的`范式`。

对于上述内容，我或许应该补充以下几点。据我所知，`Codd` 本人在他关于该主题的早期著作中从未提及他引入`范式`或`规范化`术语的原因。但许多年后，他确实记录了他自己的、可能是半开玩笑的解释<sup>19</sup>：

- **_`采访者`_**：`规范化` 这个词是从哪里来的？
- **_`Codd`_**：在我看来，必须在`数据库设计`中引入某种规范是至关重要的。我称之为`规范化`，因为当时的`总统尼克松`正在大量谈论与中国的`关系正常化`。我想，如果他能`规范化关系`，我也能。
