Healdsburg, California, USA

> _"什么是正常的，反正？"_
> —— 匿名：《虫子去哪儿》

我想谈谈`` `denormalization` ``（反规范化）。到目前为止，在本书中我尚未考虑任何高于`` `BCNF` ``（Boyce-Codd 范式）的规范化级别（至少没有详细讨论）。但`` `denormalization` ``（反规范化），如果它有任何意义的话，不能仅仅适用于`` `BCNF` ``本身；我的意思是，它不能仅指退回到低于`` `BCNF` ``本身的某个规范化级别。相反，它必须意味着从`` `any` ``（任何）给定的规范化级别退回到某个更低的级别。

话虽如此，我也需要说明，处于`` `BCNF` ``但不处于某个更高范式的`` `relvar` ``（关系变量）相对少见（尽管我急于补充，并非完全未知）。因此，在实践中，`` `denormalization` ``（反规范化）通常确实相当具体地指退回到低于`` `BCNF` ``的某个规范化级别；因此本章被包含在本书的这一部分中。

## "为性能而反规范化"（？）

自从`` `SQL` ``产品首次上市以来，"有必要'为性能而`` `denormalize` ``（反规范化）"的说法已被广泛传播。（错误的！）支持论点大致如下：

1. `` `Normalization` ``（规范化）意味着很多`` `relvar` ``（关系变量）。
2. 很多`` `relvar` ``意味着很多存储文件。
3. 很多存储文件意味着很多 I/O。

以供应商和零件为例，获取供应红色零件的供应商详情的请求涉及两个二元连接——可能是供应商先连接到发货，然后将该连接的结果连接到零件。如果这三个`` `relvar` ``对应于三个物理上独立的存储文件，那么这两个连接将需要大量 I/O，因此性能会很差。

如前所述，至少在原则上，这个论点是错误的。原因是`` `relational model` ``（关系模型）从未规定`` `relvar` ``必须一对一地映射到存储文件。以供应商和零件为例，没有逻辑上的理由说明我们不能将这三个`` `relvar` ``的连接（可能去掉明显的冗余）物理存储为磁盘上的一个单独存储文件，<sup>1</sup> 这可以显著减少所考虑查询的 I/O 量。然而，这一点对于当前目的无关紧要，因为：

- 首先，这个领域是大多数`` `DBMS` ``供应商严重让我们失望的地方；大多数`` `SQL` ``产品确实几乎是一对一地将`` `relvar` ``映射到存储文件。<sup>2</sup> 即使是例外情况，也未能提供我们可能期望的、或关系系统理论上能够实现的那种数据独立性。因此，作为一个实际问题，那个"错误的"论点对于今天的大多数`` `SQL` ``产品来说是有效的，遗憾的是。
- 其次，即使`` `relvar` ``不是一对一映射到存储文件，在存储文件级别进行`` `denormalization` ``（反规范化）可能仍然是可取的。事实上，非一对一映射之所以可取的一个主要原因，恰恰是它们允许在物理级别进行`` `denormalization` ``（反规范化）——这正是它应该属于的地方——而无需让它显示到逻辑级别并因此破坏逻辑级别。

因此，为了讨论起见，让我们假设`` `denormalization` ``（反规范化）有时确实必须在某个级别或其他级别进行。但`` `denormalization` ``（反规范化）`` `is` ``（究竟）是什么？

## 反规范化意味着什么？

奇怪的是，对于一个被如此广泛提倡的做法，关于`` `denormalization` ``（反规范化）实际上由什么组成似乎存在相当大的混淆。（教科书也没多大帮助，即使是那些专门讨论设计主题的教科书。大多数甚至没有提到这个主题，而那些提到的很少提供定义，当然也没有深入讨论这个问题。）例如，几年前我曾有机会阅读一篇专门讨论`` `commercial` ``（商业）`` `SQL` ``产品中`` `denormalization` ``（反规范化）问题的论文。<sup>3</sup> 在下文中我将把那篇论文称为"反规范化论文"。现在，作者一开始就论证`` `against` ``（反对）`` `denormalization` ``（反规范化）。引用如下：

> 我认为`` `normalization principles` ``（规范化原则）应该被视为`` `commandments` ``（戒律）……除非你面临金钱、硬件可扩展性、当前`` `SQL` ``技术、网络优化、并行化或其他性能技术无法解决的性能问题〔_略有改写，斜体添加_〕。

我完全同意这个立场。事实上，我有记录表明我自己也说过非常类似的话：在我 1990 年写的一篇关于在实践中使用`` `SQL` ``系统的论文中，<sup>4</sup> 我推荐`` `denormalization` ``（反规范化）作为性能策略"仅在所有其他方法都失败时"。然而，不幸的是，反规范化论文的其余部分往往表明作者并不真正知道`` `denormalization` ``（反规范化）是什么；在引用了上述开场立场声明之后，论文继续给出了八个"为性能而设计"的示例，其中除了一个之外，绝对与`` `denormalization` ``（反规范化）毫无关系！

然而，为作者辩护，我再次说，在文献中似乎确实很难找到`` `denormalization` ``（反规范化）的精确定义。当然，可以争辩说，鉴于 (a) `` `denormalization` ``（反规范化），无论它可能是什么，肯定必须是`` `normalization` ``（规范化）的逆过程，以及 (b) `` `normalization` ``（规范化）本身确实有精确定义，因此不需要这样的定义。然而，为了记录在案，我将在稍后给出`` `denormalization` ``（反规范化）的精确定义可能是什么样子的概念。在我这样做之前，让我明确表示，我对反规范化论文中建议的具体设计策略没有任何特别的异议；事实上，我在 1982 年自己写的一篇论文中建议了其中几种相同的策略。<sup>5</sup> 我的异议仅在于作者将它们特别称为`` `denormalization tactics` ``（反规范化策略）这一事实。

因此，以下是我自己的定义，仅供参考（如果它看起来有点冗长，我表示歉意）。我从这样一个观察开始，说得稍微宽松一点，`` `normalizing` ``（规范化）某个`` `relvar` `` `` `R` ``意味着`` `decreasing redundancy` ``（减少冗余）。更具体地说，它通过以下方式减少冗余：

1. 将`` `R` ``替换为一组投影`` `R1` ``, ..., `` `Rn` ``，使得`` `R1` ``, ..., `` `Rn` ``中至少有一个处于比`` `R` ``更高的规范化级别，并且还使得
2. 对于`` `R` ``的所有可能值`` `r` ``，如果将相应的`` `R1` ``, ..., `` `Rn` ``的值`` `r1` ``, ..., `` `rn` ``（分别）重新连接在一起，那么该连接的结果等于`` `r` ``。

因此，以下提出的定义：

> **定义（反规范化）：** 对一组`` `relvar` `` `` `R1` ``, ..., `` `Rn` ``进行`` `denormalizing` ``（反规范化）意味着通过以下方式增加冗余：
>
> 1. 将`` `R1` ``, ..., `` `Rn` ``替换为它们的连接`` `R` ``，使得`` `R` ``处于比`` `R1` ``, ..., `` `Rn` ``中至少一个更低的规范化级别，并且还使得
> 2. 对于`` `R1` ``, ..., `` `Rn` ``（分别）的所有可能值`` `r1` ``, ..., `` `rn` ``，将相应的`` `R` ``的值`` `r` ``在`` `Ri` ``的属性上进行投影的结果等于`` `ri` ``（`` `i` `` = 1, ..., `` `n` ``）。

由此定义引出的要点：

- 观察`` `denormalization` ``（反规范化）是一个应用于`` `set` ``（集合）`` `relvar` ``的过程，而不是应用于单独考虑的单个`` `relvar` ``。例如，考虑`` `relvar` `` `` `SNC` ``和`` `CT` ``，其标题分别为 {SNO,SNAME,CITY} 和 {CITY,STATUS}（参见第 3 章图 3-2 中的样本值）。这两个`` `relvar` ``处于`` `BCNF` ``。如果我们将它们连接在一起，我们得到供应商`` `relvar` `` `` `S` ``（它只处于`` `2NF` ``，不在`` `3NF` ``，因此也不在`` `BCNF` ``），因此`` `relvar` `` `` `S` ``可以被视为`` `relvar` `` `` `SNC` ``和`` `CT` ``的反规范化。当然，更重要的是，`` `relvar` `` `` `S` ``比`` `relvar` `` `` `SNC` ``和`` `CT` ``涉及更多的冗余。
- 如果 (a) `` `R1` ``, ..., `` `Rn` ``最初是通过取`` `R` ``的投影获得的——换句话说，如果反规范化实际上是在撤销早期的规范化，可以说，如前一个要点中的供应商示例，并且如果 (b) 那个早期的规范化纯粹是为了减少冗余而不是修复逻辑上不正确的设计（参见第 3 章中关于这两种可能性之间差异的评论），那么 (c) 对于`` `R` ``的所有可能值`` `r` ``，将`` `r` ``在`` `Ri` ``的属性上进行投影必须产生`` `ri` ``（`` `i` `` = 1, ..., `` `n` ``）的要求将自动满足。

支持`` `denormalization` ``（反规范化）的通常论点基本上是它使检索更容易表达并使其性能更好。<sup>6</sup> 这个论点在多大程度上可能有效，我将在稍后的章节中检验。然而，首先，我想指出，一旦我们决定进行`` `denormalization` ``（反规范化），我们就踏上了一条非常滑的斜坡。问题是：我们在哪里停止？这种情况与`` `normalization` ``（规范化）不同，在规范化中有明确的逻辑理由继续该过程直到我们达到可能的最高范式。那么我们是否得出结论，在`` `denormalization` ``（反规范化）中我们应该继续直到我们达到可能的最低范式？肯定不是；然而，没有逻辑标准来决定该过程究竟应该在哪里停止。换句话说，在选择进行`` `denormalization` ``（反规范化）时，我们已经从一个至少有一些坚实科学和逻辑理论支持的立场后退，并用一个纯粹实用主义的立场（以及通常基于对整体问题相当狭隘的视角，这一点我稍后将详细阐述）取而代之。

## 反规范化不是什么（一）

我说过`` `denormalization` ``（反规范化）意味着增加冗余。但这并不意味着增加冗余就意味着`` `denormalization` ``（反规范化）！这是反规范化论文陷入的陷阱之一；它描述的设计策略通常会增加冗余，但正如前面指出的，除了一个唯一的例外之外，它们都不是`` `denormalization per se` ``（反规范化本身）的应用。（在逻辑中，如果`` `p implies q` ``为真，并不意味着`` `q implies p` ``为真，否则论证是众所周知的错误推理示例：如此众所周知，事实上，它有一个特殊的名称，`` `The Fallacy of False Conversion` ``（错误转换谬误）。）

让我们检查反规范化论文中的几个示例。在其中一个示例中，我们得到如下所示的`` `relvar` `` `` `ITEM` ``和`` `SALES` ``：

```tutorial_d
ITEM  { INO , INAME }
      KEY { INO }

SALES { SNO , INO , QTY }
      KEY { SNO , INO }
      FOREIGN KEY { INO } REFERENCES ITEM
```

谓词分别是`` `Item INO has name INAME` ``（项目 INO 的名称为 INAME）和`` `Quantity QTY of item INO were sold in store SNO` ``（项目 INO 的数量 QTY 在商店 SNO 售出）。出于性能原因，该论文建议在`` `ITEM` `` `` `relvar` ``中添加一个`` `TOTAL_QTY` ``属性，其值对于任何给定项目是该项目在所有商店的总销售额。但虽然结果设计确实涉及一些冗余，但事实仍然是两个`` `relvar` ``仍然处于`` `BCNF` ``（特别注意函数依赖 {INO} → {TOTAL_QTY} 在修订版的`` `relvar` `` `` `ITEM` ``中成立）。换句话说，在这个`` `example` ``（示例）中，没有`` `denormalization` ``（反规范化），作为这样的。

第二个示例涉及论文所谓的"内部数组"：

```tutorial_d
EMP { ENO , JAN_PAY , FEB_PAY , ..., DEC_PAY }
    KEY { ENO }
```

谓词是`` `Employee ENO was paid an amount JAN_PAY in January, FEB_PAY in February, ..., and an amount DEC_PAY in December` ``（员工 ENO 在一月份获得金额 JAN_PAY，二月份获得 FEB_PAY，……，十二月份获得金额 DEC_PAY）。并且可能，尽管论文没有明确说明，这种"元组级"设计旨在与——并且出于性能原因，可能优于——以下"属性级"类比进行对比：

```tutorial_d
EMP { ENO , MONTH , PAY }
    KEY { ENO , MONTH }
```

但两种设计都在`` `BCNF` ``中。同样，这里没有`` `denormalization` ``（反规范化）；事实上，为了提前说一点（参见第 17 章），我会说也没有增加冗余。（另一方面，原始的"元组级"设计可能不好，如果你考虑查询"获取至少有一个月工资低于 5K 的员工，以及相关月份"，你就会明白。）

关于上述示例，一位同事最近让我注意到 IBM 为其`` `DBMS` ``产品`` `DB2` ``编写的参考手册中的以下摘录：

> **第一范式：**
> 如果实体的每个实例只包含一个值，从不包含多个重复属性，则关系`` `entity` ``（实体）满足第一范式的要求。重复属性，通常称为重复组，是本质上相同但不同的属性。在满足第一范式要求的实体中，每个属性在其含义和名称上都是独立且唯一的。
>
> **示例：**
> 假设一个实体包含以下属性：
> EMPLOYEE_NAME
> JANUARY_SALARY_AMOUNT
> FEBRUARY_SALARY_AMOUNT
> MARCH_SALARY_AMOUNT
> 这种情况违反了第一范式的要求，因为 JANUARY_SALARY_AMOUNT、FEBRUARY_SALARY_AMOUNT 和 MARCH_SALARY_AMOUNT 本质上是同一个属性，EMPLOYEE_MONTHLY_SALARY_AMOUNT。
>
> ——_DB2 for z/OS Administration Guide_, IBM Form No. SC27-8844-2

嗯，写作当然很糟糕，但它传达的信息——即使能被理解的话——可以说仍然更糟。我能说什么呢。

第三个示例涉及将`` `RESELLERS` `` `` `relvar` ``"水平"拆分为两个单独的`` `relvar` ``，`` `ACTIVE_RESELLERS` ``和`` `INACTIVE_RESELLERS` ``。换句话说，原始`` `relvar` ``通过`` `restriction` ``（限制）（而非投影）进行分解，并通过`` `union` ``（并集）（而非连接）从两个限制中重建。因此，我们显然根本不是在这里谈论经典意义上的`` `normalization` ``（规范化）；因此，我们显然也不是在谈论经典的`` `denormalization` ``（反规范化）。<sup>7</sup>

我再举一个反规范化论文中的示例。这个示例从如下所示的`` `STORE` ``和`` `EMP` `` `` `relvar` ``开始：

```tutorial_d
STORE { SNO , REGION , STATE , ... }
      KEY { SNO , REGION , STATE }

EMP   { ENO , SNO , REGION , STATE , ... }
      KEY { ENO }
      FOREIGN KEY { SNO , REGION , STATE } REFERENCES STORE
```

谓词分别是`` `Store SNO is located in region REGION within state STATE` ``（商店 SNO 位于州 STATE 内的区域 REGION）和`` `Employee ENO is employed at store SNO within region REGION within state STATE` ``（员工 ENO 受雇于州 STATE 内区域 REGION 的商店 SNO）。冗余是显而易见的，因此论文建议为商店引入一个`` `surrogate` ``（代理）标识符，例如 SID，从而将设计修改如下：

```tutorial_d
STORE { SID , SNO , REGION , STATE , ... }
      KEY { SID }
      KEY { SNO , REGION , STATE }

EMP   { ENO , SID , ... }
      KEY { ENO }
      FOREIGN KEY { SID } REFERENCES STORE
```

但这个修订后的设计不仅不涉及任何`` `denormalization` ``（反规范化），它实际上`` `decreases` ``（减少）了冗余！<sup>8</sup>——因为给定的 SNO 与给定的 REGION 和 STATE 的关联现在只出现一次，而不是对于相关商店的每个员工出现一次。（为了彻底说明这一点，这里`` `obviously` ``（显然）没有`` `denormalization` ``（反规范化），因为——除其他外——每个人肯定同意的一件事是，`` `denormalization` ``（反规范化）应该增加冗余。）

顺便说一下，我知道这个示例可能会给人留下我认为`` `surrogates` ``（代理）是好主意的印象。嗯，它们可以是——但遗憾的是，它们并不`` `always` ``（总是）是好主意。事实是，`` `surrogates` ``（代理），虽然它们可能解决一些问题，但也会引入它们自己的进一步问题。有关进一步讨论，请参见本章末尾的练习 8.3，以及第 17 章。

在结束本节时，我想非常清楚地表明，上述讨论绝不是为了攻击反规范化论文或其作者。事实上，该论文中的以下引用清楚地表明，在更大的问题上，作者和我实际上站在同一边：

> [我们应该] 停止批评`` `relational model` ``（关系模型），并明确区分什么是`` `SQL` ``什么是关系型的……两者完全不同。

我完全同意这个立场，也同意其含义，即我们不得不在逻辑级别担心诸如`` `denormalizing` ``（反规范化）之类的事情的唯一原因是当今`` `SQL` ``产品的失败。正如我在其他地方所写，`` `in fact` ``（事实上）：<sup>9</sup> 在一个理想的系统中，我们在逻辑级别根本不需要进行`` `denormalize` ``（反规范化）。即使在今天的系统中，这些系统通常远非理想，我相信我们应该仅作为最后手段进行`` `denormalize` ``（反规范化）。也就是说，只有当所有其他提高性能的策略都以某种方式未能满足要求时，我们才应该从完全规范化的设计后退。（当然，我在这里遵循通常的假设，即`` `normalization` ``（规范化）具有性能影响——正如它在当前`` `SQL` ``产品中通常确实如此。）

## 反规范化不是什么（二）

到目前为止，在本章中，我给出了我认为对`` `denormalization` ``（反规范化）是什么的合理定义，并给出了一些它不是什么的示例。然而，也许我简单地认为该术语应该以任何精确或逻辑意义使用，这本身就是错误的。当然，它在整个行业中并没有非常精确地使用；事实上，它似乎主要用于——特别是在数据仓库上下文中——指几乎任何在我看来只能被描述为不良设计实践的事情。不仅如此，这种不良实践通常还被明确推荐！此类"推荐"的不良实践示例包括：

- 使用重复组<sup>10</sup>
- 允许重复行
- 使用`` `null` ``；可能更糟，允许`` `null` ``出现在键中
- 在同一列中混合不同类型的信息（使用单独的"标志"列来指定该列中每个单独值的类型）
- 使用单个文本列来表示逻辑上应该是不同列的内容

在星型模式术语中，本例中的`` `SP` ``是`` `the` `` `` `fact table` ``（事实表），而`` `TI` ``是`` `a dimension table` ``（维度表）。<sup>13</sup> 供应商`` `relvar` `` `` `S` ``和零件`` `relvar` `` `` `P` ``也是维度表（参见图 8-2）。整体结构被称为"星型模式"，因为相应的实体/关系图被想象成类似星星，事实表被维度表包围——并通过"射线"或"辐条"连接——如图 8-2 所示。（那些射线或辐条当然对应于外键引用。）

## 反规范化有害（一）

在本节中，我想提出一个论点——一个逻辑论点，即，一个你可能以前没见过的论点——来支持你应该仅作为最后手段进行`` `denormalize` ``（反规范化）的立场。本质上，这个论点是，虽然（众所周知）`` `denormalization` ``（反规范化）在逻辑上可能对更新不利，`` `it can be logically bad for retrieval as well` ``（它在逻辑上也可能对检索不利），在某种意义上它可能使某些查询更难表达。（或者，它可能使它们更容易错误地表达，这意味着，如果它们执行，你得到的答案可能本身是正确的，但却是错误问题的答案。）让我举例说明。

再次考虑`` `relvar` `` `` `S` ``，其函数依赖 {CITY} → {STATUS}。如前所述（在"反规范化意味着什么？"一节中），该`` `relvar` ``可以被视为对`` `relvar` `` `` `SNC` ``（属性为 SNO、SNAME 和 CITY）和`` `CT` ``（属性为 CITY 和 STATUS）进行反规范化的结果。现在考虑查询"获取平均供应商城市状态值"。给定我们通常的样本值（参见第 3 章图 3-2），雅典、伦敦和巴黎的状态值分别为 30、20 和 30，因此平均值为 80/3，即四舍五入到三位小数为 26.667。以下是用`` `SQL` ``表达此查询的一些尝试（为简单起见，我假设`` `S` ``非空，因此我们不必担心如果在`` `SQL` ``中尝试将`` `AVG` ``运算符应用于空集会发生什么）：<sup>15</sup>

```sql
SELECT AVG ( STATUS ) AS RESULT
FROM   S
```

_结果_（不正确）：26。这里的问题是伦敦的状态和巴黎的状态都被计算了两次。也许我们需要在`` `AVG` ``调用内部使用`` `DISTINCT` ``？让我们试试：

```sql
SELECT AVG ( DISTINCT STATUS ) AS RESULT
FROM   S
```

_结果_（不正确）：25。不，我们需要检查的是不同的`` `cities` ``（城市），而不是不同的状态值。我们可以通过分组来实现：

```sql
SELECT CITY , AVG ( STATUS ) AS RESULT
FROM   S
GROUP  BY CITY
```

_结果_（不正确）：(雅典,30), (伦敦,20), (巴黎,30)。这个表达式给出的是`` `per city` ``（每个城市）的平均状态，而不是总体平均值。啊，所以也许我们想要的是平均值的平均值？——

```sql
SELECT CITY , AVG ( AVG ( STATUS ) ) AS RESULT
FROM   S
GROUP  BY CITY
```

_结果：_ 语法错误——`` `SQL` ``标准相当正确地不允许以这种方式嵌套"集合函数"调用。<sup>16</sup> 再试一次：

```sql
SELECT AVG ( TEMP.STATUS ) AS RESULT
FROM   ( SELECT DISTINCT S.CITY , S.STATUS
         FROM   S ) AS TEMP
```

_结果_（终于正确了）：26.667。但请注意，与完全规范化设计（`` `relvar` `` `` `SNC` ``和`` `CT` ``）的类比相比，这个表达式是多么复杂：

```sql
SELECT AVG ( STATUS ) AS RESULT
FROM   CT
```

## 反规范化有害（二）

我在前一节说过，支持`` `denormalization` ``（反规范化）的论点是它使检索更容易表达并使其性能更好。但这个论点真的经得起仔细分析吗？让我们仔细看看。

首先，检索更容易表达显然不是普遍正确的；前一节提供了一个详细的反例，但这个观点可以用更简单的示例来说明。作为说明，考虑针对 (a) 图 1-1 的规范化设计和 (b) 一个反规范化设计（其中`` `relvar` `` `` `S` ``、`` `SP` ``和`` `P` ``被替换为一个单一的"连接"`` `relvar` ``，例如称为`` `SSPP` ``）表达查询"获取所有供应商详情"所涉及的内容。以下是`` `Tutorial D` ``表达式：

```tutorial_d
a. S
b. SSPP { SNO , SNAME , STATUS , CITY }
```

或者如果你想要`` `SQL` ``解决方案：

```sql
a. SELECT *
   FROM S
b. SELECT DISTINCT SNO , SNAME , STATUS , CITY
   FROM SSPP
```

下一点是，许多查询的性能也可能更差。造成这种状况有几个原因。一个是`` `denormalization` ``（反规范化）导致`` `redundancy` ``（冗余），这反过来可能导致需要执行重复消除（在这方面，注意上述第二个`` `SQL` ``表达式中的`` `DISTINCT` ``）。另一个原因如下：

- 假设供应商、发货和零件的连接再次表示为一个单独的存储文件。另外，为简单起见，假设任何给定的存储文件由物理上连续的存储记录集合组成，每个记录对应于该存储文件所表示的`` `relvar` ``中当前出现的每个元组。
- 让我们也为了论证起见假设查询"获取供应红色零件的供应商详情"在这种物理结构下将表现相当好。好的；但查询"获取所有供应商详情"的性能将`` `worse` ``（更差）于三个`` `relvar` ``映射到三个物理上独立的存储文件的结构！为什么？因为在后一种设计中，所有供应商存储记录在物理上将是连续的，而在前一种设计中，它们实际上将分布在更广泛的区域，因此将需要更多的 I/O。类似的评论适用于任何仅访问供应商、或仅访问零件、或仅访问发货的查询，而不是执行某种连接。
- 还要注意，`` `denormalization` ``（反规范化），再次因为它增加了冗余，很可能导致更大的存储记录，这个事实也可能导致更多的 I/O，而不是更少。作为一个简单的例子，一个 4K 页面可以容纳两个 2K 存储记录，但只能容纳一个 3K 存储记录；因此，一个使冗余增加 50% 的反规范化可能使 I/O 增加 100%。（当然，我说得相当宽松。）

我的下一个观察是，即使我们接受`` `denormalization` ``（反规范化）使检索更容易表达并表现更好的说法，它肯定使更新更难表达并表现更差。现在，这一点（如我之前所说）被广泛理解；然而，不那么被广泛理解的是，`` `denormalization` ``（反规范化）也为完整性违规打开了大门。例如，在`` `relvar` `` `` `S` ``中（与投影`` `relvar` `` `` `SNC` ``和`` `CT` ``相反），`` `someone` ``（某人）——无论是系统还是用户，在当前实践中可能是后者——将不得不负责维护函数依赖 {CITY} → {STATUS}；如果该维护没有完成，完整性就会丢失。相比之下，在规范化的两`` `relvar` ``设计中，必须做的只是强制执行`` `CT` ``上的键约束——这肯定将由系统而不是用户完成——然后每个城市有一个状态的事实将"自动"维护。

本章的一位审稿人指出，进行规范化设计的另一个优点是，这种设计倾向于简化复杂查询的表达，因为它使代表此类表达式的表达式能够通过嵌套简单的子表达式而更加模块化。这在原则上是正确的，但我有义务补充说，虽然这种模块化表达式可以在像`` `Tutorial D` ``这样设计良好的语言中轻松构建（使用该语言的`` `WITH` ``特性，在本书其他地方的示例中有说明），但在`` `SQL` ``中它们要困难得多——有时甚至不可能。为什么？因为`` `SQL` ``的`` `WITH` ``特性远不如`` `Tutorial D` ``的有用，原因是 (a) `` `SQL` ``的`` `SELECT` ``子句的语义对整体表达式中是否存在某些其他构造（特别是但不限于`` `GROUP BY` ``和`` `HAVING` ``子句以及聚合运算符调用）很敏感，因此也由于 (b) 在`` `SQL` ``中语义构造和语法构造之间根本没有一对一的映射这一结果事实。

我的最后一点是：无论我们谈论的是

1. 真正的`` `denormalization` ``（反规范化），它仅在物理级别进行，还是
2. 我们在今天大多数`` `SQL` ``产品中必须进行的`` `denormalization` ``（反规范化），它也影响逻辑级别，

这一点没有被足够广泛地认识到，当人们说"为性能而`` `denormalize` ``（反规范化）"时，他们实际上指的是`` `specific applications` ``（特定应用程序）的性能。正如我之前所说（在"反规范化意味着什么？"一节中），`` `denormalization` ``（反规范化）通常基于对整体问题相当狭隘的视角。`` `Any` ``（任何）特定的物理设计都可能对某些应用程序有利，但对其他应用程序不利（就其性能影响而言）。

## 结语

在本章中，我提出了强有力的论据反对进行`` `denormalizing` ``（反规范化）；实际上，因此，我提出了支持`` `normalizing` ``（规范化）的论据。良好的设计通常是完全规范化的，这当然是正确的。但重要的是要理解，相反的情况不一定正确！也就是说，一个设计可以是完全规范化的，但仍然不好。例如，`` `relvar` `` `` `S` ``在属性 SNO 和 STATUS 上的投影`` `ST` ``肯定在`` `BCNF` ``中——事实上，它处于可能的最高范式，我们将在本书第四部分看到——但它显然不是一个好的设计，正如我们在第 6 章中看到的。
