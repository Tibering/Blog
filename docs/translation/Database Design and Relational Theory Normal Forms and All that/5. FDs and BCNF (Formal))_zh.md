> _非形式即非规范_

现在，我想退后一步，深吸一口气，重新审视 `FDs` 和 `BCNF`——但这次我想以正确的方式进行（对于涉及少量重复的内容表示歉意）。正如您将很快看到的，本章的处理方式比上一章更为抽象；如果您已完全掌握上一章的内容，理解本章应该不会太困难，但本章的形式化程度肯定会更高。因此，**在您完全吸收上一章的所有内容之前，我不希望您查看本章**。（当然，这应该不难做到，因为上一章的大部分内容对您来说可能已经很熟悉了。）

首先说明一个总体要点：由于 `BCNF` 是关于 `FDs` 的*那个*`范式`，因此本章不会涉及 `2NF` 或 `3NF`（甚至 `1NF`）的内容。正如我或多或少已经说过的，`2NF` 和 `3NF` 本身已不再那么有趣。

### 预备定义

在本节中，我只给出一些熟悉但绝对基础的概念的定义，几乎不做进一步阐述——这些定义比文献中通常找到的定义更为精确（在某些情况下，也比本书前面给出的定义更为精确）。举例说明这些定义的任务留给读者作为练习。

> **`定义`（`标题`）**：`标题` _H_ 是属性名称的集合。

我提醒您，上述定义故意与我第 2 章中给出的定义不完全相同，请参阅该章。事实上，它比之前的定义更简单，<sup>1</sup> 因此接下来的一些定义也会相应简化。

> **`定义`（`元组`）**：具有`标题` _H_ 的`元组` 是一组有序对 &lt;_A_,_v_&gt;（对于出现在 _H_ 中的每个属性名称 _A_ 恰好有一个这样的有序对），其中 _v_ 是一个值。如果 _H_ 在当前上下文中已明确或无关紧要，短语*具有标题 H 的元组* 可简写为*`元组`*。
>
> **`定义`（`元组投影`）**：设 _t_ 是具有`标题` _H_ 的`元组`，并设 _X_ 是 _H_ 的子集。那么 _t_ 在 _X_ 的属性上的（`元组`）`投影` _t_{_X_} 是一个具有`标题` _X_ 的`元组`——即，_t_ 的子集，仅包含那些 _A_ 出现在 _X_ 中的 &lt;_A_,_v_&gt; 有序对。<sup>2</sup>

上述定义定义了通常的`关系` `投影` 运算符的一个版本，该版本适用于单个`元组`（参见第 2 章练习 2.11）。在接下来的篇幅中，我将多次引用此定义。请注意，正如`关系` 的每个`投影` 本身也是一个`关系`，`元组` 的每个`投影` 本身也是一个`元组`。

> **`定义`（`关系`）**：`关系` _r_ 是一个有序对 &lt;_H_,_h_&gt;，其中 _h_ 是一组`元组`（_r_ 的`体`），所有这些`元组` 都具有`标题` _H_。_H_ 是 _r_ 的`标题`，_H_ 的属性是 _r_ 的属性。_h_ 的`元组` 是 _r_ 的`元组`。
>
> **`定义`（`投影`）**：设 _r_ 是`关系` &lt;_H_,_h_&gt;，并设 _X_ 是 _H_ 的子集。那么 _r_ 在 _X_ 的属性上的`投影` _r_{_X_} 是`关系` &lt;_X_,_x_&gt;，其中 _x_ 是所有`元组` _t_{_X_} 的集合，使得 _t_ 是 _h_ 的一个`元组`。
>
> **`定义`（`连接`）**：设`关系` _r1_, ..., _rn_（_n_ ≥ 0）是`可连接的`——即，设它们使得具有相同名称的属性具有相同的类型。那么 _r1_, ..., _rn_ 的`连接`，`JOIN` {_r1_,...,_rn_}，是一个`关系`，其 (a) `标题` 是 _r1_, ..., _rn_ 的`标题` 的并集，(b) `体` 是所有`元组` _t_ 的集合，使得 _t_ 是来自 _r1_ 的一个`元组`、...、以及来自 _rn_ 的一个`元组` 的并集。

出现的问题：

- 刚才定义的`连接` 类型有时更明确地称为*`自然连接`*。
- 请仔细注意，此处定义的`连接` 是一个 _n_ 元运算符，而不仅仅是二元运算符——_n_ = 2 只是一个常见的特例。（至于 _n_ &lt; 2，请参见本章末尾的练习 5.1。）正因为这个事实，**`Tutorial D`** 使用定义中提到的前缀语法风格——

```
JOIN { r1 , ..., rn }
```

——尽管在 _n_ = 2 的特例中，它也支持中缀风格：

```
r1 JOIN r2
```

然而，在本书中，从此刻起我将偏爱前缀风格。

- 同样请注意，在操作数`关系` _r1_, ..., _rn_ 中没有任何两个具有相同属性名称的特例中，`连接` 简化为`笛卡尔积`。

本节最后一个定义：

> **`定义`（`关系变量`，`relvar`）**：具有`标题` _H_ 的`关系变量` 或 `relvar` 是一个变量 _R_，使得只有当值 _r_ 是具有`标题` _H_ 的`关系` 时，该值 _r_ 才能被赋值给该变量。_H_ 的属性是 _R_ 的属性。此外，如果`关系` _r_ 被赋值给 `relvar` _R_，那么在该`赋值` 下，_r_ 的`体` 和`元组` 分别是 _R_ 的`体` 和`元组`。

_注意_：上述定义指出，`关系` _r_ 只有在其具有与 _R_ 相同的`标题` 时，才能被赋值给 `relvar` _R_。更准确地说，`关系` _r_ 可以被赋值给 `relvar` _R_ **当且仅当** (a) 它具有与 _R_ 相同的`标题`，并且 (b) 它满足适用于 _R_ 的所有`约束`，其中短语"适用于 _R_ 的所有约束" 包括在 _R_ 中成立的`函数依赖`——请参见紧随其后的章节——但不仅限于`函数依赖`。

### `函数依赖` 再探

现在，我已经能够正确地处理`函数依赖` 的概念了。同样，我将给出精确的定义——但在本节中（与上一节相比），我将对这些定义及其一些含义有更多的阐述。

> **`定义`（`函数依赖`）**：设 _H_ 是一个`标题`；那么关于 _H_ 的`函数依赖`（`FD`）是形式为 _X_ → _Y_ 的表达式，其中 _X_（`决定因素`）和 _Y_（`依赖项`）都是 _H_ 的子集。如果 _H_ 已明确，短语*关于 H 的 FD* 可简写为*`FD`*。

以下是几个示例：

```
{ CITY } → { STATUS }
{ CITY } → { SNO }
```

请仔细注意——可能与流行观点相反——`FD` 在形式上是关于某个*`标题`* 定义的，而不是关于某个`关系` 或某个 `relvar` 定义的。例如，刚才显示的两个 `FD` 是关于任何包含名为 `CITY`、`STATUS` 和 `SNO` 的属性（可能还有其他属性）的`标题` 定义的。同样请注意，从形式角度来看，`FD` 只是一个*表达式*：一个表达式，当关于某个特定`关系` 解释时，成为一个*命题*，该命题（根据定义）求值为 `TRUE` 或 `FALSE`。例如，如果上面显示的两个 `FD` 是关于作为 `relvar` `S` 当前值的`关系`（参见第 1 章图 `1-1`）解释的，那么第一个求值为 `TRUE`，第二个求值为 `FALSE`。当然，非正式地，通常只在某个特定上下文中将此类表达式定义为 `FD`，仅当它在该上下文中求值为 `TRUE` 时；然而，此类定义使得我们无法说某个给定`关系` 未能满足，或者说违反了，某个给定 `FD`。为什么？因为，根据那个非正式定义，一个未被满足的 `FD` 根本就不会是 `FD`！例如，我们将无法说作为 `relvar` `S` 当前值的`关系` 违反了上面显示的第二个 `FD`。

我真的无法足够强调上述观点。对大多数人来说，这代表了一种思维转变；然而，如果您想真正理解`设计理论` 是关于什么的，这种转变是必须做出的。要点是：大多数关于 `FD` 的著作——包括首次引入该概念的早期研究论文——实际上根本没有定义 `FD` 的概念本身！相反，它们说的是类似"_在关系 r 中，Y 函数依赖于 X，当且仅当，每当 r 的两个元组在 X 上一致时，它们在 Y 上也一致_"之类的话。这当然是完全正确的——但它不是 `FD` 的定义；相反，它是关于 `FD` _被满足_ 意味着什么的定义。但是，如果我们想发展关于 `FD` 本身的理论，那么我们显然需要能够将 `FD` 作为它们自身的对象来讨论，脱离某个特定`关系` 或某个特定 `relvar` 的上下文。更具体地说，我们需要将 `FD` 本身的概念与它可能在某个上下文中具有某种解释或意义的想法分离开来。事实上，`设计理论` 可以被视为一小部分逻辑，而逻辑根本不关乎意义——它关乎形式化的操作。

回到定义：

> **`定义`（`满足` 或 `违反` `FD`）**：设`关系` _r_ 具有`标题` _H_，并设 _X_ → _Y_ 是关于 _H_ 的 `FD`，记为 _F_。如果 _r_ 的所有`元组` 对 _t1_ 和 _t2_ 都满足：每当 _t1_{_X_} = _t2_{_X_} 时，则 _t1_{_Y_} = _t2_{_Y_}，那么 _r_ `满足` _F_；否则 _r_ `违反` _F_。

请注意，是`关系`，而不是 `relvar`，`满足` 或 `违反` 某个给定 `FD`。例如，作为 `relvar` `S` 当前值的`关系`（参见第 3 章图 `3-1`）`满足` 这两个 `FD`——

```
{ CITY }  → { STATUS }
{ SNAME } → { CITY }
```

——并`违反` 这个：

```
{ CITY } → { SNO }
```

> **`定义`（`FD 成立`）**：`FD` _F_ 在 `relvar` _R_ 中`成立`（等价地，`relvar` _R_ `受制于` `FD` _F_）**当且仅当** 可以被赋值给 `relvar` _R_ 的每个`关系` 都`满足` _F_。在 `relvar` _R_ 中`成立` 的 `FD` 是 _R_ 的 `FD`。

_重要_：请注意我在这里划分的术语区别——`FD` 被`关系` _满足_（或`违反`），但*在* `relvar` 中*成立*（或不成立）。请注意，我将在本书中始终遵循这一区别。举例来说，以下 `FD` 在 `relvar` `S` 中`成立`——

```
{ CITY } → { STATUS }
```

——而这两个不成立：

```
{ SNAME } → { CITY }
{ CITY }  → { SNO }
```

（与前面定义后的示例对比。）因此，现在，我们终于确切地知道某个给定 `relvar` `受制于` 某个给定 `FD` 意味着什么了。

### `Boyce/Codd 范式` 再探

有了对 `FD` 的正确理解，我现在可以继续处理 `relvar` 处于 `BCNF` 意味着什么的问题。同样，我通过一系列精确的定义来进行。

> **`定义`（`平凡 FD`）**：设 _X_ → _Y_ 是关于`标题` _H_ 的 `FD`，记为 _F_。那么 _F_ 是`平凡` 的**当且仅当** 它被具有`标题` _H_ 的每个`关系` 所`满足`。

现在，在第 4 章中，我将`平凡` `FD` 定义为不可能被违反的 `FD`。当然，那个定义没有问题；然而，刚才给出的定义更可取，因为它明确提到了相关的`标题`。我在第 4 章中也说过，很容易看出 `FD` _X_ → _Y_ 是`平凡` 的**当且仅当** _Y_ 是 _X_ 的子集。嗯，这也是真的；但我现在可以说，后一个事实实际上不是一个`定义`，而是一个*`定理`*，很容易从作为这样的定义中证明。（另一方面，作为这样的定义在确定给定 `FD` 是否为`平凡` 时并不是很有帮助，而该`定理` 是。因此，我们可以将该`定理` 视为一个*`操作性`* 定义，因为它提供了一个可以轻松应用于实践的有效测试。）让我明确陈述该`定理` 以供记录：

> **`定理`**：设 _X_ → _Y_ 是 `FD`，记为 _F_。那么 _F_ 是`平凡` 的**当且仅当** `依赖项` _Y_ 是`决定因素` _X_ 的子集（用符号表示，_Y_ ⊆ _X_）。

_重要_：我在这里划分的此类区别有时被描述为*`语义` 与 `句法` 区别*。详细说明：原始定义——_F 是平凡的当且仅当它被具有相关标题的每个关系所满足_——是`语义` 的，因为它定义了概念的含义；相比之下，该`定理`，或我称之为"操作性"的定义——_F 是平凡的当且仅当 Y 是 X 的子集_——是`句法` 的，因为它提供了一个可以以纯句法方式执行的检查。在接下来的篇幅中，我们将再次遇到`语义` 和`句法` 概念之间的这种区别（事实上，相当频繁）。事实上，几乎立即就出现了一个与 `FD` `不可约性` 概念相关的例子（见下文）。

继续定义：

> **`定义`（`超键`）**：`relvar` _R_ 的`超键` 是 _R_ 的`标题` _H_ 的子集 _SK_，使得 `FD` _SK_ → _H_ 在 _R_ 中`成立`（"是 _R_ 的 `FD`"）。该 `FD` 是 _R_ 上的`超键约束`。

例如，`{SNO}`、`{SNO,CITY}` 和 `{SNO,CITY,STATUS}` 都是 `relvar` `S` 的`超键`。

> **`定义`（`不可约 FD`）**：`FD` _X_ → _Y_ 关于 `relvar` _R_ 是`不可约` 的（或者如果 _R_ 已明确，则简称为`不可约`）**当且仅当** 它在 _R_ 中`成立`，并且对于 _X_ 的任何真子集 _X'_，`FD` _X'_ → _Y_ 不在 _R_ 中`成立`。

例如，`FD` `{CITY} → {STATUS}` 关于 `relvar` `S` 是`不可约` 的。相比之下，`FD` `{CITY,SNO} → {STATUS}` 虽然是 `S` 的 `FD`，但关于 `S` 是`可约` 的。请注意，虽然 `FD` 本身是关于某个*`标题`* 定义的，但 `FD` `不可约性` 是关于某个*`relvar`* 定义的。换句话说，`FD` 本身只是一个`句法` 概念（`FD` 只是采用某种句法形式的表达式），而 `FD` `不可约性` 是一个`语义` 问题（它与相关 `FD` 在相关 `relvar` 上下文中的含义有关）。_注意_：在接下来的内容中，我不假设我们讨论的 `FD` 仅仅是`不可约` 的，尽管在实践中我们通常是这样做的。

> **`定义`（`键`）**：`relvar` _R_ 的`键` 是 _R_ 的`标题` _H_ 的子集 _K_，使得 `FD` _K_ → _H_ 是 _R_ 的`不可约` `FD`。该 `FD` 是 _R_ 上的`键约束`。

请注意上述定义中对 `FD` `不可约性` 的引用。

> **`定义`（`由键隐含的 FD`）**：设 `relvar` _R_ 具有`标题` _H_，并设 _X_ → _Y_ 是关于 _H_ 的 `FD`，记为 _F_。那么 _F_ `由 R 的键隐含` **当且仅当** 每个`满足` _R_ 的`键约束` 的`关系` _r_ 也`满足` _F_。

这个定义需要一些阐述。首先，如果某个`关系` `满足` 某个`键约束`，那么它当然`满足` 相关的`唯一性` 要求；并且如果它`满足` 构成某个`键` 的属性集的`唯一性` 要求，那么它肯定也`满足` 该*属性集的每个超集* 的`唯一性` 要求（只要该超集仍然是相关`标题` 的子集，当然）——换句话说，对于每个相应的`超键`。因此，定义中的短语"满足 _R_ 的键约束" 可以用短语"满足 _R_ 的`超键约束`" 替换，而不会造成任何显著差异。同样，概念"`由键隐含`" 同样可以是"`由超键隐含`"，同样不会造成任何显著差异。

其次，如果定义中提到的 `FD` _F_ 是`平凡` 的，会发生什么？嗯，在这种情况下，根据定义，_F_ 被具有`标题` _H_ 的每个`关系` _r_ 所`满足`，因此 _F_ 肯定被每个`满足` _R_ 的`键约束` 的`关系` _r_ 所`满足`，_a fortiori_。因此，`平凡` `FD` 总是"`由键隐含`"，`平凡` 地。

第三，那么，假设 _F_ 是`非平凡` 的。那么很容易证明以下`定理`：

> **`定理`**：设 _F_ 是在 `relvar` _R_ 中`成立` 的 `FD`。那么 _F_ `由 R 的键隐含` **当且仅当** 它是 _R_ 上的`超键约束`。

换句话说，就像`平凡` `FD` 的情况一样：作为这样的形式化定义在确定给定 `FD` 是否`由键隐含` 时并没有太大帮助，但该`定理` 是。因此，我们可以将该`定理` 视为一个*`操作性`* 定义，因为它提供了一个可以轻松应用于实践的有效测试。

现在，终于，我可以定义 `BCNF` 了：

> **`定义`（`Boyce/Codd 范式`）**：`relvar` _R_ 处于 `Boyce/Codd normal form`（`BCNF`）**当且仅当** _R_ 的每个 `FD` 都`由 R 的键隐含`。

然而，鉴于本节已经讨论过的各种定义和`定理`，我们可以看到以下`操作性` 或"`句法`"定义（或`定理`）也是有效的：

> **`定义`（`Boyce/Codd 范式`）**：`relvar` _R_ 处于 `Boyce/Codd normal form`（`BCNF`）**当且仅当** 对于在 _R_ 中`成立` 的每个`非平凡` `FD` _X_ → _Y_，_X_ 是 _R_ 的`超键`。

正如我在第 4 章中所说，从这个定义可以得出，在 `BCNF` `relvar` 中`成立` 的唯一 `FD` 要么是`平凡` 的（我们显然无法摆脱它们），要么是从`超键` 出发的箭头（我们也无法摆脱它们）。不过现在我想补充一点，当我谈论"`摆脱`"某个 `FD` 时，恐怕我有点——我希望是不典型地——有点不严谨... 例如，考虑 `relvar` `S`。该 `relvar` `受制于` `FD` `{CITY} → {STATUS}`，以及其他；因此，如第 3 章所述，建议是将该 `relvar` `分解` 为其在 `{SNO,SNAME,CITY}` 上的`投影` `SNC` 和在 `{CITY,STATUS}` 上的`投影` `CT`。但是，如果我们这样做，那么 `FD` `{SNO} → {STATUS}`（它也在 `relvar` `S` 中`成立`）在某种意义上"消失"了；因此，我们在某种意义上确实"`摆脱了它`"。但是，说 `FD` 消失了是什么意思呢？答案是：它已被一个`多 relvar 约束`（即，跨越两个或多个 `relvar` 的`约束`）所取代。因此，该`约束` 当然仍然存在——它只是不再是 `FD` 了。<sup>3</sup> 每当我在本书中谈论"`摆脱`"任何类型的依赖（无论是 `FD` 还是其他）时，类似的评论都适用。

### `Heath 定理`

再次考虑 `relvar` `S` 及其 `FD` `{CITY} → {STATUS}`。假设我们`分解` 该 `relvar`，不是像第 3 章那样分解为 `relvar` `SNC` 和 `CT`，而是分解为 `relvar` _`SNT`_ 和 `CT`——其中 `CT` 与之前相同，但 `SNT` 具有`标题` `{SNO,SNAME,STATUS}` 而不是 `{SNO,SNAME,CITY}`。对应于图 `3-1` 中显示的 `S` 的值的 `SNT` 和 `CT` 的样本值显示在图 `5-1` 中：

**图 5-1**：`relvar` `SNT` 和 `CT`——样本值

（此处为图示：显示 `SNT` 和 `CT` 的样本数据）

给定这个`分解`，我希望您能看到：

- `relvar` `SNT` 和 `CT` 都处于 `BCNF`（`键` 分别是 `{SNO}` 和 `{CITY}`，并且在这两个 `relvar` 中`成立` 的唯一`非平凡` `FD` 是"`从超键出发的箭头`"）。
- 然而，与第 3 章中的`分解` 不同，这个`分解` 不是`无损` 的，而是*`有损`* 的。<sup>4</sup> 例如，我们无法从图 `5-1` 判断供应商 `S2` 是在巴黎还是在雅典——请注意如果我们`连接` 这两个`投影` 会发生什么——因此我们丢失了信息。

让我们稍微仔细看看这个例子。首先，以下是 `relvar` `SNT` 和 `CT` 的`谓词`：

- `SNC`：_供应商 SNO 名为 SNAME 且具有状态 STATUS。_
- `CT`：_城市 CITY 具有状态 STATUS。_

因此，这两个 `relvar` 的`连接` 的`谓词` 是：

- _供应商 SNO 名为 SNAME 且具有状态 STATUS_ **并且** _城市 CITY 具有状态 STATUS。_

现在回忆 `relvar` `S` 的`谓词`（参见第 2 章练习 2.6 的答案）：

- _供应商 SNO 名为 SNAME 且位于城市 CITY，_ **该城市** _具有状态 STATUS。_

后一个`谓词` 显然与`连接` 的`谓词` 不同。更准确地说，如果某个给定`元组` _t_ `满足` 它，那么该`元组` _t_ 也`满足` `连接` 的`谓词`，但反之则不然。这就是为什么`连接` "丢失信息" 或"`是有损的`"——仅仅因为某个`元组` 出现在`连接` 中，我们不能假设它也出现在原始 `relvar` `S` 中。

那么，究竟是什么使得某些`分解` 是`无损` 的，而其他是`有损` 的呢？这是位于`规范化理论` 核心的问题。它可以形式化地表述如下：

> **_设 r 是一个关系，并设 r1, ..., rn 是 r 的投影。为了使 r 等于这些投影的连接，必须满足什么条件？_**

（顺便说一句，请注意这里隐含的假设——如前所述——`连接` 是一个 _n_ 元运算符。）

对这个问题的重要但不完全的答案由 Ian `Heath` 于 1971 年提供<sup>5</sup>，当时他证明了以下`定理`：

> **`Heath 定理`**（_针对关系_）：设`关系` _r_ 具有`标题` _H_，并设 _X_、_Y_ 和 _Z_ 使得它们的并集等于 _H_（因此 _X_、_Y_ 和 _Z_ 都是 _H_ 的子集）。设 _XY_ 表示 _X_ 和 _Y_ 的并集，_XZ_ 同理。如果 _r_ `满足` `FD` _X_ → _Y_，那么 _r_ 等于其在 _XY_ 和 _XZ_ 上的`投影` 的`连接`。

举例来说，再次考虑供应商`关系`（即，如图 `3-1` 所示的 `relvar` `S` 的当前值）。该`关系` `满足` `FD` `{CITY} → {STATUS}`。因此，取 _X_ 为 `{CITY}`，_Y_ 为 `{STATUS}`，_Z_ 为 `{SNO,SNAME}`，`Heath 定理` 告诉我们，将该`关系` `分解` 为其在属性集 `{CITY,STATUS}` 和 `{CITY,SNO,SNAME}`<sup>6</sup> 上的`投影` 是`无损` 的——正如我们已经知道的那样。

现在，重要的是要理解（重复一遍），`Heath` 对原始问题的答案只是部分的。我将用前面的例子来解释这意味着什么。基本上，该`定理` 确实告诉我们`分解` 为`投影` `SNC` 和 `CT`（参见第 3 章图 `3-2`）是`无损` 的；然而，它并没有告诉我们`分解` 为 `SNT` 和 `CT`（参见图 `5-1`）是`有损` 的。换句话说，如果我们基于 `FD` 进行`分解`，如图 `3-2` 的例子中那样，那么 `Heath 定理` 说该`分解` 将是`无损` 的；但是如果我们基于其他基础进行`分解`，如图 `5-1` 的例子中那样，那么该`定理` 对此事没有任何说明。因此，该`定理` 给出了给定（二元）`分解` 为`无损` 的`充分条件`，但不是`必要条件`。由此可知，即使`关系` _r_ 不`满足` `FD` _X_ → _Y_，也可能以`无损` 方式将其`分解` 为其在 _XY_ 和 _XZ_ 上的`投影`。_注意_：我将在本书后面描述 `Heath 定理` 的一个更强形式，该形式给出了给定`分解` 为`无损` 的`必要且充分条件`（参见第 12 章）。

顺便说一句，我指出，在他证明其`定理` 的论文中，`Heath` 还给出了他所谓的"`第三`" `范式` 的定义，该定义实际上是 `BCNF` 的定义。由于该定义比 `Boyce` 和 `Codd` 的定义早了大约三年，在我看来，`BCNF` 按理应该被称为 _`Heath 范式`_。但它不是。

在第 3 章"`规范化服务于两个目的`"一节中，我说过类似以下的话：

> 如果您一直在仔细注意，您可能会合理地指责我在上述讨论中实践了一个小小的欺骗。具体来说，我考虑了`关系` 的`分解` 为`无损` 意味着什么；但`规范化`，也就是我们本应讨论的内容，不是`分解关系` 的问题，而是`分解 relvar` 的问题。

嗯，这些评论在这里也适用！所以让我们回到 `relvar`... 再次考虑 `relvar` `S`。假设我们确实决定执行推荐的`分解`，分解为"`投影`" `relvar` `SNC` 和 `CT`；此外，假设我们希望该`分解` 是`无损` 的，这确实是我们肯定想要的。换句话说，我们想要的是`分解` 使得，_在任何时候_，`relvar` `S` 的当前值等于 `SNC` 和 `CT` 的当前值的`连接`。<sup>7</sup> 也就是说，我们希望 `relvar` `S` `受制于` 以下`完整性约束`，`YCT`（实际上它是一个`等式依赖`——参见第 3 章）：

```
CONSTRAINT YCT
   S = JOIN { S { SNO , SNAME , CITY } , S { CITY , STATUS } } ;
```

现在，从第 2 章回忆，`relvar` `S` 肯定`受制于` 我在那里称为 `XCT` 的`约束`：

```
CONSTRAINT XCT
   COUNT ( S { CITY } ) = COUNT ( S { CITY , STATUS } ) ;
```

只是为了提醒您，后一个`约束` 仅仅说 `FD` `{CITY} → {STATUS}` 在 `S` 中`成立`。因此，援引 `Heath 定理`，我们看到 `relvar` `S` 的每个可能值，由于它必然`满足约束` `XCT`，也必然`满足约束` `YCT`。由此可知，`约束` `XCT` _`隐含`_ `约束` `YCT`（意思是，详细说明，如果 `relvar` `S` `受制于` `XCT`——它确实是——那么它也必然`受制于` `YCT`）。因此，`约束` `YCT` 确实`成立`，并且 `relvar` `S` `分解` 为 `relvar` `SNC` 和 `CT` 确实是`无损` 的，符合要求。由此可知，我们毕竟可以将 `Heath 定理` 视为适用于 `relvar`，而不仅仅是`关系`。因此，让我们相应地重述它：

> **`Heath 定理`**（_针对 relvar_）：设 `relvar` _R_ 具有`标题` _H_，并设 _X_、_Y_ 和 _Z_ 使得它们的并集等于 _H_（因此 _X_、_Y_ 和 _Z_ 都是 _H_ 的子集）。设 _XY_ 表示 _X_ 和 _Y_ 的并集，_XZ_ 同理。如果 _R_ `受制于` `FD` _X_ → _Y_，那么 _R_ 可以`无损分解` 为其在 _XY_ 和 _XZ_ 上的`投影`。

关于`无损分解`（到 `BCNF` 或其他）的一般主题，我想再说明一点。再次考虑 `relvar` `S` 及其 `FD` `{CITY} → {STATUS}`。根据 `Heath 定理`，该 `relvar` 可以`无损分解` 为其在 `{SNO,SNAME,CITY}` 和 `{CITY,STATUS}` 上的`投影`。然而，它显然也可以`无损分解` 为这两个`投影` _以及_（比如说）在 `{SNAME,STATUS}` 上的`投影`；也就是说，如果我们`连接` 所有这三个`投影`，我们就会回到起点。（如果这不立即明显，请使用我们通常的 `relvar` `S` 的样本值亲自验证这个说法。）然而，在重建原始 `relvar` 的过程中，第三个`投影` 显然不是必需的。现在，当我们进行`数据库设计` 时，出于显而易见的原因，我们通常只考虑每个`投影` _确实_ 在重建过程中必需的`分解`——但在这本书中，我正在讨论一般的`分解`，我不会将自己限制在每个`投影` 都是重建所必需的那些`分解` 中（当然，除非有明确的相反声明）。

### 答案

5.1 单个`关系` 的`连接`，`JOIN`{_r_}，就是 _r_；没有任何`关系` 的`连接`，`JOIN`{ }，是 `TABLE_DEE`（度为零且基数为一的唯一`关系`）。<sup>8</sup> 有关进一步解释，请参见 _`SQL and Relational Theory`_。

5.2 请参见本章正文。

5.3 `FD` c. 和 d.（仅此两个）是`平凡` 的。所有八个 `FD` a. – h. 都被 `relvar` `S` 的当前值所`满足`。除 h. 外，所有都在 `relvar` `S` 中`成立`。`FD` a.、c.、e. 和 g. 关于 `relvar` `S` 是`不可约` 的；`FD` b.、d. 和 f. 是`可约` 的。（至于 h.，`不可约性` 的问题不出现，因为该 `FD` 不在该 `relvar` 中`成立`。如果您不立即理解这一点，请检查 `FD` `不可约性` 的定义。）

5.4 `Heath 定理` 的原始版本说，如果 (a) `关系` _r_ 具有`标题` _H_，(b) _X_、_Y_ 和 _Z_ 是 _H_ 的子集且其并集等于 _H_，并且 (c) _r_ `满足` `FD` _X_ → _Y_，那么 (d) _r_ 等于其在 _XY_ 和 _XZ_ 上的`投影` 的`连接`（其中 _XY_ 表示 _X_ 和 _Y_ 的并集，_XZ_ 同理）。在下文中，我展示了该`定理` 的证明，细节极其详尽。_注意_：该证明大量使用了形式为"_t_ ∈ _r._"的表达式。此表达式可以读作"_元组 t 出现在关系 r 中_"。

    首先考虑最简单的情况，其中 *X*、*Y* 和 *Z* 是单元素集（即，每个只包含一个属性）。设相关属性分别为 *A*、*B* 和 *C*。现在，我们从第 3 章练习 3.2 的答案中知道，通过取 *r* 在 *XY*（即，在 {*A*,*B*}）上的`投影` *r1* 和在 *XZ*（即，在 {*A*,*C*}）上的`投影` *r2*，然后将 *r1* 和 *r2* `连接` 回一起，*r* 的任何`元组` 都不会丢失。我现在证明，相反，`连接` 的每个`元组` 确实是 *r* 的`元组`（换句话说，`连接` 不会生成任何"虚假"`元组`）。设 (*a*,*b*,*c*) ∈ `JOIN` {*r1*,*r2*}。为了在`连接` 中生成这样的`元组`，我们必须有 (*a*,*b*) ∈ *r1* 和 (*a*,*c*) ∈ *r2*。因此，对于某个 *b'* 和某个 *c'*，必须存在`元组` (*a*,*b*,*c'*) ∈ *r* 和 (*a*,*b'*,*c*) ∈ *r*。但是 *r* `满足` {*A*} → {*B*}；因此 *b* = *b'*，所以 (*a*,*b*,*c*) ∈ *r*。

    下一个最简单的情况是 *X*、*Y* 和 *Z* 不一定是单元素集但两两不相交的情况。在这种情况下，我们可以有效地将构成 *X* 的属性视为单个属性（*Y* 和 *Z* 同理），然后上一段的论证直接适用。

    我们现在需要考虑如果 *X*、*Y* 和 *Z* 不是两两不相交会发生什么。有三种情况需要考虑：*X* 和 *Y* 不相交，*X* 和 *Z* 不相交，以及 *Y* 和 *Z* 不相交。

    首先，设 *X* 和 *Y* 不相交，但设 *X* 和 *Z* 不相交且 *Y* 和 *Z* 不相交（因此 *Z* = *H* – *XY*）。现在回忆，如果 *X* → *Y* 被`满足`，那么对于 *Y* 的所有子集 *Y'*，*X* → *Y'* 也被`满足`。因此 `FD` *X* → *Y* – *X* 被`满足`。但是 *X* 和 *Y* – *X* 不相交；因此，根据前面的结果，*r* 等于其在 (a) *X* 和 *Y* – *X* 的并集以及 (b) *XZ* 上的`投影` 的`连接`。但是（再次）*X* 和 *Y* – *X* 不相交，因此它们的并集等于 *XY*。因此，该`定理` 在这种情况下也适用，并且我们可以在证明的剩余部分不失一般性地假设 *X* 和 *Y* 不相交。

    现在设 *X* 和 *Z* 不相交，但设 *Y* 和 *Z* 不相交。那么，根据前面的结果，*r* 等于其在 (a) *XY* 和 (b) *X* 和 *Z* – *X* 的并集上的`投影` 的`连接`。但是 *X* 和 *Z* – *X* 的并集等于 *XZ*。因此，该`定理` 在这种情况下也适用，并且我们可以在证明的剩余部分不失一般性地假设 *X* 和 *Z* 不相交。

    现在设 *Y* 和 *Z* 不相交。设 *W* = *Z* – *Y*。由于 *r* `满足` `FD` *X* → *Y*，那么它也`满足` `FD` *X* → *Y* – *W*，并且 *Y* – *W* 和 *Z* 不相交。因此，根据前面的结果，*r* 等于其在 (a) *X* 和 *Y* – *W* 的并集以及 (b) *XZ* 上的`投影` 的`连接`。我现在引用一个容易证明的`引理`（见下文），其大意是，如果 (a) *r1* 和 *r2* 是 *r* 的`投影` 使得 `JOIN`{*r1*,*r2*} = *r*，(b) *H'* 是 *H* 的子集但是 *r1* 的`标题` 的超集，并且 (c) *r'* 是 *r* 在 *H'* 上的`投影`，那么 (d) `JOIN`{*r'*,*r2*} = *r*；换句话说，粗略地说，*r1* 可以用 *r2* 的任意属性扩展而不改变`连接` 的结果。从这个`引理`，立即可以得出 *r* 等于其在 *XY* 和 *XZ* 上的`投影` 的`连接`；因此，该`定理` 在这种情况下也适用。

    *结论*：`Heath 定理` 在所有可能的情况下都有效。

    *`引理`*：设 *r* 具有`标题` *H*，并设 *H* 被划分为 *A*、*B*、*C* 和 *D*，并为简单起见假设这四个子集都不为空。（将证明扩展到涵盖该假设不成立的情况留给读者作为子练习。）不失一般性，我们可以将 *A*、*B*、*C* 和 *D* 视为好像是单个属性。因此设 *r1* = *R*{*A*,*B*} 和 *r2* = *r*{*B*,*C*,*D*}，并设 (*a*,*b*) ∈ *r1* 和 (*b*,*c*,*d*) ∈ *r2*。由于 *r* = `JOIN`{*r1*,*r2*}，由此可知 (*a*,*b*,*c*,*d*) ∈ *r*；因此 (*a*,*b*,*c*) ∈ *r*{*A*,*B*,*C*} 且 (*b*,*c*,*d*) ∈ *r*{*B*,*C*,*D*}；因此 (*a*,*b*,*c*,*d*) ∈ `JOIN`{*r*{*A*,*B*,*C*},*r*{*B*,*C*,*D*}}。所需结果随之得出——*r*{*B*,*C*,*D*} 是 *r2*，并且 *r*{*A*,*B*,*C*} 可以取为 *r'*，其中 *H'* = {*A*,*B*,*C*}。

    `Heath 定理` 的逆命题会说，如果`关系` *r* 等于其在 *XY* 和 *XZ* 上的`投影` 的`连接`，那么 *r* `满足` `FD` *X* → *Y*。这个逆命题是错误的。为了证明这一点，展示一个反例就足够了。因此考虑一个 `relvar` `CTX`，具有属性 `CNO`（课程）、`TNO`（教师）和 `XNO`（教材），以及`谓词` *课程 CNO 可以由教师 TNO 教授并使用教材 XNO*。以下是该 `relvar` 的样本值：

5.6 假设我们从一个具有属性 `D`、`P`、`S`、`L`、`T` 和 `C` 的 `relvar` 开始，这些属性以明显的方式对应于`谓词` 的参数。那么以下`非平凡` `FD` 在该 `relvar` 中`成立`：

    ```
    { L }         → { D , P , C , T }
    { D , P , C } → { L , T }
    { D , P , T } → { L , C }
    { D , P , S } → { L , C , T }
    ```

    一组可能的 `BCNF` `relvar`（大纲）是：<sup>8</sup>

    ```
    SCHEDULE { L , D , P , C , T }
             KEY { L }
             KEY { D , P , C }
             KEY { D , P , T }
    STUDYING { S , L }
             KEY { S , L }
    ```

    请注意，`FD` `{D,P,S} → {L,C,T}` 在这个`分解` 中"丢失"了（参见第 6 章）。

    *子练习*：这些 `relvar` 的`谓词` 是什么？

5.7 最简单的设计（大纲）是：

    ```
    EMP  { ENO , ENAME , SALARY }
         KEY { ENO }
    PGMR { ENO , LANG }
         KEY { ENO }
         FOREIGN KEY { ENO } REFERENCES EMP
    ```

    每个员工在 `EMP` 中都有一个`元组`（并且 `EMP` 没有其他`元组`）。碰巧是程序员的员工在 `PGMR` 中额外有一个`元组`（并且 `PGMR` 没有其他`元组`）。请注意，`EMP` 和 `PGMR` 的`连接` 为程序员（仅此）提供完整信息——员工编号、姓名、薪水和语言技能。

    如果程序员可以有两个或更多语言技能，唯一显著的区别是 `relvar` `PGMR` 将是"`全键`"的（即，其唯一`键` 将是 `{ENO,LANG}`），因此：

    ```
    PGMR { ENO , LANG }
         KEY { ENO , LANG }
         FOREIGN KEY { ENO } REFERENCES EMP
    ```
